# 讲解规范（用于模块代码讲解）

## 1. 讲解目标
每次讲解必须同时回答三个问题：
1. 为什么改：背景问题、风险、目标、非目标。
2. 怎么改：文件改动地图、关键函数执行路径、请求/数据流。
3. 如何证明改对了：测试映射、边界与风险、回滚策略。

## 2. 固定讲解顺序
1. 架构定位与边界。
2. 改造前问题与改造目标。
3. 文件级改动地图。
4. 核心函数逐段执行路径。
5. 端到端请求/数据流示例。
6. 设计取舍与替代方案。
7. 兼容性、风险、回滚。
8. 测试验证与覆盖边界。
9. 面试深挖问答。
10. 一分钟复述稿。

## 3. 深度硬性要求
1. 每个关键改动文件都要解释其职责变化。
2. 每个核心函数至少讲清：输入、分支、失败路径、副作用、返回语义。
3. 至少给一个完整链路步骤，不允许只给结论。
4. 测试章节必须做到“测试用例 ↔ 行为目标”一一对应。
5. 至少给 4 条面试深挖问答。

## 4. 表达风格
1. 先宏观后微观：先系统角色，再代码细节。
2. 先事实后结论：先讲代码行为，再做设计评价。
3. 必须包含“改造前 vs 改造后”对比。
4. 用可验证描述，避免空泛结论。

## 5. 质量自检（10 分制）
满足 8 分以上才可提交：
1. 覆盖所有关键改动文件。
2. 覆盖关键函数分支语义。
3. 有端到端流程。
4. 有前后对比。
5. 有设计取舍。
6. 有兼容/回滚策略。
7. 有测试行为映射。
8. 有未覆盖边界说明。
9. 有 4 条以上深挖问答。
10. 有一分钟复述稿。
