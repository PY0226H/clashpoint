## 2. 北极星与产品边界

### 2.1 北极星

让“辩论结束后 5 分钟内获得可读、可信、可执行的官方裁决”成为稳定用户体验。

### 2.2 产品定位

1. EchoIsle 是“在线辩论 + AI 官方裁决”平台，不是赛中陪聊型 AI 产品。
2. 核心价值是赛后官方裁决与结构化、可解释、可追溯的裁决报告，不只是给一个胜负。
3. 争议或不确定场景必须有可执行保护机制，包括 `draw`、复核与后续分支流程。
4. iPhone、Mac 与 Web 需要提供一致的核心产品能力。

### 2.3 目标规模与核心指标

目标用户量采用“在线并发等效”口径：

1. L1：500 在线，基础可用档。
2. L2：2,000 在线，公测可用档。
3. L3：5,000 在线，增长可用档。
4. L4：10,000 在线，目标档，需在真实发布环境中验证。

核心指标：

1. AI 判决时效：`P95 <= 5 分钟`，目标值 `2 分钟`。
2. 判决可用率：成功进入终局结果或明确进入复核状态的案件占比 `>= 99%`。
3. 平局流程完成率：出现 `draw` 后进入“投票/二番战”可执行终局的比例 `>= 99%`。
4. 发言链路成功率 `>= 99.5%`，端到端实时广播 `P95 < 1s`。
5. 置顶扣费 + 置顶成功率 `>= 99.9%`，`P95 < 500ms`。

### 2.4 当前版本硬边界

1. AI 裁判正式公开结果只支持 `pro / con / draw`。
2. AI 裁判只做阵营级判定与评分，不做公开个人评分。
3. 用户端不暴露置信度、内部 prompt、原始审计细节等内部字段。
4. 平台不对所有普通场次开放无限制自由申诉，只支持受规则约束的 challenge / 复核。
5. 当前内容供给聚焦游戏（云顶之弈）领域，但长期应支持规则版本化的多领域扩展。
6. 手机号是账号主联系标识；邮箱是可选联系信息。
7. 产品仍处于本地开发阶段，默认不为未上线旧行为保留长期兼容层、双轨逻辑或 legacy fallback。

### 2.5 默认工程策略

1. 对尚未上线的行为改动，默认采用硬切换与直接清理。
2. 不要为了“未来可能兼容”预留长期双路径。
3. 只有在明确存在多调用方、迁移窗口或用户显式要求时，才允许短期兼容层。
4. 若保留短期兼容层，必须写明移除条件。

## 3. 核心业务闭环

### 3.1 主流程

1. 用户通过邮箱、手机号、微信等方式登录。
2. 登录后进入首页，可访问我的会话、交锋广场、搜索、个人信息与充值入口。
3. 运营创建并排期辩题与场次，Lobby 展示待开启与进行中场次。
4. 用户在 Lobby 浏览、搜索场次，可加入待开启场次的正反方，或进入进行中场次观战。
5. 用户在 Debate Room 实时发言，可购买置顶消息。
6. 场次进入可判决状态后，平台创建 AI 裁判案件并组织完整辩论材料。
7. AI 裁判输出官方裁决与丰富裁决报告；高争议或高风险案件可先进入复核状态。
8. 复核完成后，系统必须明确给出“维持原判 / 改判 / 转为 draw / 继续保留复核状态”等可见结论。
9. 若正式公开结果为 `draw`，平台自动进入平局投票流程。
10. 投票按门槛与多数决议；若进入二番战，系统自动生成下一场次。

### 3.2 责任边界

1. `ai_judge_service` 负责判决、富内容报告、公平门禁、复核判定、证据链与可信记录。
2. 主业务系统负责用户界面展示、draw 投票、二番战、受约束 challenge 入口与后续业务编排。
3. 非法状态下不得触发判决，且同一场次所有消费者看到的最终公开结果必须一致。
4. 实时链路异常后，用户应能通过重连或刷新恢复到正确状态。

## 4. 关键模块约束

### 4.1 认证与账号

1. 支持邮箱密码、手机号密码、手机号验证码、微信登录长期共存。
2. 邮箱注册需绑定手机号并完成短信验证。
3. 微信首登必须绑定手机号后才能进入辩论与判决核心页面。
4. 同一手机号只允许绑定一个主账号。
5. 邮箱为可选字段；若填写邮箱，应格式正确且全局唯一。
6. 设置或更新密码等敏感动作需要短信验证码二次校验。
7. 平台是单租户模型，无 workspace 概念；用户只能访问已授权资源。
8. 未登录用户不能访问辩论与判决核心页面。

### 4.2 首页 / Lobby / Search

1. 登录后首页必须可直接触达：我的会话、交锋广场、搜索框、个人信息与充值入口。
2. Lobby 必须明确区分 `待开启` 与 `进行中`。
3. 场次生命周期至少覆盖 `scheduled/open/running/judging/closed`，并支持取消态管理。
4. 只有在可加入状态才能选择正反方并加入阵营。
5. 进行中场次可直接进入观战态页面。
6. 搜索框支持按辩题关键词检索场次，结果覆盖待开启和进行中两类场次。
7. 搜索无结果或为空时必须提供明确反馈。

### 4.3 Room

1. Room 需要支持实时消息、历史分页、状态恢复与断线重连。
2. 房间内展示置顶消息区与钱包余额。
3. 重连后消息与房间状态必须恢复到一致最新状态。
4. 历史分页不能丢消息、不能重复消息。

### 4.4 AI Judge

1. 判决只能在合法案件状态下触发；平台可在场次进入 `judging/closed` 后自动或按授权策略触发。
2. 对长对局，平台应支持保留阶段性裁判记录，但官方终局结果只在终局案件完成后公开。
3. 支持“允许重判 / 允许复核 / 允许 challenge”类策略开关，且必须受赛事规则与系统规则约束。
4. 任务状态至少覆盖：处理中、已完成（`pro/con/draw`）、复核中、失败。
5. AI 裁判报告必须稳定包含：
   - 最终结果或复核状态
   - 阵营总分
   - 四维评分：逻辑、证据、反驳、清晰度
   - 正反方核心观点或主要争点摘要
   - 用户可读裁决理由
   - 关键转折点或决定性回合
   - 阶段摘要或争点展开
   - 证据引用与背景来源
   - 规则版本、公平性说明或复核说明
6. 高分歧、高不确定、证据不足或公平风险高时，应进入 `draw` 或受保护复核，不要强判。
7. 输入必须做盲化与中立性处理，避免身份标签、消费能力、历史声誉等直接污染单场裁判。
8. 判决必须保留消息级证据链、规则版本、复核状态和关键结果历史。
9. RAG / 背景检索的职责是补充证据与解释，不是直接替代裁判结论。

### 4.5 Draw / Review / Challenge

1. `draw` 不是失败，而是正式公开结果之一。
2. AI 结果为 `draw` 时，平台必须自动创建投票。
3. 投票窗口为 5 分钟。
4. 参与门槛为本场参与用户的 70%。
5. 达门槛后按多数票决议：同意平局则结束，不同意平局则开启二番战。
6. 超时未达门槛时，默认开启二番战。
7. 二番战必须生成可用场次 ID，并可在产品内直接被发现和加入。
8. 复核与 challenge 必须受赛事规则、价值等级或系统策略约束，不能成为无限制自由申诉。
9. 复核必须绑定原案件、原裁决版本与补充材料。
10. 复核完成后，产品上必须能明确落到“维持原判 / 改判 / 转为 draw / 关闭 challenge 并保留复核状态说明”之一。

### 4.6 Wallet / Pinned Message

1. 充值入口必须能从个人信息区进入。
2. 用户可查看余额并完成“余额查看 -> 充值 -> 返回辩论消费”闭环。
3. 参赛用户可为自己发言购买置顶时长。
4. 置顶按时长扣费，余额不足必须阻断并提示。
5. 同一幂等键不能重复扣费。
6. 置顶规则为 30s-600s，按 30s 计费单位结算。
7. 置顶成功后必须进入置顶区并显示到期时间。

### 4.7 Ops / Release

1. 运营可创建和编辑辩题、场次与定时开启计划。
2. 运营权限必须与普通用户严格隔离。
3. 运营改动应实时反映到 Lobby 与 Room。
4. 定时场次应在设定时间前显示为待开启，到点后按规则进入可参与状态。
5. 运营侧需要能看成功率、失败、重试、复核、平局、回放、告警与规则版本表现。
6. 运营复盘应支持按时间窗口、辩题、规则版本与结果类型查看裁判表现。
7. 发布目标包含 iPhone、Mac 与 Web；Apple 平台需满足支付、合规、审核等上线要求。

## 5. 非功能与质量约束

1. 判决为赛后异步，不阻塞用户主操作。
2. 判决结果需兼具理性、可读性与层次感，既能快速看结果，也能展开看依据。
3. 同一场次所有用户看到的最终公开结果必须一致。
4. 实时链路异常后，用户可通过重连或刷新恢复到正确状态。
5. 登录与验证码链路需具备限流、验证码有效期与敏感操作二次校验能力。
6. 认证与分析链路应尽量避免明文账号标识扩散，满足最小必要使用原则。
7. 关键判决、规则版本与复核状态应可被平台追溯。
8. 平台应支持规则版本化、裁判策略演进与未来多组织制度扩展。

## 6. 当前明确非目标

当前版本默认不要误做这些能力：

1. 公开个人评分、个人排行榜或个人胜负裁决
2. 不受规则约束的无限制用户申诉系统
3. 赛中实时 AI 教练、陪聊或代打式建议
4. 向普通用户暴露内部 prompt、置信度或完整审计明细
5. 把投票、二番战、挑战编排错误地下沉到 AI 裁判服务内部
6. 把完整 PRD 当成开发排期、当前代码盘点或阶段缺口清单

## 7. 何时必须回读完整 PRD

默认先读本摘要；命中以下任一情况时，必须回读完整 PRD：

1. 认证、账号安全、权限、短信验证码、微信登录、密码链路
2. 支付、钱包、账本、充值、置顶扣费
3. AI 裁判结果结构、复核、challenge、draw、投票、二番战或用户展示字段
4. RAG、证据链、输入盲化、公平门禁、规则版本、Judge Kernel、可信记录
5. 运营后台、角色隔离、定时场次、审核发布或 App Store 相关要求
6. 搜索、Lobby、Room、跨端首页入口或核心用户旅程改动
7. 目标指标、并发规模、性能口径或发布形态相关改动
8. 跨服务边界、关键数据流、核心用户主流程改动
9. 任务本身显式要求“完整 PRD 对齐”或“按最终形态重新校准”
10. 当前摘要不足以覆盖该模块的产品约束

## 8. 开发前快速自检

进入编码前，至少回答这 6 个问题：

1. 这次改动服务的是哪个用户价值或主流程节点
2. 它会不会破坏登录、首页、Lobby、Search、Room、Judge、Draw、Review、Wallet、Ops 这些关键链路中的约束
3. 它会不会误做当前版本明确不做的能力
4. 它会不会让 AI 裁判偏离“公正优先、证据可追溯、结果可解释”的主线
5. 它会不会影响同一场次最终结果一致性、实时恢复、账号安全或隐私最小化
6. 如果本次属于高风险模块，是否已经回读完整 PRD
