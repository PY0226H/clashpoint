## 2. 北极星与产品边界

### 2.1 北极星

让用户在辩论结束后 5 分钟内获得可读、可信、可执行的官方裁决，或明确进入受保护复核状态。

### 2.2 产品定位

1. EchoIsle 是“在线辩论 + AI 裁判”的平台，不是赛中陪聊型 AI 产品。
2. 核心价值是赛后官方裁决与丰富裁决报告，不只是给一个胜负。
3. AI 裁判要兼顾判决、公平门禁、复核、证据可追溯与可信记录。
4. iPhone、Mac 与 Web 需要提供一致的核心体验。

### 2.3 当前版本硬边界

1. AI 裁判公开结果只支持 `pro / con / draw`。
2. AI 裁判是阵营级裁决，不做公开个人评分。
3. 用户端不暴露置信度、内部 prompt、原始审计细节等内部字段。
4. 平台不对所有普通场次开放无限制自由申诉，只支持受规则约束的 challenge / 复核。
5. 当前内容供给聚焦云顶之弈，但设计上应允许未来按规则版本扩展到多领域。
6. 产品仍处于本地开发阶段，默认不为未上线旧行为保留长期兼容层、双轨逻辑或 legacy fallback。

### 2.4 默认工程策略

1. 对尚未上线的行为改动，默认采用硬切换与直接清理。
2. 不要为了“未来可能兼容”预留长期双路径。
3. 只有在明确存在多调用方、迁移窗口或用户显式要求时，才允许短期兼容层。
4. 若保留短期兼容层，必须写明移除条件。

## 3. 核心业务闭环

### 3.1 主流程

1. 用户通过邮箱/手机号/微信等方式登录。
2. 登录后进入首页，可访问聊天、交锋广场、搜索、个人信息与充值入口。
3. 用户在 Lobby 浏览待开启/进行中场次，加入正反方或进入观战。
4. 用户在 Debate Room 实时发言，可购买置顶。
5. 场次进入可判决状态后，平台创建 AI 裁判案件并组织完整辩论材料。
6. AI 裁判输出官方裁决与丰富裁决报告；高争议或高风险案件可先进入复核状态。
7. 若正式公开结果为 `draw`，平台自动进入投票 / 二番战链路。
8. 用户可查看裁决、复核状态、历史记录，并继续参与后续分支流程。

### 3.2 责任边界

1. `ai_judge_service` 负责判决、富内容报告、公平门禁、复核判定、证据链与可信记录。
2. 主业务系统负责用户界面展示、draw 投票、二番战、受约束 challenge 入口与后续业务编排。
3. 非法状态下不得触发判决，且同一场次所有消费者看到的最终公开结果必须一致。
4. 实时链路异常后，用户应能通过重连或刷新恢复到正确状态。

## 4. 关键模块约束

### 4.1 认证与账号

1. 支持邮箱密码、手机号密码、手机号验证码、微信登录长期共存。
2. 微信首登必须绑定手机号后才能进入核心页面。
3. 同一手机号只允许绑定一个主账号。
4. 设置或更新密码等敏感动作需要短信验证码二次校验。
5. 未登录用户不能访问辩论与判决核心页面。

### 4.2 Lobby / Room

1. Lobby 必须明确区分 `待开启` 与 `进行中`。
2. 只有在可加入状态才能加入阵营。
3. Room 需要支持实时消息、历史分页、状态恢复与断线重连。
4. 重连后消息与房间状态必须恢复到一致最新状态。

### 4.3 AI Judge

1. 判决只能在合法案件状态下触发，任务状态至少要有：处理中、已完成（`pro/con/draw`）、复核中、失败。
2. AI 裁判报告必须稳定包含：
   - 最终结果或复核状态
   - 阵营总分
   - 四维评分：逻辑、证据、反驳、清晰度
   - 正反方核心观点或主要争点摘要
   - 裁决理由
   - 关键转折点
   - 阶段摘要或争点展开
   - 证据引用与背景来源
   - 必要的裁判说明、规则版本或复核说明
3. 高分歧、高不确定、证据不足或公平风险高时，应进入 `draw` 或受保护复核，不要强判。
4. 输入必须做盲化与中立性处理，避免身份标签、消费能力、历史声誉等直接污染单场裁判。
5. 判决必须保留消息级证据链、规则版本、复核状态和关键结果历史。
6. 知识 / RAG / 背景检索的职责是补充证据与解释，不是直接替代裁判结论。
7. 关键裁决事实、规则版本与复核状态应可审计、可追溯，并为未来可验证承诺预留空间。

### 4.4 Draw / Review / Challenge

1. `draw` 不是失败，而是正式公开结果之一。
2. AI 结果为 `draw` 时，平台必须自动进入投票流程；投票未达门槛时默认开启二番战。
3. 二番战分支必须生成可用场次 ID 且可进入。
4. 复核与 challenge 必须受赛事规则、价值等级或系统策略约束，不能成为无限制自由申诉。
5. 复核必须绑定原案件、原裁决版本与补充材料。
6. 复核完成后，产品上必须能明确落到“维持原判 / 改判 / 转为 draw / 保留复核说明”之一。

### 4.5 Wallet / Pinned Message

1. 充值入口必须能从个人信息区进入。
2. 置顶按时长扣费，余额不足必须阻断。
3. 同一幂等键不能重复扣费。
4. 置顶成功后必须进入置顶区并显示到期时间。

### 4.6 Ops / Release

1. 运营可创建和编辑辩题、场次与定时开启计划。
2. 运营权限必须与普通用户严格隔离。
3. 运营侧需要能看成功率、失败、重试、复核、平局、回放、告警与规则版本表现。
4. 发布目标仍然包含 App Store 审核、支付、认证、权限、内容安全等高风险要求。

## 5. 当前明确非目标

当前版本默认不要误做这些能力：

1. 公开个人评分、个人排行榜或个人胜负裁决
2. 不受规则约束的无限制用户申诉系统
3. 赛中实时 AI 教练、陪聊或代打式建议
4. 向普通用户暴露内部 prompt、置信度或完整审计明细
5. 把投票、二番战、挑战编排错误地下沉到 AI 裁判服务内部

## 6. 何时必须回读完整 PRD

默认先读本摘要；命中以下任一情况时，必须回读完整 PRD：

1. 认证、账号安全、权限、短信验证码、微信登录、密码链路
2. 支付、钱包、账本、充值、置顶扣费
3. AI 裁判结果结构、复核、challenge、draw、投票、二番战或用户展示字段
4. RAG、证据链、输入盲化、公平门禁、规则版本、Judge Kernel、可信记录
5. 运营后台、角色隔离、定时场次、审核发布或 App Store 相关要求
6. 跨服务边界、关键数据流、核心用户主流程改动
7. 任务本身显式要求“完整 PRD 对齐”或“按最终形态重新校准”
8. 当前摘要不足以覆盖该模块的产品约束

## 7. 开发前快速自检

进入编码前，至少回答这 5 个问题：

1. 这次改动服务的是哪个用户价值或主流程节点
2. 它会不会破坏登录、Lobby、Room、Judge、Draw、Review、Wallet、Ops 这些关键链路中的约束
3. 它会不会误做当前版本明确不做的能力
4. 它会不会让 AI 裁判偏离“公正优先、证据可追溯、结果可解释”的主线
5. 如果本次属于高风险模块，是否已经回读完整 PRD
