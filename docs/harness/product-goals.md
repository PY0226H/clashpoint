## 2. 北极星与产品边界

### 2.1 北极星

让用户在辩论结束后 5 分钟内获得可读、可信、可执行的官方裁决。

### 2.2 当前产品定位

1. 在线辩论 + AI 裁判平台
2. 核心价值是“赛后官方结论”，不是赛中聊天型 AI
3. 产品最终目标是 App Store 手机端正式上线并可持续运营

### 2.3 当前版本边界

1. AI 裁判只做阵营级判定与评分，不做个人评分
2. 判决结果只支持 `pro / con / draw`
3. 用户端不暴露置信度等内部风控字段
4. 当前不开放用户申诉入口
5. MVP 先聚焦单领域：云顶之弈
6. 账号支持多登录方式并存，但手机号是唯一主联系标识
7. 当前产品仍处于本地开发阶段，默认不为尚未上线的旧实现保留兼容层、灰度开关或 legacy fallback

### 2.4 当前工程切换默认策略

1. 对尚未上线的行为改动，默认采用硬切换与直接清理
2. 不要为了“未来可能兼容”预先保留双路径
3. 只有在明确存在多调用方、迁移窗口或用户显式要求时，才允许短期兼容层
4. 若确实保留短期兼容层，必须明确写出后续删除条件

## 3. 核心业务闭环

### 3.1 主流程

1. 用户通过邮箱/手机号/微信等方式登录
2. 登录后进入首页，可访问聊天、交锋广场、搜索、个人信息与充值入口
3. 用户进入 Lobby 浏览待开启/进行中场次
4. 用户加入正反方或进入观战
5. 用户在 Debate Room 发言，可购买置顶
6. 场次结束后触发 AI 判决
7. 系统展示判决报告
8. 若结果为 draw，则进入投票或二番战链路

### 3.2 关键一致性要求

1. 同一场次所有用户看到的最终结果必须一致
2. 实时链路异常后，用户应能通过重连/刷新恢复正确状态
3. 非法状态下不得触发判决
4. 权限与角色隔离必须明确，运营能力不能泄漏给普通用户

## 4. 关键模块约束

### 4.1 认证与账号

1. 支持邮箱密码、手机号密码、手机号验证码、微信登录长期共存
2. 微信首登必须绑定手机号后才能进入核心页面
3. 同一手机号只允许绑定一个主账号
4. 设置或更新密码等敏感动作需要短信验证码二次校验
5. 未登录用户不能访问辩论与判决核心页面

### 4.2 Lobby / Room

1. Lobby 要明确区分 `待开启` 与 `进行中`
2. 只有在可加入状态才能加入阵营
3. Room 需要支持实时消息、历史分页、状态恢复
4. 重连后消息与房间状态必须恢复到一致最新状态

### 4.3 AI Judge / Draw / Rematch

1. 判决必须包含胜负、总分、四维评分、摘要、理由、阶段摘要、背景来源
2. 结果为 draw 时，必须自动进入投票流程
3. 投票未达门槛时，默认开启二番战
4. 二番战分支必须生成可用场次 ID 且可进入

### 4.4 Wallet / Pinned Message

1. 充值入口必须能从个人信息区进入
2. 置顶按时长扣费，余额不足必须阻断
3. 同一幂等键不能重复扣费
4. 置顶成功后必须进入置顶区并显示到期时间

### 4.5 Ops / Release

1. 运营可创建/编辑辩题与场次
2. 运营权限必须与普通用户严格隔离
3. 发布目标是满足 App Store 审核与上线要求
4. 认证、支付、权限、内容安全与提审要求都属于高风险范围

---

## 5. 明确非目标

当前版本不要误做这些能力：

1. 用户个人评分与个人排行榜
2. 面向用户的申诉系统
3. 多领域同时上线
4. 赛中实时 AI 教练或陪聊

---

## 6. 何时必须回读完整 PRD

默认先读本摘要；命中以下任一情况时，必须再回读完整 PRD：

1. 认证、账号安全、权限、短信验证码、微信登录、密码链路
2. 支付、钱包、账本、充值、置顶扣费
3. AI 裁判结果结构、draw 流程、投票、二番战、RAG、解释性字段
4. 运营后台、角色隔离、定时场次、审核/发布/App Store
5. 跨服务边界、关键数据流、核心用户主流程改动
6. 任务本身显式要求“完整 PRD 对齐”或“按最终形态重新校准”
7. 当前摘要不足以覆盖该模块的产品约束

---

## 7. 开发前快速自检

进入编码前，至少回答这四个问题：

1. 这次改动服务的是哪个用户价值或主流程节点
2. 它会不会破坏登录、Lobby、Room、Judge、Draw、Wallet、Ops 这些关键链路中的任一约束
3. 它会不会误做当前版本明确不做的能力
4. 如果本次属于高风险模块，是否已经回读完整 PRD
