## 2. 北极星与产品边界

### 2.1 北极星

让“辩论结束后 5 分钟内获得可读、可信、可执行的官方裁决”成为稳定用户体验。

### 2.2 产品定位

1. EchoIsle 是“在线辩论 + AI 官方裁决”平台，赛后官方裁决仍是权威主线。
2. 平台允许受边界约束的赛中 AI 能力，包括会员私人辩论助手与公开虚拟裁判 NPC。
3. 核心价值是赛后官方裁决与结构化、可解释、可追溯的裁决报告，不只是给一个胜负。
4. 赛中 AI 只能辅助理解、表达、秩序、气氛和复盘，不能替代、污染或提前泄露正式裁决。
5. 争议或不确定场景必须有可执行保护机制，包括 `draw`、复核与后续分支流程。
6. iPhone、Mac 与 Web 需要提供一致的核心产品能力。

### 2.3 目标规模与核心指标

目标用户量采用“在线并发等效”口径：

1. L1：500 在线，基础可用档。
2. L2：2,000 在线，公测可用档。
3. L3：5,000 在线，增长可用档。
4. L4：10,000 在线，目标档，需在真实发布环境中验证。

核心指标：

1. AI 判决时效：`P95 <= 5 分钟`，目标值 `2 分钟`。
2. 判决可用率：成功进入终局结果或明确进入复核状态的案件占比 `>= 99%`。
3. 平局流程完成率：出现 `draw` 后进入“投票/二番战”可执行终局的比例 `>= 99%`。
4. 发言链路成功率 `>= 99.5%`，端到端实时广播 `P95 < 1s`。
5. 置顶扣费 + 置顶成功率 `>= 99.9%`，`P95 < 500ms`。
6. 会员助手回答满意度保持在可接受阈值以上，“代打、不公平、误导、泄露内部裁判信息”类投诉率保持低位。
7. NPC 启用房间的停留、互动或观战反馈提升，“打断过多、偏袒一方、混淆正式裁决”类负反馈率保持低位。

### 2.4 当前版本硬边界

1. AI 裁判正式公开结果只支持 `pro / con / draw`。
2. AI 裁判只做阵营级判定与评分，不做公开个人评分。
3. 用户端不暴露置信度、内部 prompt、原始审计细节等内部字段。
4. 平台不对所有普通场次开放无限制自由申诉，只支持受规则约束的 challenge / 复核。
5. 当前内容供给聚焦游戏（云顶之弈）领域，但长期应支持规则版本化的多领域扩展。
6. 手机号是账号主联系标识；邮箱是可选联系信息。
7. 用户辩论助手是会员专属私人辅助，不代表平台正式观点，不进入公开房间消息流，不自动替用户发言。
8. 虚拟裁判 NPC 是房间公开可见的娱乐化现场角色，不作为最终胜负判定来源，不提供私聊或个人专属建议。
9. 赛中助手、虚拟裁判 NPC 与赛后 AI 裁判团必须在产品表达、数据使用、用户可见性与权限边界上清晰区分。
10. 产品仍处于本地开发阶段，默认不为未上线旧行为保留长期兼容层、双轨逻辑或 legacy fallback。

### 2.5 默认工程策略

1. 对尚未上线的行为改动，默认采用硬切换与直接清理。
2. 不要为了“未来可能兼容”预留长期双路径。
3. 只有在明确存在多调用方、迁移窗口或用户显式要求时，才允许短期兼容层。
4. 若保留短期兼容层，必须写明移除条件。

## 3. 核心业务闭环

### 3.1 主流程

1. 用户通过邮箱、手机号、微信等方式登录。
2. 登录后进入首页，可访问我的会话、交锋广场、搜索、个人信息与充值入口。
3. 运营创建并排期辩题与场次，Lobby 展示待开启与进行中场次。
4. 用户在 Lobby 浏览、搜索场次，可加入待开启场次的正反方，或进入进行中场次观战。
5. 用户在 Debate Room 实时发言，可购买置顶消息。
6. 启用虚拟裁判 NPC 的房间可获得公开阶段提示、秩序提醒、赞赏和气氛反馈。
7. 会员用户可打开私人辩论助手，用于理解局势、总结争点、整理发言结构或赛后复盘。
8. 场次进入可判决状态后，平台创建 AI 裁判案件并组织完整辩论材料。
9. AI 裁判输出官方裁决与丰富裁决报告；高争议或高风险案件可先进入复核状态。
10. 复核完成后，系统必须明确给出“维持原判 / 改判 / 转为 draw / 继续保留复核状态”等可见结论。
11. 若正式公开结果为 `draw`，平台自动进入平局投票流程。
12. 投票按门槛与多数决议；若进入二番战，系统自动生成下一场次。

### 3.2 责任边界

1. `ai_judge_service` 负责判决、富内容报告、公平门禁、复核判定、证据链与可信记录。
2. 主业务系统负责用户界面展示、draw 投票、二番战、受约束 challenge 入口与后续业务编排。
3. 用户辩论助手负责会员私人理解、总结、建议和复盘，不公开代表平台观点，不影响正式裁决。
4. 虚拟裁判 NPC 负责房间公开互动、气氛、节奏和秩序提示，不决定最终胜负。
5. 非法状态下不得触发判决，且同一场次所有消费者看到的最终公开结果必须一致。
6. 实时链路异常后，用户应能通过重连或刷新恢复到正确状态。

## 4. 关键模块约束

### 4.1 认证与账号

1. 支持邮箱密码、手机号密码、手机号验证码、微信登录长期共存。
2. 邮箱注册需绑定手机号并完成短信验证。
3. 微信首登必须绑定手机号后才能进入辩论与判决核心页面。
4. 同一手机号只允许绑定一个主账号。
5. 邮箱为可选字段；若填写邮箱，应格式正确且全局唯一。
6. 设置或更新密码等敏感动作需要短信验证码二次校验。
7. 平台是单租户模型，无 workspace 概念；用户只能访问已授权资源。
8. 未登录用户不能访问辩论与判决核心页面。

### 4.2 首页 / Lobby / Search

1. 登录后首页必须可直接触达：我的会话、交锋广场、搜索框、个人信息与充值入口。
2. Lobby 必须明确区分 `待开启` 与 `进行中`。
3. 场次生命周期至少覆盖 `scheduled/open/running/judging/closed`，并支持取消态管理。
4. 只有在可加入状态才能选择正反方并加入阵营。
5. 进行中场次可直接进入观战态页面。
6. 搜索框支持按辩题关键词检索场次，结果覆盖待开启和进行中两类场次。
7. 搜索无结果或为空时必须提供明确反馈。

### 4.3 Room

1. Room 需要支持实时消息、历史分页、状态恢复与断线重连。
2. 房间内展示置顶消息区与钱包余额。
3. 重连后消息与房间状态必须恢复到一致最新状态。
4. 历史分页不能丢消息、不能重复消息。
5. 房间应能承载虚拟裁判 NPC 的公开展示、消息样式、状态反馈与必要强动作提示。
6. 房间应为会员用户提供私人助手入口，并保证助手面板不遮挡发言、观战和关键状态信息。
7. NPC 消息、助手入口与正式裁决入口必须在视觉和语义上可区分。

### 4.4 AI Judge

1. 判决只能在合法案件状态下触发；平台可在场次进入 `judging/closed` 后自动或按授权策略触发。
2. 对长对局，平台应支持保留阶段性裁判记录，但官方终局结果只在终局案件完成后公开。
3. 支持“允许重判 / 允许复核 / 允许 challenge”类策略开关，且必须受赛事规则与系统规则约束。
4. 任务状态至少覆盖：处理中、已完成（`pro/con/draw`）、复核中、失败。
5. AI 裁判报告必须稳定包含：
   - 最终结果或复核状态
   - 阵营总分
   - 四维评分：逻辑、证据、反驳、清晰度
   - 正反方核心观点或主要争点摘要
   - 用户可读裁决理由
   - 关键转折点或决定性回合
   - 阶段摘要或争点展开
   - 证据引用与背景来源
   - 规则版本、公平性说明或复核说明
6. 高分歧、高不确定、证据不足或公平风险高时，应进入 `draw` 或受保护复核，不要强判。
7. 输入必须做盲化与中立性处理，避免身份标签、消费能力、历史声誉等直接污染单场裁判。
8. 判决必须保留消息级证据链、规则版本、复核状态和关键结果历史。
9. RAG / 背景检索的职责是补充证据与解释，不是直接替代裁判结论。

### 4.5 Draw / Review / Challenge

1. `draw` 不是失败，而是正式公开结果之一。
2. AI 结果为 `draw` 时，平台必须自动创建投票。
3. 投票窗口为 5 分钟。
4. 参与门槛为本场参与用户的 70%。
5. 达门槛后按多数票决议：同意平局则结束，不同意平局则开启二番战。
6. 超时未达门槛时，默认开启二番战。
7. 二番战必须生成可用场次 ID，并可在产品内直接被发现和加入。
8. 复核与 challenge 必须受赛事规则、价值等级或系统策略约束，不能成为无限制自由申诉。
9. 复核必须绑定原案件、原裁决版本与补充材料。
10. 复核完成后，产品上必须能明确落到“维持原判 / 改判 / 转为 draw / 关闭 challenge 并保留复核状态说明”之一。

### 4.6 用户辩论助手

1. 用户辩论助手是会员用户在 Debate Room 内可打开的私人 AI 助手。
2. 助手服务于私人理解、总结、建议与赛后复盘，不是公开裁判，也不代表平台正式观点。
3. 会员用户可用助手回答当前状态、公开规则、背景、公开裁决报告相关问题。
4. 助手可总结双方观点、关键争点、未回应问题、关键发言和最新进展。
5. 助手可提供发言结构、反驳方向、草稿优化和注意事项。
6. 助手对话默认仅当前用户可见，不进入公开房间消息流。
7. 助手不得自动替用户发送消息、一键代发、持续操控下一句发言或绕过房间规则。
8. 助手不得宣称最终胜负、提供正式胜负预测、暴露裁判团内部字段或查看其他用户私密信息。
9. 正式裁决标准不得因用户是否开通会员、是否使用助手而改变。
10. 非会员用户可看到锁定态入口和权益说明，但基础辩论、观战与正式裁决体验不受影响。

### 4.7 虚拟裁判 NPC

1. 虚拟裁判 NPC 是 Debate Room 中公开可见、娱乐导向、可主动观察辩论并与房间内用户互动的 AI 角色。
2. NPC 服务于现场感、气氛、节奏、秩序、规则提醒和观赛可理解性，不替代赛后 AI 裁判团。
3. 启用 NPC 的房间应展示明确身份、动态形象或角色展示区、当前状态和与正式裁决不同的视觉样式。
4. NPC 可公开发送开场、阶段、规则、争点、回应、节奏、高能、赞赏和收束提示。
5. NPC 可对精彩发言、高质量反驳、幽默表达或关键追问做出公开赞赏和气氛反馈。
6. NPC 可在规则允许并经运营配置后触发暂停、恢复、警告、特效、阶段总结等房间行为。
7. 用户可公开呼叫 NPC，但不能私聊 NPC，也不能通过 NPC 获取个人专属辩论建议。
8. NPC 不得宣布最终胜负、公开阵营评分、展示胜率预测或内部裁判依据。
9. NPC 对用户发言的赞赏不构成阵营优势、官方评分或正式裁决证据。
10. NPC 不可用或被关闭时，房间核心辩论、观战和正式裁决流程不受影响。

### 4.8 Wallet / Pinned Message

1. 充值入口必须能从个人信息区进入。
2. 用户可查看余额并完成“余额查看 -> 充值 -> 返回辩论消费”闭环。
3. 参赛用户可为自己发言购买置顶时长。
4. 置顶按时长扣费，余额不足必须阻断并提示。
5. 同一幂等键不能重复扣费。
6. 置顶规则为 30s-600s，按 30s 计费单位结算。
7. 置顶成功后必须进入置顶区并显示到期时间。

### 4.9 Ops / Release

1. 运营可创建和编辑辩题、场次与定时开启计划。
2. 运营权限必须与普通用户严格隔离。
3. 运营改动应实时反映到 Lobby 与 Room。
4. 定时场次应在设定时间前显示为待开启，到点后按规则进入可参与状态。
5. 运营侧需要能看成功率、失败、重试、复核、平局、回放、告警与规则版本表现。
6. 运营复盘应支持按时间窗口、辩题、规则版本与结果类型查看裁判表现。
7. 运营可控制 NPC 启用策略、风格、能力开关、强动作权限与人工接管。
8. 运营应能观察助手与 NPC 的负反馈、边界风险和体验质量。
9. 发布目标包含 iPhone、Mac 与 Web；Apple 平台需满足支付、合规、审核等上线要求。

## 5. 非功能与质量约束

1. 判决为赛后异步，不阻塞用户主操作。
2. 判决结果需兼具理性、可读性与层次感，既能快速看结果，也能展开看依据。
3. 同一场次所有用户看到的最终公开结果必须一致。
4. 实时链路异常后，用户可通过重连或刷新恢复到正确状态。
5. 登录与验证码链路需具备限流、验证码有效期与敏感操作二次校验能力。
6. 认证与分析链路应尽量避免明文账号标识扩散，满足最小必要使用原则。
7. 关键判决、规则版本与复核状态应可被平台追溯。
8. 平台应支持规则版本化、裁判策略演进与未来多组织制度扩展。
9. 用户辩论助手的会员身份、使用记录、提问内容或建议内容不得进入正式裁决评分标准。
10. 虚拟裁判 NPC 的赞赏、阶段提示、特效或现场行为不得作为正式裁决的唯一依据，也不得改变 AI 裁判团评分规则。
11. 赛中 AI 能力必须遵循边界隔离：私人助手只对本人可见，NPC 只在公开房间语境中互动，正式裁决只来自赛后官方裁判流程。

## 6. 当前明确非目标

当前版本默认不要误做这些能力：

1. 公开个人评分、个人排行榜或个人胜负裁决
2. 不受规则约束的无限制用户申诉系统
3. 无边界的赛中 AI 教练、自动代打、一键代发、强操控式“下一句该怎么说”
4. 向普通用户暴露内部 prompt、置信度或完整审计明细
5. 把投票、二番战、挑战编排错误地下沉到 AI 裁判服务内部
6. 把完整 PRD 当成开发排期、当前代码盘点或阶段缺口清单
7. 让虚拟裁判 NPC 公开评分、预测胜负、提供私聊、替代正式裁决或把赞赏作为裁决依据
8. 让会员助手影响正式裁决标准、查看其他用户私密信息或绕过房间/赛事规则

## 7. 何时必须回读完整 PRD

默认先读本摘要；命中以下任一情况时，必须回读完整 PRD：

1. 认证、账号安全、权限、短信验证码、微信登录、密码链路
2. 支付、钱包、账本、充值、置顶扣费
3. AI 裁判结果结构、复核、challenge、draw、投票、二番战或用户展示字段
4. RAG、证据链、输入盲化、公平门禁、规则版本、Judge Kernel、可信记录
5. 运营后台、角色隔离、定时场次、审核发布或 App Store 相关要求
