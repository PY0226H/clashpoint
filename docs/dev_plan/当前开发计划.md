## 3. 本轮北极星与不可变边界

1. 官方裁决结果仍只能是 `pro / con / draw`；高风险场景进入 `review_required` 或 `blocked_failed`，不得静默强判。
2. `Fairness Sentinel -> Chief Arbiter -> Opinion Writer` 的顺序不能绕过；P36 的 trust layer 只能承诺和验证既有裁决事实，不得反向改写 verdict。
3. 六对象主链仍是事实核心：`case_dossier / claim_graph / evidence_ledger / verdict_ledger / fairness_report / opinion_pack`。
4. Trust Registry 是对六对象、workflow、policy kernel、audit anchor 的制度化承诺层，不是新的平行 winner 写链。
5. `NPC Coach / Room QA` 继续保持 `advisory_only`，不得写 `verdict_ledger`、`judge_ledger_snapshots` 或 trust registry 官方链路。
6. 客户端仍只通过 `chat_server` 间接使用 AI 能力；`ai_judge_service` 不暴露公众直连接口。
7. Redis 只能做幂等锁、短 TTL 缓存、速率限制和临时上下文；trust registry、challenge registry、artifact registry 均不能只存在 Redis。
8. 不引入 LangChain/LangGraph 作为生产主骨架；不在 P36 引入 Temporal/Kafka 强依赖。
9. 真实环境结论与本地参考结论分层表达；没有真实环境前，不宣称 real-env `pass`。

3. 保持 P35 六对象主链不被 trust 写入反向污染。

**落地内容**

1. 在 final/phase 成功报告或 receipt 持久化后构建 trust registry snapshot。
2. 写入顺序必须满足：
   - verdict/opinion/fairness 事实先锁定
   - trust registry 只读这些事实生成承诺
   - callback/report 不因 trust registry 非关键失败而伪造成功状态
3. 对写入失败明确错误语义：
   - 可重试的 registry write failure
   - registry pending
   - contract violation
4. trust read routes 支持 read-through：
   - registry present：返回 durable snapshot
   - registry missing：若仍在短期过渡期，可从 receipt 构建并标注 `source=derived_from_receipt`
   - P36 收尾前评估是否移除 fallback

**DoD**

1. phase/final 主路径可产出 durable trust snapshot。
2. trust routes 合同测试覆盖 registry source、derived source、missing source。
3. trust 写入不能改变公开 winner、draw/review 结果。

## 8. 架构方案第13章一致性校验

1. **角色一致性**：P36 不新增或合并 Judge App 8 Agent 官方职责；trust layer 只承诺和验证 Agent 输出事实，不参与裁决。
2. **数据一致性**：P36 以六对象和 workflow/receipt 为输入，新增 trust registry 与 artifact refs；不得引入平行 winner、平行 verdict 或不可追踪 JSON 袋。
3. **门禁一致性**：Fairness Sentinel 仍在 Chief Arbiter 前，trust registry 不得绕过 fairness gate 修改结果；challenge/review 必须保留可审计状态机。
4. **边界一致性**：NPC/Room QA 保持 `advisory_only`，不写 trust registry 官方链路，不触发 public verdict attestation。
5. **跨层一致性**：trust/public verify/ops read model 字段变更必须同步检查 `chat_server` 与前端 ops domain；未发布能力默认 hard-cut。
6. **收口一致性**：P36 继续区分 `local_reference_ready` 与真实环境 `pass`；当前没有真实环境时，只执行本地参考和 on-env 补证准备，不宣称 real-env 已通过。

## 9. 风险与处理策略

| 风险 | 表现 | 处理策略 |
| --- | --- | --- |
| Trust Registry 变成第二条裁决链 | public verify 或 attestation 反向覆盖 winner | 明确 trust 只读 verdict ledger；合同测试禁止写 verdict facts |
| registry 与 receipt/ledger 不一致 | 同案不同哈希、不同 trace 或 challenge count 漂移 | upsert 前做 component consistency 校验，ops read model 暴露 mismatch |
| Public Verify 泄露内部信息 | raw prompt、trace、fairness 细节或身份字段进入公开 payload | 建立 redaction contract 与禁止字段测试，默认 fail-closed |
| Artifact Store 伪装生产对象存储 | 本地文件证据被写成 S3/MinIO pass | 文档统一标注 local artifact；生产对象存储验收后置 |
| `app_factory.py` 继续膨胀 | P36 trust/ops 装配把入口文件重新堆厚 | dependency bootstrap 下沉，app_factory 只保留薄装配 |
| `ops_read_model_pack.py` 继续膨胀 | trust/artifact coverage 继续堆进单体函数 | 新增 trust/artifact read model helper，ops pack 聚合调用 |
| 真实环境缺失导致误报 | local reference 被当成 pass | 所有证据使用 local reference 口径，real-env/on-env 独立后置 |
