# API067 开发计划（slot: default）

更新时间：2026-04-09
计划来源：`docs/dev_plan/当前开发文档.md`
目标接口：`POST /api/debate/ops/sessions/{id}/judge/rejudge`

---

## 0. 计划目标

1. 先完成 S1（最小止血）：让 API067 在契约、边界、安全、可观测、测试层面达到“可稳定维护”的基线。
2. 再推进 S2（语义硬切设计与落地）：将“重判触发”升级为“真重判版本化能力”。
3. S3 作为上线前收口：限流、幂等、压测、告警与故障演练闭环。

---

## 1. 范围与边界

本计划包含：

1. API067 相关后端路由、模型、OpenAPI、测试、文档治理。
2. 与 `judge_phase_jobs/judge_final_jobs/judge_final_reports` 相关的数据模型调整（S2）。
3. 观测、审计、限流、幂等和稳定性收口（S3）。

本计划不包含：

1. 无关 API 的大规模重构。
2. 非 API067 主路径的 UI 视觉改造。

---

## 2. 执行策略（默认）

1. `slot default` 本轮先执行 S1 全量。
2. S2 在 S1 完成且评审通过后进入实现。
3. S3 默认后置为发布前治理，不与 S1/S2 混做。

原因：

1. S1 能快速收敛风险并稳定后续开发基线。
2. S2 是结构性改造，变更面大，需要更清晰的中间里程碑。
3. S3 依赖真实流量/压测窗口，提前做收益有限。

---

## 3. 里程碑与任务拆解

## M1：S1 最小止血（本轮默认执行）

### S1-01 OpenAPI 错误谱系补齐

目标：

1. 让 API067 OpenAPI 与运行时错误谱系一致。

改动点：

1. `chat/chat_server/src/handlers/debate_ops.rs`

任务：

1. 在 `request_judge_rejudge_ops_handler` 的 `#[utoipa::path]` 补齐 `400/401/403/500` 响应声明。
2. 保留现有 `202/404/409`。

完成定义（DoD）：

1. OpenAPI 文档可见完整错误状态码集合。
2. 与中间件/模型运行时行为一致。

---

### S1-02 输入边界治理（`session_id` checked 转换）

目标：

1. 移除 `u64 as i64` 的隐式语义风险。

改动点：

1. `chat/chat_server/src/models/judge/request_report.rs`

任务：

1. 在 `request_judge_job_internal` 入口将 `session_id` 转换为 `session_id_i64`（checked）。
2. 超界时返回稳定 `400` 业务错误码（建议：`ops_judge_rejudge_session_id_out_of_range`）。
3. 统一替换后续 SQL bind 里的 `session_id as i64`。

完成定义（DoD）：

1. 超界输入返回稳定 `400`，不再依赖隐式截断。
2. 正常路径行为不变。

---

### S1-03 Handler 结构化观测补齐

目标：

1. 建立 API067 成功/失败路径的结构化日志。

改动点：

1. `chat/chat_server/src/handlers/debate_ops.rs`

任务：

1. 在 handler 增加 `request_id/session_id/user_id/latency_ms/result` 等字段。
2. 成功分支记录 `status/reason/queued_phase_jobs/queued_final_job/trigger_mode`。
3. 失败分支记录分类结果（`not_found/conflict/validation/auth/server_error`）。

完成定义（DoD）：

1. 成功与失败都有可检索结构化日志。
2. 排障时可直接关联“请求输入 -> 业务输出 -> 失败类型”。

---

### S1-04 route/model 专项回归测试补齐

目标：

1. 建立 API067 的最小回归门禁，防止语义回退。

改动点：

1. `chat/chat_server/src/handlers/debate.rs`
2. `chat/chat_server/src/models/judge/tests/request_judge_job.rs`（或同模块新增测试文件）

任务：

1. route 用例：
   1. 未登录 `401`
   2. 未绑手机 `403`
   3. 无 `JudgeRejudge` 权限 `409`
   4. session 不存在 `404`
   5. 缺少历史报告 `409`（`judge_request_rejudge_requires_existing_report`）
   6. 成功受理 `202`
2. model 用例：
   1. 权限拒绝
   2. 状态不允许（`judge_request_session_not_allowed`）
   3. `require_existing_report` 拒绝
   4. 超界 `session_id` 返回稳定 `400` 错误码

完成定义（DoD）：

1. route/model 两层都有 API067 专项测试组。
2. 能覆盖关键失败分支与成功主路径。

---

### S1-05 文档同步与收口

目标：

1. 让学习文档、开发文档、实际实现三者一致。

改动点：

1. `docs/learning/Restful_Api/067_post_api_debate_ops_sessions_id_judge_rejudge.md`

任务：

1. 若代码行为与文档有新增差异，回写学习文档。
2. 在开发文档补充本轮已解决/未解决清单。

完成定义（DoD）：

1. 文档无“全量重判必然产出新报告”类错误叙述。
2. 风险、方案、当前状态可被复核。

## M2：S2 语义硬切（评审后执行）

### S2-01 数据模型版本化设计

目标：

1. 支持同一 session 多次 rejudge 且保留历史。

任务：

1. 设计 `rejudge_run_id`（或 `judge_run_no`）字段。
2. 调整唯一约束：
   1. `judge_phase_jobs` 从 `(session_id, phase_no)` 到 `(session_id, rejudge_run_id, phase_no)`
   2. `judge_final_jobs` 从 `(session_id)` 到 `(session_id, rejudge_run_id)`
   3. `judge_final_reports` 从 `(session_id)` 到 `(session_id, rejudge_run_id)`
3. 设计迁移与历史数据回填策略（默认 run=1）。

DoD：

1. schema 支持多 run。
2. 历史数据可平滑迁移。

---

### S2-02 API067 真重判写路径

目标：

1. 每次 API067 调用创建新 run 并触发新的 phase/final 任务链。

任务：

1. `request_judge_rejudge_by_owner` 改为“创建新 run + 新 trace/idempotency 体系”。
2. 保证 run 级审计可追踪（`requested_by/request_id/run_id/reason/result`）。
3. 明确并发策略：同 session 同时重判的互斥规则。

DoD：

1. 同一 session 多次调用可产生多条 run。
2. 每条 run 可独立追踪执行与结果。

---

### S2-03 查询面与兼容收敛

目标：

1. 默认查询返回最新 run，且支持历史 run 回看。

任务：

1. `GET judge-report` 默认 latest run。
2. 增可选 run 查询参数（仅内部/Ops 先开放）。
3. 更新 OpenAPI 与前端契约映射。

DoD：

1. 最新态消费不受影响。
2. 历史态可查且口径一致。

---

### S2-04 S2 测试与迁移演练

任务：

1. 增加 run 维度 unit/integration 回归。
2. 迁移前后数据一致性校验脚本。

DoD：

1. run 模型端到端可回归。
2. 迁移演练可复现且可回滚。

## 5. 风险与回滚策略

主要风险：

1. S2 迁移改动面大，可能影响现有报告查询。
2. run 版本化可能引入新的一致性边界问题。

回滚策略：

1. S1 改动保持可原子回退（代码级回滚即可）。
2. S2 使用“迁移脚本 + 兼容读策略短窗口 + 明确回滚 SQL”方案。
3. 任一阶段若关键回归失败，立即停止后续阶段推进。
