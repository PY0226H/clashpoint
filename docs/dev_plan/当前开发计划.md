## 5. 执行顺序与依赖

1. 先推进 `ai-judge-p10-registry-triple-productization`，因为 fairness gate v2 与 panel profile 都依赖统一 registry。
2. 再推进 `ai-judge-p10-fairness-release-gate-v2`，把“可发布”与“公平可接受”绑定。
3. 然后推进 `ai-judge-p10-panel-runtime-profile-v2`，补齐多 panel 独立运行证据。
4. 接着推进 `ai-judge-p10-case-fairness-read-model`，确保 Ops 可读可追溯。
5. 再落 `ai-judge-p10-agent-entrypoints-shell`，建立 NPC/Room QA 入口但不侵入裁决主链。
6. 最后在环境可用时收口 `ai-judge-p10-real-env-hard-pass-closure`。

## 8. 风险与对策

1. 风险：registry 三分化后，调用链更复杂，易出现解析不一致。  
   对策：所有 runtime 统一走单一 registry resolver；新增端到端契约测试覆盖 phase/final/replay。
2. 风险：fairness gate 绑定发布可能抬高发布门槛，影响研发效率。  
   对策：保留可审计 override 机制，但默认阻断；所有 override 必须落审计原因与责任人。
3. 风险：panel profile 独立会抬升时延与成本。  
   对策：先做可配置 profile，不默认开启高成本策略；通过 runtime SLA freeze 持续观测。
4. 风险：real-env 长期不可用，导致 P10 末端持续阻塞。  
   对策：保持 local_reference 收口证据持续刷新，等待环境窗口时一键复跑并收口为 pass。
