## 2. 本轮主线定位

P43 的目标不是直接把 NPC Coach / Room QA 上线为真实智能助手，也不是补真实环境 pass。

P43 的目标是把 P42 的 deterministic placeholder 推进到“真实 LLM executor 可安全接入”的 readiness 状态：

1. 明确真实 assistant executor 的配置开关、模式、模型、超时、重试与成本预算。
2. 建立 NPC Coach / Room QA 的 prompt bundle 与结构化输出 schema。
3. 让真实 LLM executor 只通过 `LLM Gateway` 调用 provider。
4. 保持 `assistant_advisory_contract_v1` 不变，所有输出仍是 advisory-only。
5. 在 provider 缺失、预算超限、超时、输出越权、结构化解析失败时 fail-closed。
6. 增加本地 fake/canary 证据，证明 chat/frontend 已可消费 ready advisory output。
7. 不写 `verdict_ledger`，不写 `judge_trace`，不触发 `Fairness Sentinel / Chief Arbiter`，不改变官方裁决链。

一句话：

`P43 要把 Interactive Guidance Plane 从“可展示的安全占位符”推进到“可接入真实 LLM 的安全运行骨架”，但仍不能把它变成官方裁决能力。`

## 3. 架构方案一致性校验

| 检查项 | P43 结论 |
| --- | --- |
| Official Verdict Plane 隔离 | 不改变 Judge App 8 Agent 官方裁决主链；NPC Coach / Room QA 不替代 Clerk / Recorder / Claim / Evidence / Panel / Sentinel / Arbiter / Opinion |
| Interactive Guidance Plane 定位 | 继续只做辅助建议和房间问答；结果必须显式 `advisoryOnly=true` 与 `officialVerdictAuthority=false` |
| 统一门面 | 客户端仍只通过 `chat_server`，不直连 `ai_judge_service` |
| LLM Gateway | 所有真实模型调用必须走 `DefaultLlmGateway` / `LlmGatewayPort`，不得在 assistant executor 内直接散落 HTTP provider 调用 |
| Prompt / Policy Registry | NPC Coach / Room QA 使用独立 advisory policy 与 prompt version；不复用 official verdict prompt |
| 成本与延迟 | 本地阶段不冻结生产阈值，但必须支持预算字段、超时、重试、usage metadata 与 fail-closed |
| 真实环境 | 真实 provider/callback/对象存储/服务窗口未就绪时仍保持 `env_blocked`，不把 local canary 记为 real-env pass |
| 产品边界 | PRD 明确当前不是赛中陪聊型 AI 产品；P43 只能准备受控辅助入口，默认不扩大成实时陪聊、代打或无限制建议 |

## 9. 风险与边界

1. **真实环境风险**：没有真实 provider/callback/对象存储窗口时，不跑 real-env pass，不写 production-ready。
2. **产品定位风险**：NPC Coach / Room QA 不应变成赛中陪聊或代打建议；默认只提供受控辅助建议。
3. **裁决污染风险**：任何输出含胜负、评分、裁决理由、fairness gate、trust attestation 都必须 fail-closed。
4. **成本失控风险**：真实 executor 需要预算、timeout、max retries 与 usage accounting，不能只接 provider。
5. **跨层契约风险**：AI 输出 schema 变动必须同步 chat validator、OpenAPI、frontend domain guard 与测试。
6. **热点膨胀风险**：P43 的 executor/prompt/schema 必须拆 helper，不能继续把逻辑塞进 `agent_runtime.py` 或 `app_factory.py`。
