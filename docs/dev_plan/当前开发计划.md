# API058 完整开发计划（slot: default）

更新时间：2026-04-08
对应分析文档：`docs/dev_plan/当前开发文档.md`
目标接口：`GET /api/debate/ops/rbac/roles`
推进策略：方案 A（立即收口）+ 方案 B（治理统一）两段式

---

## 1. 计划目标与原则

### 1.1 总体目标

围绕 `RBACROLES-R1 ~ R12` 建立一份可直接执行的完整计划，分阶段完成：

1. **阶段 A（立即收口）**：先封住契约漂移与回归缺口，确保接口正确性与可验证性。
2. **阶段 B（治理统一）**：再收敛错误模型、语义规范、可观测与审计，提升长期可维护性。

### 1.2 执行原则

1. 先正确性后增强性：优先 P0，再推进 P1/P2。
2. 保持当前产品行为稳定：避免一次性改动导致前端/运维断裂。
3. 每个阶段都要求“可测试、可回滚、可追踪”。

---

## 2. 范围定义

### 2.1 本计划覆盖范围

1. `GET /api/debate/ops/rbac/roles` 的路由、中间件、服务层、OpenAPI、回归测试。
2. 与其直接相关的 RBAC 管理面一致性（`rbac/me`、`rbac/roles/{userId}` 的治理协同）。
3. 控制平面读接口的工程治理（错误模型、限流、指标、审计、runbook）。

### 2.2 本计划暂不覆盖

1. 多租户/多 owner 架构重构。
2. 大规模权限策略引擎化改造。
3. 全平台所有接口的一次性错误模型迁移（本计划先做 RBAC 管理面）。

## Phase A：立即收口（P0）

### A1. OpenAPI 契约对齐（已完成）

- 目标：消除文档与运行态状态码漂移。
- 改动：`list_ops_role_assignments_handler` responses 从 `200/409` 扩展到 `200/401/403/409/500`。
- 文件：
  - `chat/chat_server/src/handlers/debate_ops.rs`
- 状态：`[x] 已完成`

### A2. route 回归矩阵补齐（已完成）

- 目标：封住中间件 + handler 组合回归风险。
- 新增用例：
  1. owner -> `200`
  2. 无 token -> `401 auth_access_invalid`
  3. 未绑手机 -> `403 auth_phone_bind_required`
  4. 非 owner -> `409 ops_permission_denied:role_manage:*`
  5. owner 真源缺失 -> `500 platform_owner_not_configured`
- 文件：
  - `chat/chat_server/src/handlers/debate.rs`
- 状态：`[x] 已完成`

### A3. owner 真源异常恢复 runbook（已完成）

- 目标：降低 `platform_owner_not_configured` 值班恢复时间。
- 输出物：
  1. 标准恢复步骤（SQL/脚本/应用入口）
  2. 演练记录（至少一次）
- 建议落点：`docs/consistency_reports/` 或 `docs/harness/` 运行手册。
- 恢复步骤（冻结版）：
  1. 诊断：`SELECT owner_user_id, updated_by, updated_at FROM platform_admin_owners WHERE singleton_key = TRUE;`
  2. 快速恢复（推荐）：执行应用层入口 `grant_platform_admin(target_user_id)`，同步回填 `platform_user_roles + platform_admin_owners`。
  3. 紧急 SQL 恢复（仅应急）：向 `platform_admin_owners` 写入单行 `singleton_key=TRUE`，并保证 `owner_user_id/updated_by` 都是存在的用户 id。
  4. 恢复后验证：调用 `GET /api/debate/ops/rbac/roles` 与 `GET /api/debate/ops/rbac/me`，确认 owner 权限恢复。
- 演练证据：
  1. route 用例 `get_ops_rbac_roles_route_should_return_500_when_owner_source_missing` 已验证缺失场景可稳定复现。
  2. route 用例 `get_ops_rbac_roles_route_should_return_200_for_owner` 已验证恢复后可正常读列表。
- 状态：`[x] 已完成`

### A4. A 阶段验收门禁

1. OpenAPI 声明含 `200/401/403/409/500`。
2. route 用例全部通过。
3. runbook 可复用、可演练。

## 7. 风险、依赖与决策点

### 7.1 关键风险

1. `409 -> 403` 迁移可能影响前端既有分支。
2. 限流阈值设定不当会误伤运维操作。
3. 审计落库若设计不当会增加写放大。

### 7.2 外部依赖

1. 产品/安全对门禁策略的最终裁决。
2. 前端对结构化错误模型的同步消费。
3. 运维对 dashboard 与告警阈值的评审。

### 7.3 必须确认的决策

1. 权限拒绝是否长期保留 `409`（已确认：RBAC 管理面阶段内保持 `409`）。
2. RBAC 审计查询面是否维持“脚本级受控查询”还是升级为受控 API。
3. owner-only 场景下 PII 返回最小化边界。
