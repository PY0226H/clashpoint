## 4. 详细实施计划（按阶段）

### 4.1 阶段 S1（本轮必须完成，本地可执行）

#### `RBACDEL-S1-1` DELETE invalid If-Match 回归补齐（对应 `R13`）

- 目标：补齐删除路径 header 语法回归，防止 parser 变更引发静默回归。
- 计划改动：
  1. 在 `chat/chat_server/src/handlers/debate.rs` 新增用例：
     - `delete_ops_rbac_roles_user_id_route_should_return_400_when_if_match_invalid_and_record_audit`
  2. 断言：
     - HTTP 400
     - `code=ops_rbac_if_match_invalid`
     - `ops_rbac_audits` 与 `ops_rbac_audit_outbox_jobs` 均有对应失败分类（`validation_error`）。
- 验收：DELETE invalid `If-Match` 场景具备 route 级门禁。

#### `RBACDEL-S1-2` 错误优先级策略冻结（对应 `R12`）

- 目标：将当前优先级语义从“代码现状”升级为“契约承诺”，避免后续漂移。
- 计划改动：
  1. 在 `docs/learning/Restful_Api/060_delete_api_debate_ops_rbac_roles_user_id.md` 增加“错误优先级矩阵”小节：
     - `If-Match` invalid（早期 400）
     - `If-Match` missing（model 侧 400）
     - permission denied（409）
  2. 在 `chat/chat_server/src/handlers/debate.rs` 补 1~2 条断言用例锁定现有优先级（至少覆盖 non-owner + missing If-Match 的当前行为）。
- 验收：文档与测试双重锁定当前优先级，不再依赖隐式实现细节。

#### `RBACDEL-S1-3` DELETE 观测信号补强（对应 `R14`）

- 目标：为后续 dashboard/阈值封板提供可直接消费的结构化信号。
- 计划改动（二选一，优先最小风险方案）：
  1. 指标方案：在 `OpsRbacRolesWriteMetrics` 增加 revoke 细分计数（`removed_true/removed_false`、`if_match_invalid/missing`）。
  2. 日志方案：在现有 `tracing` 里稳定输出上述字段并补文档字典。
- 验收：DELETE 关键分支在日志或指标层可直接统计，无需 SQL 反推。

### 4.2 阶段 S2（本轮建议完成“设计评审”，代码可后置）

#### `RBACDEL-S2-1` strict delete 模式评估（对应 `R8`）

- 目标：判断是否需要“幂等模式 + 严格模式”双语义。
- 计划产出：
  1. 设计草案：触发方式（query/header）、兼容性影响、错误码策略。
  2. 业务评审结论：是否纳入后续里程碑。
- 验收：有明确 Go/No-Go 决策，不处于长期悬空状态。

#### `RBACDEL-S2-2` DELETE 审计去噪评估（对应 `R7`）

- 目标：基于数据决定是否引入 DELETE 幂等协商或审计聚合去噪。
- 计划产出：
  1. 采样统计：重试率、重复 success 审计占比。
  2. 对比方案：`idempotency-key` vs 审计聚合去噪。
  3. 结论：保持现状 / 进入实现。
- 验收：形成评审材料并给出可执行结论。

### 4.3 阶段 S3（环境依赖，先准备执行清单）

#### `RBACDEL-S3-1` trusted proxy 生产白名单与 runbook（对应 `R10/R14`）

- 目标：完成生产可用的来源信任边界治理闭环。
- 计划动作：
  1. 维护 `trusted_proxy_ids/cidrs` 清单。
  2. 执行误配/缺配演练并沉淀 runbook。
- 验收：预发演练通过且有证据归档。

#### `RBACDEL-S3-2` forwarded-header 治理跨接口复用（对应 `R10`）

- 目标：收敛 auth/payment/ticket 控制面写路径来源信任边界。
- 计划动作：
  1. 输出接口清单与改造顺序。
  2. 分批落地并回归验证。
- 验收：跨接口策略一致，不再出现“同类写接口边界不一致”。

#### `RBACDEL-S3-3` RBAC 写链路观测与压测封板（对应 `R14`）

- 目标：形成上线前可复核基线。
- 计划动作：
  1. 形成 `ops.rbac.roles_write.*` 看板与告警阈值。
  2. 产出压测报告（冲突率、限流率、投递时延）。
- 验收：有 dashboard 导出、阈值说明、压测报告三件套。
