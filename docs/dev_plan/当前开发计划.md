## 5. 一致性校验

1. **角色一致性**：虚拟裁判 NPC 是娱乐导向公开房间角色，不是赛后 AI 裁判团成员，也不替代正式裁决报告。
2. **数据一致性**：`chat` 是 NPC action 的房间事实源；`npc_service` 不直接写 DB、不直接广播 WS。
3. **门禁一致性**：候选动作必须经过 `npc_service` guard 与 `chat` 二次校验；LLM 输出不能绕过 schema、限频、幂等和 forbidden fields。
4. **边界一致性**：`llm_executor_v1` 是主路径，`rule_executor_v1` 只是 fallback；`NPC Coach` / `Room QA` 不恢复。
5. **跨层一致性**：新增 API/DTO/WS payload 时同步检查 chat、notify、realtime-sdk、debate-domain、app-shell 与测试。
6. **收口一致性**：阶段完成后再写 completed/todo；当前只更新 active 计划，不写长期完成或债务条目。
