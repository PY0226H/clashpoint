## 背景与目标

当前 `PUT /api/debate/ops/topics/{id}` 已具备基础权限与字段校验，但相比创建接口（053）在写路径治理上存在缺口（重复辩题治理、并发保护、审计、契约一致性、回归覆盖）。

本轮目标：在不引入大规模架构改造的前提下，完成该接口的第一轮治理增强，达到“可回归、可审计、可并发防护、契约可联调”的工程基线。

## 范围说明

### 本轮纳入

1. 更新链路重复辩题冲突治理（排除自身）。
2. 乐观并发保护（revision/updatedAt 预期值校验）。
3. 更新操作审计落库。
4. OpenAPI 响应契约补齐（与真实运行态对齐）。
5. 边界输入安全转换（`u64 -> i64`）。
6. 相关模型/路由测试补齐。
7. 学习文档与开发文档对齐回写。

### 本轮不纳入

1. category 字典配置中心化。
2. 全局错误模型 `{code,message}` 统一改造。
3. 更新接口完整限流/幂等框架接入（后续迭代）。

## 验收标准（DoD）

1. `PUT` 无法通过更新制造同分类重复标题。
2. 并发覆盖可被检测并返回稳定冲突错误。
3. 更新成功写入审计记录。
4. OpenAPI 与运行态主要错误谱系一致。
5. 相关测试通过，且无编译回归。

## 验证策略

1. 单元/模型测试：`models::debate::tests::ops_and_listing`。
2. 必要时路由测试：`handlers::debate::*` 中与 topic ops 相关用例。
3. 静态检查：`cargo test -p chat-server <target>`（最小范围先行）。

## 风险与回退

1. 风险：新增并发控制字段可能影响旧调用方。
   - 处理：字段设计为可选，保持向后兼容；未传时维持原行为。
2. 风险：审计约束迁移在历史库存在差异。
   - 处理：migration 使用显式 drop/add check constraint 并保证幂等执行路径。
3. 风险：去重规则引入新 409，前端未适配。
   - 处理：沿用已存在冲突码风格，文档补充示例与联调提示。
