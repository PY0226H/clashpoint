### S0-1 口径设计冻结

- 动作：冻结新增/调整字段语义，避免实现中途漂移。

字段口径冻结（本轮目标）：

1. `timedOutFailedTotal`：仅统计请求超时失败。

4. （可选）新增细分：`phaseTickErrorTotal`、`finalEnqueueErrorTotal`、`finalTickErrorTotal`。

### S0-2 改造边界确认

- 只改 `chat_server` 与文档，不改前端行为。
- 不引入兼容层，直接硬切到新口径（符合当前 pre-release 规则）。

验收：

2. 输出 JSON 字段为 `camelCase`。

### S1-2 派发错误映射修正

涉及文件：

1. `chat/chat_server/src/models/judge_dispatch.rs`
2. `chat/chat_server/src/models/judge_dispatch/dispatch_worker.rs`

实现项：

1. 增加 timeout 可识别错误码（建议 `DispatchFailureCode::Timeout`）。
2. 在 `send_phase_dispatch_request`、`send_final_dispatch_request` 中基于 `reqwest::Error::is_timeout()` 归类 timeout。
3. `HttpUnexpectedStatus` 不再计入 5xx，单独计入 `failed_http_unexpected`。
4. phase/final report 汇总时填充 `timed_out_failed`。

验收：

1. timeout 失败可驱动 `timedOutFailedTotal` 增长。
2. unexpected status 与 5xx 指标分离。

### S1-3 Worker 错误计数覆盖修正

## 5. 风险、回滚与决策点

### 5.1 主要风险

1. 新增字段导致下游解析器严格校验失败。
2. timeout 归因策略与现有看板口径冲突。
3. enqueue error 纳入后告警阈值短时突变。

### 5.2 回滚策略

1. 代码层：按提交粒度回滚（先回滚 S2，再回滚 S1 非核心项）。
2. 契约层：保留原有字段不删除，新字段仅增不减，保证向后兼容。
3. 运维层：告警阈值先观察一轮再收紧。

### 5.3 需要你确认的决策（执行前）

1. `tickErrorTotal` 是否保持单总数，还是本轮直接引入三段细分字段。
2. `failedHttpUnexpectedTotal` 是否立即对外暴露，还是先内部统计后再开放。

默认建议（若无额外指示）：

1. 本轮直接对外暴露细分字段，避免继续黑盒。
