## 1. 目标与范围

本轮聚焦 `RBACROLES-R1/R2/R3` 的 P0 收口，并确保可验证落地：

1. 补齐接口 OpenAPI 失败响应契约（`401/403/500`）。
2. 增加 route 级回归矩阵，覆盖 `200/401/403/409/500`。
3. 在计划文档中沉淀 owner 真源缺失场景的恢复策略说明。

本轮不包含：

1. 统一错误模型重构（`code/message/details` 全平台化）。
2. 角色列表接口分页化与 revision 升级。
3. 读取审计落库与完整可观测看板建设。

---

## 2. 执行计划（按阶段）

### Phase A：契约对齐（P0）

1. 修改 `list_ops_role_assignments_handler` 的 OpenAPI responses：
   - 新增 `401`（鉴权失败）
   - 新增 `403`（手机门禁失败）
   - 新增 `500`（内部错误）
2. 保留现有 `200/409` 语义不变，避免跨端行为突变。

### Phase B：route 回归矩阵补齐（P0）

在 `chat/chat_server/src/handlers/debate.rs` 新增 `GET /api/debate/ops/rbac/roles` 路由测试：

1. owner 且已绑手机：返回 `200`。
2. 缺失 token：返回 `401 auth_access_invalid`。
3. 未绑手机：返回 `403 auth_phone_bind_required`。
4. 非 owner 已绑手机：返回 `409 ops_permission_denied:role_manage:*`。
5. owner 真源缺失：返回 `500 platform_owner_not_configured`。

### Phase C：验证与回填（P0）

1. 运行针对性测试集验证新增用例。
2. 回填本计划文档的执行状态、证据与结论。

### Phase D：风险说明（文档收口）

1. 在计划里补充 owner 真源异常恢复建议：
   - 通过 `grant_platform_admin` 或写入 `platform_admin_owners` 恢复单行真源。
2. 明确未完成项进入下一轮（错误模型/限流/可观测/审计）。

## 3. 验收标准（DoD）

1. OpenAPI 响应声明覆盖 `200/401/403/409/500`。
2. 新增 route 测试通过并覆盖上述状态码路径。
3. owner 真源缺失场景具备可复现测试与恢复说明。
4. 现有语义（非 owner 返回 409）不被破坏。

---

## 4. 风险与回滚

1. 风险：若后续要把权限拒绝状态码改为 `403`，会影响前端既有 `409` 分支。
2. 风险：owner 真源被误删会导致角色管理读写统一 `500`。
3. 兜底：可通过 `grant_platform_admin(user_id)` 自动回填 owner 真源。
4. 回滚：若新增测试暴露历史契约问题，可先保留测试并临时标注预期，避免静默放过回归。
