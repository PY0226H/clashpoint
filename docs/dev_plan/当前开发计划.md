## 3. 风险映射与优先级策略

## 4. 详细实施计划（按阶段）

### 4.1 阶段 S0（本迭代必做）

#### `RBACPUT-S0-1` 输入边界修复（对应 `R2`）

- 目标：消除 `u64 -> i64 as` 截断风险。
- 计划改动：
  1. 在 `models/rbac.rs` 里对 path `user_id` 做 `i64::try_from` 安全转换。
  2. 为越界场景定义稳定 machine code（如 `ops_role_target_user_id_out_of_range`）。
  3. 统一映射到可预期状态码（建议 400）。
- 验收：超大 `user_id` 不再出现语义漂移，错误码稳定可机读。

#### `RBACPUT-S0-2` 原子写路径收敛（对应 `R3`）

- 目标：消除“预检通过后目标被删除”的竞态窗口。
- 计划改动：
  1. 将“用户存在校验 + UPSERT”收敛为单 SQL 原子路径（CTE 或等价方案）。
  2. 保持当前响应结构与 UPSERT 语义不变。
- 验收：目标用户不存在稳定返回 404，避免外键异常漂移到 5xx。

#### `RBACPUT-S0-3` Owner 真源守护（对应 `R1`）

- 目标：降低 `platform_admin_owners` 缺失导致的控制面硬故障概率。
- 计划改动：
  1. 增加启动/健康检查层面对 owner 真源的显式守护或告警。
  2. 保持运行态错误码可识别（`platform_owner_not_configured`）。
- 验收：owner 真源异常可提前暴露，不依赖业务请求触发后才发现。

#### `RBACPUT-S0-4` 回归测试补齐（对应 `R13`）

- 目标：补齐关键边界回归，形成 CI 防线。
- 计划改动（`handlers/debate.rs` route tests）：
  1. PUT owner 真源缺失 -> 500。
  2. PUT IP 限流触发 -> 429。
  3. PUT 超大 user_id -> 400 + 稳定 code。
- 验收：新增用例稳定通过，且失败时可清晰定位。

#### `RBACPUT-S0-5` 契约同步（对应 `R13` 延伸）

- 目标：确保实现与文档、OpenAPI、前端认知一致。
- 计划改动：
  1. 如有新增 code/status，更新 OpenAPI 注解。
  2. 回写学习文档与开发文档差异说明。
- 验收：文档、注解、运行行为三方一致。

#### `RBACPUT-S2-1` 委派式角色管理（对应 `R4`）

- 目标：降低 Owner 单点瓶颈，同时保持权限边界可控。

  2. `role_manage` 判定从“仅 Owner”扩展为“Owner 或 delegated role admin”。

  4. 补齐 route/model 回归：委派管理员 list/put/delete 成功链路 + 边界拒绝链路。

#### `RBACPUT-S2-2` 审计可靠投递（对应 `R9`）

- 目标：将 RBAC 审计从“仅 best-effort 直写”演进到“可回放 outbox”。
- 执行结果：已完成
  1. 新增 `ops_rbac_audit_outbox_jobs` 表与索引（due/locked/event）。
  2. 审计写入改为 `enqueue -> dispatch_once`，失败任务自动回退到 `next_retry_at`。
  3. 保留直写兜底：outbox enqueue 失败时回退 legacy insert，避免迁移窗口丢失日志。
  4. route 回归新增 outbox 断言：成功写与权限拒绝场景均可在 outbox 中检索到对应记录。

#### `RBACPUT-S2-3` Extractor 失败观测补齐（对应 `R11`）

- 目标：把 415/422 提取失败纳入 RBAC 写链路专项观测与审计。
- 执行结果：已完成
  1. PUT handler 改为显式处理 `JsonRejection`，将 `missing content-type` / `invalid json` 映射为稳定 machine code。
  2. 提取失败分支接入 `ops_rbac_audit_outbox_jobs`，并同步落 `ops_rbac_audits`，failure_reason 统一为 `validation_error`。
  3. 写接口指标新增 `ops_rbac_roles_write_extractor_rejected_total`，与成功/失败/限流指标同层输出。
  4. route 回归新增 2 条：`415` 与 `422`，并断言审计表与 outbox 均记录对应失败分类。

### 4.4 阶段 S3（并发协议与治理收口）

#### `RBACPUT-S3-1` 条件写并发冲突协议（对应 `R5`）

- 目标：让控制平面写冲突可见化，避免静默覆盖。
- 执行结果：已完成
  1. model 层 PUT/DELETE 新增 `expected_rbac_revision`，事务内执行 revision 校验。
  2. 通过事务级 advisory lock 串行化 `If-Match` 校验与写入，减少并发窗口漂移。
  3. 版本冲突稳定返回 `409 ops_rbac_revision_conflict`，并落审计/outbox（`failure_reason=conflict`）。
  4. 非法 `If-Match` 稳定返回 `400 ops_rbac_if_match_invalid`，并落审计/outbox（`failure_reason=validation_error`）。
  5. DELETE 成功响应补齐 `x-rbac-revision`，与 PUT 行为对齐。
  6. route 回归新增 4 条：PUT 成功匹配、PUT 冲突、PUT 非法头、DELETE 冲突。

#### `RBACPUT-S3-2` 审计 outbox worker 化（对应 `R9`）

- 目标：摆脱“仅请求驱动 dispatch_once”的重试推进限制，确保低流量下 outbox 也能持续收敛。
- 执行结果：已完成
  1. runtime worker 新增 `spawn_ops_rbac_audit_outbox_worker`，按 `worker_runtime.ops_rbac_audit_outbox_poll_interval_secs` 周期执行。
  2. AppState 新增 `retry_ops_rbac_audit_outbox_once(batch_size)`，沉淀 worker tick 的统一入口与报告结构。
  3. 请求快路径继续保留 `enqueue -> dispatch_once`，实现“请求内快投递 + 后台重试推进”双轨。
  4. 配置新增：
     - `worker_runtime.ops_rbac_audit_outbox_worker_enabled`（默认 `true`）
     - `worker_runtime.ops_rbac_audit_outbox_poll_interval_secs`（默认 `1` 秒）
  5. 回归新增：`ops_rbac_audit_outbox_worker_once_should_deliver_pending_jobs`。

#### `RBACPUT-S3-3` Owner 自写语义提示收口（对应 `R12`）

- 目标：把“Owner 自写角色不影响 Owner 权限”的语义显式机读化，避免控制台误读。
- 执行结果：已完成
  1. PUT/DELETE 在 Owner 对自身写角色时回写 `x-rbac-warning=owner_self_role_assignment_no_effect`。
  2. warning 判定走后端 owner 真源（`platform_admin_owners`），不依赖前端猜测。
  3. route 回归新增 2 条 owner-self warning 断言。

#### `RBACPUT-S3-4` 控制台默认条件写（对应 `R5` 残余风险）

- 目标：收敛“控制台漏带 If-Match 导致无条件覆盖写”的残余风险。
- 执行结果：已完成
  1. `ops-domain` 角色写 SDK 改为必须传 `expectedRevision`，并统一携带 `If-Match` 请求头。
  2. Ops Console 默认用 `roleAssignments.rbacRevision`（回退 `rbacMe.rbacRevision`）作为条件写 revision。
  3. revision 未就绪时，Grant/Revoke 按钮禁用并提示，避免漏发条件写。
  4. `ops_rbac_revision_conflict` 时自动刷新 RBAC 快照并提示重试。
  5. 前端回归：`ops-domain/app-shell` test+typecheck+lint 全通过，`@smoke` E2E 通过。

#### `RBACPUT-S3-5` IP `unknown` 桶治理（对应 `R8`）

- 目标：降低无真实 IP 头时的跨用户共享 `unknown` 桶误伤。
- 执行结果：已完成
  1. 新增 `request_rate_limit_ip_key_with_user_fallback(headers, user_id)`：有真实 IP 时沿用 IP 哈希；无真实 IP 时改为 `unknown_user_scope:{user_id}` 哈希键。
  2. Ops 相关接口统一接入 user-scoped fallback（topics/sessions/rbac），避免多个操作者挤占同一 `unknown` 桶。
  3. 新增 `request_guard` 单测 2 条，覆盖“优先真实 IP”与“无 IP 按用户隔离”。
  4. 定向回归通过：`put_ops_rbac_roles_user_id_route_should_return_429_when_ip_rate_limited`。

#### `RBACPUT-S3-6` 成功审计同事务 outbox（对应 `R9` 残余窗口）

- 目标：把“RBAC 成功写 + 审计 outbox 入队”收敛到同事务，避免业务成功但审计缺失。
- 执行结果：已完成
  1. `upsert_ops_role_assignment_by_owner_with_meta` 与 `revoke_ops_role_assignment_by_owner_with_meta` 新增 `success_request_id` 参数，成功路径在同一 DB 事务内写入 `ops_rbac_audit_outbox_jobs`。
  2. PUT 幂等 replay 场景保持原语义：`replayed=true` 不重复入 success outbox，避免重复审计。
  3. handler 成功分支移除 `insert_ops_rbac_audit_log_best_effort` 直入路径，仅在提交后执行 `dispatch_ops_rbac_audit_outbox_once`，保持低延迟投递。
  4. 新增 model 回归 3 条（upsert success 入队、replay 不重复入队、revoke success 入队）并通过；RBAC 路由回归组 28 条通过。

#### `RBACPUT-S3-7` 服务端强制 If-Match（对应 `R5` 残余窗口）

- 目标：补齐非控制台调用方条件写治理，避免绕过前端 SDK 的无条件覆盖写。
- 执行结果：已完成
  1. model 层新增 `ops_rbac_if_match_required` 约束：API 路径缺失 `If-Match` 时稳定返回 `400` + machine code。
  2. `OpsRbacUpsertMeta/OpsRbacRevokeMeta` 新增 `require_if_match` 开关，仅对 RBAC API 写路径强制；内部调用/测试 seed 不受影响。
  3. PUT/DELETE 路由回归新增“缺失 If-Match”两条，断言错误码 + 审计/outbox 落库（`failure_reason=validation_error`）。
  4. 既有 RBAC 路由回归组扩展为 30 条并全部通过。

#### `RBACPUT-S3-8` 可信代理 IP 来源治理（对应 `R8` 残余窗口）

- 目标：收敛“客户端伪造转发头可影响 IP 限流分桶”的来源信任边界。
- 执行结果：已完成
  1. 配置层新增 `server.forwarded_header_trust`（`trusted_proxy_ids/trusted_proxy_cidrs`），用于声明可被信任的代理链路。
  2. `request_guard` 新增可信判定门控：仅当命中 trusted proxy 白名单时，才消费 `x-forwarded-for/x-real-ip` 生成 IP 限流 key。
  3. 未命中可信代理时统一回退 `unknown_user_scope:{user_id}` 哈希 key，避免伪造头污染限流桶。
  4. Ops 路径（topics/sessions/rbac）统一接入该治理策略，避免接口间信任边界漂移。
  5. `request_guard` 新增 3 条单测，覆盖 proxy-id trusted / CIDR trusted / untrusted fallback 三类语义。
