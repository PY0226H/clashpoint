## 3. 本轮北极星与不可变边界

1. 官方裁决结果仍只能是 `pro / con / draw`；高风险场景进入 `review_required` 或 `blocked_failed`，不得静默强判。
2. `Fairness Sentinel -> Chief Arbiter -> Opinion Writer` 的顺序不能绕过；Opinion 只能消费已锁定的 `verdict_ledger`，不得直接基于原始 prompt 改写最终事实。
3. 六对象主链必须收敛到明确事实源：`case_dossier / claim_graph / evidence_ledger / verdict_ledger / fairness_report / opinion_pack`。
4. `NPC Coach / Room QA` 仅能复用平台层能力，输出必须保持 `advisory_only`，不得写 `verdict_ledger`、不得影响官方裁决链。
5. 客户端仍只通过 `chat_server` 间接使用 AI 能力；`ai_judge_service` 不暴露公众直连接口。
6. Redis 只能做幂等锁、短 TTL 缓存、速率限制和临时上下文；不得作为最终裁决、复核或 replay 的唯一事实源。
7. 不引入 LangChain/LangGraph 作为生产主骨架；主链继续使用显式 Python 类、typed DTO、显式状态机和可审计工具调用。
8. 第一阶段继续保留 FastAPI、Pydantic v2、SQLAlchemy 2.0、Alembic、Milvus + BM25 + rerank，但 Milvus 必须藏在统一 Knowledge Gateway 后面。
9. 真实环境结论与本地参考结论分层表达；没有真实环境前，不宣称 real-env `pass`。

## 9. 架构方案第13章一致性校验

1. **角色一致性**：P35 明确以 `Clerk -> Recorder -> Claim Graph -> Evidence -> Panel -> Fairness Sentinel -> Chief Arbiter -> Opinion Writer` 为 Judge App 官方链路，不允许把 Evidence、Fairness 或 Opinion 折叠为不可观察的单 pipeline。
2. **数据一致性**：P35 主体围绕 `case_dossier/claim_graph/evidence_ledger/verdict_ledger/fairness_report/opinion_pack` 六对象展开，并计划将当前混合 JSON 事实逐步收敛到可追踪 ledger/store 边界。
3. **门禁一致性**：P35 要求 Fairness Sentinel 必须先于 Chief Arbiter，证据不足、panel 分歧、身份泄漏、稳定性不足时优先触发 draw/review/blocked_failed，不允许静默强判。
4. **边界一致性**：P35 中 NPC Coach 与 Room QA 只建立 advisory-only 外壳和 shared platform core 复用边界，不写 `verdict_ledger`，不触发官方裁决链，不污染 Judge App。
5. **跨层一致性**：P35 任何 API/DTO/callback/error/status 变更都要求同轮检查 `ai_judge_service`、`chat_server` 代理、前端 ops/domain SDK 与必要测试，不保留长期 alias 或双 payload。
6. **收口一致性**：P35 区分 `local_reference_ready` 与真实环境 `pass`；当前没有真实环境时，只执行本地参考和 on-env 补证准备，不宣称 real-env 已通过。

## 10. 风险与处理策略

| 风险 | 表现 | 处理策略 |
| --- | --- | --- |
| 继续在 `phase_pipeline.py` 叠功能 | 裁判主链继续难测、难审计 | 每批将新能力落到 role runtime 或 domain/core，不把新主逻辑写回 monolith |
