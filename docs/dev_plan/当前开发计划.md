## 3. 架构方案第13章一致性校验

1. **角色一致性**：P39 不改 8 Agent 官方裁决顺序；新增的 public verification proxy、citation verifier 与 release evidence artifact 都服务于 Evidence Agent / Trust Layer / Ops 面，不绕过 `Fairness Sentinel -> Chief Arbiter -> Opinion Writer`。
2. **数据一致性**：六对象主链仍为唯一裁决事实源；public verification、release evidence 与 artifact manifest 只引用 `case_dossier / claim_graph / evidence_ledger / verdict_ledger / fairness_report / opinion_pack` 的承诺、摘要或导出结果，不建立平行 winner 写链。
3. **门禁一致性**：release gate、fairness benchmark、panel shadow、artifact healthcheck、public verification readiness 和 evidence citation gate 只能收紧发布/展示条件，不得弱化 draw/review/blocked 保护语义。
4. **边界一致性**：`NPC Coach / Room QA` 保持 `advisory_only`；P39 不把互动型 Agent、Reason Passport、Identity Proof 或 Constitution Registry 接入单场官方裁决输入。
5. **跨层一致性**：涉及 public verification proxy、DTO、错误码、缓存语义或展示字段时，同轮检查 `ai_judge_service`、`chat_server`、OpenAPI、前端共享 domain/SDK 与必要测试，不保留长期 alias。
6. **收口一致性**：P39 所有真实环境项继续区分 `local_reference_ready`、`readiness_ready`、`env_blocked` 与 real-env `pass`；未获得真实窗口前不得把本地参考证据写成真实通过。

## 4. 本轮北极星与不可变边界

1. 北极星仍是“辩论结束后 5 分钟内获得可读、可信、可追溯、可执行的官方裁决，或明确进入受保护复核状态”。
2. AI 裁判正式公开结果只支持 `pro / con / draw`；`review_required` 与 `blocked_failed` 是保护性状态，不被改写成胜负。
3. 客户端永远通过 `chat_server` 使用裁判结果、公验状态与后续产品能力；AI 服务继续保持内部服务，不新增普通用户直连接口。
4. Public Verification 对外只展示可公开验证的承诺、哈希、版本、readiness 与必要错误码；不得暴露内部 prompt、provider、raw trace、私有审计细节、对象存储路径、bucket/prefix/endpoint 或敏感证据原文。
5. Evidence / Citation hardening 只判断“证据引用结构是否可追溯、是否满足展示与发布门槛”，不直接替代 Judge Panel 或 Chief Arbiter 下结论。
6. Release readiness evidence 是发布门禁与运营证据，不是新的裁决状态；发布 blocked 不能反向篡改已锁定 verdict ledger，但可以阻止策略发布、展示公验 ready 或 stage closure 的真实 pass。
7. 未发布能力默认直接收敛主语义，不长期保留 `oldPublicVerify`、`legacyTrustPayload`、双 DTO 或 shadow-only 的主链分支。

4. 代理复用 judge report read 权限、phone bind/auth 与读限流；非参与者继续返回 `judge_report_read_forbidden`。
5. AI 返回 ready/not-ready/env-blocked 时按 public readiness 语义透出；AI 不可达、非 2xx、坏 JSON 或合同违规时返回稳定 `proxy_error`，不泄漏 AI URL/internal key。

1. 在 `chat_server` 建立 public verification 代理接口，让客户端只通过主业务门面读取 AI 裁判公验状态。

3. 将权限、会话可见性、限流、超时、错误码与 no-secret redaction 放在 `chat_server` 层处理。
4. 不把 AI 服务暴露为公网 API，不让前端保存或配置 AI internal key。
