## 4. 详细实施计划（按阶段）

### 4.1 阶段 S0（本迭代必做）

#### `RBACPUT-S0-1` 输入边界修复（对应 `R2`）

- 目标：消除 `u64 -> i64 as` 截断风险。
- 计划改动：
  1. 在 `models/rbac.rs` 里对 path `user_id` 做 `i64::try_from` 安全转换。
  2. 为越界场景定义稳定 machine code（如 `ops_role_target_user_id_out_of_range`）。
  3. 统一映射到可预期状态码（建议 400）。
- 验收：超大 `user_id` 不再出现语义漂移，错误码稳定可机读。

#### `RBACPUT-S0-2` 原子写路径收敛（对应 `R3`）

- 目标：消除“预检通过后目标被删除”的竞态窗口。
- 计划改动：
  1. 将“用户存在校验 + UPSERT”收敛为单 SQL 原子路径（CTE 或等价方案）。
  2. 保持当前响应结构与 UPSERT 语义不变。
- 验收：目标用户不存在稳定返回 404，避免外键异常漂移到 5xx。

#### `RBACPUT-S0-3` Owner 真源守护（对应 `R1`）

- 目标：降低 `platform_admin_owners` 缺失导致的控制面硬故障概率。
- 计划改动：
  1. 增加启动/健康检查层面对 owner 真源的显式守护或告警。
  2. 保持运行态错误码可识别（`platform_owner_not_configured`）。
- 验收：owner 真源异常可提前暴露，不依赖业务请求触发后才发现。

#### `RBACPUT-S0-4` 回归测试补齐（对应 `R13`）

- 目标：补齐关键边界回归，形成 CI 防线。
- 计划改动（`handlers/debate.rs` route tests）：
  1. PUT owner 真源缺失 -> 500。
  2. PUT IP 限流触发 -> 429。
  3. PUT 超大 user_id -> 400 + 稳定 code。
- 验收：新增用例稳定通过，且失败时可清晰定位。

#### `RBACPUT-S0-5` 契约同步（对应 `R13` 延伸）

- 目标：确保实现与文档、OpenAPI、前端认知一致。
- 计划改动：
  1. 如有新增 code/status，更新 OpenAPI 注解。
  2. 回写学习文档与开发文档差异说明。
- 验收：文档、注解、运行行为三方一致。

### 4.2 阶段 S1（S0 完成后执行）

#### `RBACPUT-S1-1` 写侧幂等键（对应 `R6`）

- 目标：减少重试带来的审计噪音和 `updated_at` 抖动。
- 计划：为 PUT 引入可选 `idempotency-key` + TTL 防重执行。
- 验收：同 key 重试行为稳定可预期。
- 执行结果：已完成（路由支持 `idempotency-key` 校验、30s TTL 重放窗口、replay 不重复审计）。

#### `RBACPUT-S1-2` 审计失败分类增强（对应 `R10`）

- 目标：让审计能按失败类型聚合。
- 计划：在 `ops_rbac_audits` 增加失败细分字段，并落 migration + 写入逻辑。
- 验收：可按 `error_code/failure_reason` 精准检索。
- 执行结果：已完成（新增字段 + 写入逻辑 + 失败分类回归断言）。

#### `RBACPUT-S1-3` 限流退化策略强化（对应 `R7/R8`）

- 目标：Redis 异常/禁用时仍具备基础防护。
- 计划：Disabled 场景接入本地 fallback，优化 `unknown` 桶策略。
- 验收：退化模式下仍有可控限流，不出现全量放开。
- 执行结果：已完成（RBAC 写路径改用 Disabled fallback，新增 30+1 请求回归）。

#### `RBACPUT-S1-4` 版本线索返回（对应 `R5` 过渡）

- 目标：为控制台刷新一致性提供基础版本线索。
- 计划：评估 PUT 响应附加 `rbacRevision` 或等价头字段。
- 验收：前端可基于版本线索做刷新判定。
- 执行结果：已完成（PUT 成功响应附加 `x-rbac-revision`）。

### 4.3 阶段 S2（后续里程碑）

1. 委派式 `role_manage` 能力模型（缓解 Owner 单点）。
2. 审计可靠投递（outbox/重放机制）。
3. 控制平面专项可观测看板（成功/拒绝/参数错/系统错分层）。

#### `RBACPUT-S2-1` 委派式角色管理（对应 `R4`）

- 目标：降低 Owner 单点瓶颈，同时保持权限边界可控。
- 执行结果：已完成
  1. 角色模型新增 `platform_role_admin`（DB CHECK + 角色归一化白名单同步）。
  2. `role_manage` 判定从“仅 Owner”扩展为“Owner 或 delegated role admin”。
  3. 受限边界落地：委派管理员不能管理 Owner，也不能分配/撤销 `platform_role_admin`。
  4. 补齐 route/model 回归：委派管理员 list/put/delete 成功链路 + 边界拒绝链路。
