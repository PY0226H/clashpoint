## 2.2 逐项深度分析

### `OSESSION-R1`：契约声明不完整

- 现象：Handler 注解只写 `201/400/404/409`。
- 实际：运行态明确存在 `401/403/422/500`。
- 影响：SDK 生成、前端分支、自动化联调都会出现“文档可调、实跑报错”。

### `OSESSION-R2`：TOCTOU 一致性窗口

- 现象：先 `SELECT topic exists`，再 `INSERT session`，且无显式事务。
- 风险窗口：预检成功后、插入前，topic 可能被删除。
- 后果：依赖 DB 外键兜底，最终可能抛 `sql error`（500），与预期 404 语义漂移。

### `OSESSION-R3`：ID 边界转换不安全

- 现象：`input.topic_id as i64`。
- 风险：超 `i64::MAX` 时发生符号翻转，触发不可预期 SQL 行为。
- 后果：低频但高隐蔽，属于典型输入边界治理缺口。

### `OSESSION-R4`：无幂等

- 现象：网络重试/客户端超时重放会直接再创建一条 session。
- 风险：重复场次、排期污染、运营误操作放大。
- 后果：后续需要人工清洗重复数据。

### `OSESSION-R5`：无限流

- 现象：创建入口无 user/ip 限流。
- 风险：脚本误触发或运维工具失控时短时间写爆。
- 后果：DB 压力上升，错误噪声放大，影响其他链路。

### `OSESSION-R6`：无审计

- 现象：创建 session 不写 ops 审计表。
- 风险：事故复盘无法回答“谁创建、何时创建、参数为何”。
- 后果：运维治理与合规追踪能力不足。

### `OSESSION-R7`：时间语义漏洞（过期 scheduled）

- 现象：仅校验 `end_at > scheduled_start_at`。`scheduled` 允许整段都在过去。
- 与 worker 交互：`advance_debate_sessions` 对 `scheduled_start_at <= now` 会推进到 `open`，未要求 `end_at > now`。
- 后果：可产生“已过期却处于 open 状态”的异常残留，污染运营视图。

### `OSESSION-R8`：错误模型机读弱

- 现象：多为 `DebateError("文案")`。
- 风险：客户端依赖字符串匹配，不稳定且难本地化。
- 后果：告警聚类、指标分桶、跨端行为一致性成本上升。

### `OSESSION-R9`：`joinable` 口径漂移

- 现象：create 返回 `joinable` 未包含“容量检查”；list 侧包含容量检查。
- 影响：不同接口返回的 `joinable` 语义不完全同构。
- 后果：前端在创建后立即读返回值与列表值可能出现认知差异（边界场景尤其明显）。

### `OSESSION-R10`：路由矩阵覆盖不足

- 现象：已有 model 测试覆盖核心校验，但 route 层（鉴权/门禁/提取器）专项矩阵不足。
- 风险：中间件调整后，问题只能在联调或线上暴露。

### `OSESSION-R11`：topic 活跃性策略未冻结

- 现象：仅校验 topic 存在，不校验 `is_active`。
- 风险：如果产品语义是“停用辩题不再开新场次”，当前实现会违反策略。
- 结论：需产品侧明确“允许历史题复开”还是“严格 active-only”。

### `OSESSION-R12`：缺业务去重

- 现象：没有“同 topic + 时间窗口”的重复保护。
- 风险：并发创建或重试创建可产出重复排期。
- 后果：运营后台治理负担上升。

## 3. 解决方案（企业主流 + 推荐路径）

## 3.1 方案 A（推荐，渐进式治理）

目标：在不大改架构前提下，2 个迭代把 API 055 拉齐到“治理型写接口”基线。

### A-P0（必须先做）

1. **补齐 OpenAPI 响应谱系（解决 `R1`）**
- 为 handler 明确补齐：`401/403/422/500`。
- 收益：文档、SDK、联调一致。

2. **修复 ID 安全转换（解决 `R3`）**
- 使用 `safe_u64_to_i64(topic_id, "debate_topic_id_invalid")`。
- 收益：边界输入稳定收敛到 `400`。

3. **消除 TOCTOU 窗口（解决 `R2`）**
- 推荐做法：`BEGIN -> SELECT topic FOR SHARE -> INSERT -> COMMIT`。
- 兜底：对外键冲突 `23503` 映射业务错误（404 或稳定 conflict code）。
- 收益：错误语义稳定，不受竞态影响。

4. **收紧时间语义（解决 `R7`）**
- 推荐规则：创建时统一要求 `end_at > now`（至少在 `scheduled/open` 两种入口都成立）。
- 若业务需补历史数据：通过显式参数/内部接口放行，不走公开 ops 创建口。
- 收益：避免僵尸 `open` 会话。

### A-P1（强烈建议）

5. **接入幂等键（解决 `R4`）**
- 支持 `Idempotency-Key`，记录 `user_id + key -> session_id`。
- 冲突时返回首次创建结果，实现重试无副作用。

6. **接入限流（解决 `R5`）**
- 复用现有 request_guard：user/ip 双阈值。
- 在 Redis 关闭场景保留本地降级闸门，保证最小保护。

7. **补创建审计（解决 `R6`）**
- 新增 `ops_debate_session_audits`（或复用统一 ops 审计表）记录 `create`。
- 字段至少包括：`session_id/operator_user_id/action/request_id/created_at`。

8. **新增业务去重规则（解决 `R12`）**
- 规则建议：同 `topic_id` + 同 `scheduled_start_at/end_at` 视为重复。
- 应用层先查重，后续可升级 DB 唯一索引（按产品允许策略设置 partial unique）。

### A-P2（平台化体验提升）

9. **结构化错误模型（解决 `R8`）**
- 统一到 `{code, message, details?}`，客户端完全机读 `code`。

10. **统一 `joinable` 口径（解决 `R9`）**
- 将 create/list/update 的 `joinable` 公式抽为统一 SQL 片段或视图，至少保证容量条件一致。

11. **route 回归矩阵补齐（解决 `R10`）**
- 覆盖 `201/400/401/403/404/409/422/429/500`。
- 至少新增：鉴权失败、手机号门禁、topic 不存在、时间非法、幂等回放、限流命中。

12. **topic active 策略决策（解决 `R11`）**
- 若决策为 active-only，则创建前加 `topic.is_active = true` 校验并返回稳定错误码。

## 3.2 方案 B（平台化强化版）

适用于后续 ops API 扩展期：

1. 抽象统一“Ops 写接口治理管线”：权限 -> 校验 -> 幂等 -> 限流 -> 事务写入 -> 审计 -> 观测。
2. 将 session/topic/create/update 共用一套错误码体系与审计模型。
3. 将时间语义与状态机入口配置化（策略表），减少硬编码扩散。

优点：长期维护成本更低；缺点：一次性改造面更大。

---

## 3.3 最终推荐（给你审核）

建议采用 **方案 A（P0 -> P1）**，先快收敛关键正确性风险，再做平台化。

推荐落地顺序：

1. `R1 + R3 + R2`（契约一致性 + 边界安全 + 事务一致性）
2. `R7`（时间语义修正，防僵尸场次）
3. `R4 + R5 + R6`（幂等 + 限流 + 审计）
4. `R12 + R10`（去重 + 路由回归封板）
5. `R8 + R9 + R11`（结构化与策略化增强）

## 4. 验收标准（DoD）

完成后至少满足：

1. OpenAPI 与运行态错误谱系一致。
2. `topic_id` 边界输入不再出现符号翻转语义。
3. topic 删除竞态下仍返回稳定业务错误，不暴露 SQL 异常语义。
4. 不再创建会导致生命周期异常的过期场次。
5. 创建重试具备幂等，写风暴具备限流防护。
6. 场次创建可追溯（审计可查）。
7. route 级回归矩阵可拦截中间件/提取器回归问题。
