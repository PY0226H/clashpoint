## 2. 当前实现链路（对齐代码）

### 2.1 写入链路（失败进入 DLQ）

1. Kafka 消费入口：`consume_worker_message`（`event_bus.rs`）
2. 业务处理：`process_worker_envelope`
3. 失败落库：`upsert_kafka_dlq_failure`
4. 表：`kafka_dlq_events`
   1. 主状态：`pending/replayed/discarded`
   2. 唯一键：`UNIQUE(consumer_group, event_id)`
   3. 失败次数：`failure_count`

### 2.2 查询链路（本次 API）

1. 路由：`/api/debate/ops/kafka/dlq`
2. 中间件：`verify_token_header_only` + `require_phone_bound`
3. 权限：`ensure_ops_permission(..., OpsPermission::JudgeReview)`
4. 服务逻辑：`list_kafka_dlq_events`
   1. 参数归一化（`status/eventType/limit/offset`）
   2. 两次 SQL（`COUNT + SELECT`）
   3. 列表不返回 `payload`

### 2.3 处置链路（回放/丢弃）

1. 回放：`POST /api/debate/ops/kafka/dlq/{id}/replay`（`JudgeRejudge`）
2. 丢弃：`POST /api/debate/ops/kafka/dlq/{id}/discard`（`JudgeRejudge`）
3. Readiness：`get_kafka_transport_readiness` 会读取 `pending` 数量和最近动作频率。

## 3. 深度问题分析（系统 / 模块 / 代码）

以下按优先级排序（P0 > P1 > P2）。

| 编号 | 层级 | 问题 | 代码证据 | 影响 |
|---|---|---|---|---|
| P0-1 | 模块/代码 | `replay` 与 `discard` 缺少原子状态机约束，存在并发竞态 | `replay_kafka_dlq_event` 先读状态再执行业务再 `UPDATE`；`discard_kafka_dlq_event` 无条件 `UPDATE` | 终态不确定（被并发覆盖），可能出现“已执行回放但最终显示 discarded/replayed 不一致” |
| P0-2 | 系统/治理 | 缺少可审计操作轨迹（谁在何时、为何 replay/discard） | `kafka_dlq_events` 仅有时间戳，无 `actor/reason/request_id`；handler 未记录 `user.id` 到业务表 | 线上排障责任无法追溯，不满足企业级运维审计要求 |
| P0-3 | 系统契约 | OpenAPI 声明与真实错误谱系不完整 | `list_kafka_dlq_events_handler` 注解当前只列 `200/409`，但运行时还会出现 `401/403/400/500` | 前端/SDK 误判、自动化测试漏场景、联调成本上升 |
| P1-1 | 模块 | 分页模型在大表下有性能上限（`COUNT + OFFSET`） | `list_kafka_dlq_events` 每次两次查询；`OFFSET` 深翻页即便钳制仍会扫描 | DLQ 堆积时接口延迟与 DB 负载上升 |
| P1-2 | 系统 | 缺少 DLQ 数据生命周期治理（归档/清理） | 检索未发现 `kafka_dlq_events` 清理 worker 或迁移策略 | 长期膨胀导致索引变重、备份成本与查询波动变大 |
| P1-3 | 安全/模块 | `error_message` 原样回传，且 `ops_viewer` 可读取 | `KafkaDlqEventItem.error_message` 直接映射 DB；`JudgeReview` 对 viewer 放开 | 可能暴露内部 SQL/约束名/底层错误细节 |
| P1-4 | 代码质量 | 关键接口缺乏针对性测试 | 检索无 `list/replay/discard` 用例；现有 `kafka_dlq.rs` 测试主要覆盖 readiness helper | 重构或改 SQL 时容易引入回归 |
| P2-1 | 模块体验 | 参数策略存在“静默钳制”与“精确匹配”可用性问题 | `offset.min(50_000)`、`eventType` 精确匹配且无提示 | 运维可能误以为数据缺失（实为查询条件/偏移被改写） |
| P2-2 | 系统语义 | readiness 中“replay progress”统计把 `discard` 也计入动作 | `recent_dlq_replay_action_count` 统计 `status IN (replayed, discarded)` | 指标命名与业务语义偏差，影响判断准确性 |

## 4. 解决方案（企业主流 + 推荐）

## 4.1 方案 A（企业主流：强状态机治理）

适用：你希望把 DLQ 从“能用”升级为“可审计、可并发、可追责”的中台级治理模块。

核心设计：

1. 引入显式状态机和 CAS 更新：
   1. `pending -> replaying -> replayed`
   2. `pending -> discarded`
   3. 禁止 `discarded -> replayed` 的隐式覆盖（需显式“恢复”动作）
2. `replay/discard` 改为“条件更新 + 行级锁”：
   1. `UPDATE ... WHERE id=? AND status='pending'`
   2. 未命中返回冲突，避免竞态覆盖
3. 增加审计表 `kafka_dlq_event_actions`：
   1. 字段：`event_id/action/operator_id/reason/request_id/before_status/after_status/result/error/created_at`
   2. 所有 replay/discard 必落审计
4. API 增加 `reason` 与 `idempotency_key`（企业主流做法）

优点：

1. 并发与审计一次到位
2. 方便后续接入审批、SLA 与自动化 runbook

代价：

1. 改动面较大（接口、DB、测试一起改）

---

## 4.2 方案 B（企业主流：查询性能与容量治理）

适用：DLQ 规模开始增长，Ops 列表需要稳定低延迟。

核心设计：

1. 分页升级为游标分页（keyset）：
   1. 游标：`(updated_at, id)`
   2. 避免深分页 `OFFSET` 扫描
2. `COUNT` 降级为可选：
   1. 默认 `includeTotal=false`
   2. 前端按需请求总数
3. 索引优化（按查询模式）：
   1. `status,event_type,updated_at DESC,id DESC`
   2. 针对 `pending` 的部分索引（如 `WHERE status='pending'`）
4. 生命周期治理：
   1. `replayed/discarded` 超过 N 天归档或清理
   2. 仅保留最近排障窗口热数据

优点：

1. 面向规模增长更稳
2. 对现有业务语义影响小

代价：

1. 需要前后端同步分页协议（cursor）

---

## 4.3 方案 C（安全与可观测增强）

适用：你希望提升 Ops 读接口在安全与排障效率上的平衡。

核心设计：

1. `error_message` 分级输出：
   1. `ops_viewer` 只看脱敏摘要（`error_code` / `class`）
   2. `ops_reviewer/admin` 可看完整错误
2. 增加结构化指标与日志：
   1. `dlq_list_latency_ms`
   2. `dlq_list_total_rows`
   3. `dlq_list_filter_hit_rate`
3. 统一契约：
   1. OpenAPI 覆盖完整错误码
   2. 增加契约测试防漂移

优点：

1. 安全性更好
2. 运维诊断更可度量

代价：

1. 需要在 RBAC 与响应 DTO 增加一层映射逻辑

---
