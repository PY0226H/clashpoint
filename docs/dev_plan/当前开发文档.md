# API067 开发文档（分析 + 方案）

更新时间：2026-04-09
范围：`POST /api/debate/ops/sessions/{id}/judge/rejudge`

---

## 0. 目标与结论先行

本轮目标：

1. 基于真实代码核对 API067 当前行为。
2. 识别该接口完整流程中的风险点（契约、语义、一致性、可观测性、演进性）。
3. 提出可执行的企业级治理方案，并给出推荐路径。

结论：

1. API067 当前实现与“强制全量重判”语义存在明显偏差。
2. 在现有表约束下，它更接近“补齐缺失 phase job + 唤醒 dispatch”的控制接口。
3. 建议采用“硬切语义收敛”路线（预发布项目，按 AGENTS 规则默认不保留长期兼容层）。

---

## 1. 现状基线（As-Is）

## 1.1 入口与调用链

1. 路由：`chat/chat_server/src/lib.rs` 挂载 `/api/debate/ops/sessions/:id/judge/rejudge`。
2. Handler：`request_judge_rejudge_ops_handler`。
3. Service：`request_judge_rejudge_by_owner -> request_judge_job_internal`。
4. 中间件：`verify_token_header_only` + `require_phone_bound`。

## 1.2 关键参数与固定 flow

`request_judge_rejudge_by_owner` 固定传入：

1. `allow_rejudge = true`
2. `require_participant = false`
3. `require_existing_report = true`
4. `trigger_mode = "ops_rejudge"`
5. `request_idempotency_key = None`

## 1.3 核心行为摘要

1. 先校验权限、session 状态、历史报告存在性。
2. 在事务内尝试补齐缺失 `judge_phase_jobs`（`ON CONFLICT DO NOTHING`）。
3. 事务后唤醒 dispatch，并尝试 `enqueue_due_judge_final_jobs_once`。
4. 返回 `RequestJudgeJobOutput`，常见状态是 `already_queued`。

## 2. 完整问题分析（深入）

## 2.1 P0：接口语义与真实行为偏差（最高优先级）

现象：

1. 文档/命名语义是“rejudge（重判）”。
2. 真实实现通常不会产生新的 final 结果版本。

根因：

1. `judge_final_jobs` 是 `UNIQUE(session_id)`。
2. `judge_final_reports` 是 `UNIQUE(session_id)`。
3. rejudge 接口没有“重置旧 job 状态”或“创建新 run 版本”的逻辑。
4. `enqueue_due_judge_final_jobs_once` 仅在不存在 final job 时入队。

影响：

1. Ops 误以为“已触发重判”，但实际可能无重新计算发生。
2. 操作结果的可解释性弱，易引发线上排障歧义。
3. 对产品“纠错重算”能力的可信度构成风险。

代码证据：

1. `chat/chat_server/src/models/judge/request_report.rs`
2. `chat/chat_server/src/models/judge_dispatch/dispatch_worker.rs`
3. `chat/migrations/20260224022000_ai_judge_domain.sql`

---

## 2.2 P0：自动化回归覆盖缺口（route/model 专项缺失）

现象：

1. 与 API061-066 相比，API067 尚未看到同等级 route/model 专项回归组。
2. 关键分支（权限、状态、require_existing_report、degraded）缺少固定门禁。

影响：

1. 小改动可能导致语义回退且不易及时发现。
2. 文档与实现再次漂移概率高。

---

## 2.3 P1：OpenAPI 契约与运行时错误谱系不一致

现象：

1. OpenAPI 当前仅声明 `202/404/409`。
2. 运行时还会出现 `400/401/403/500`（中间件与提取器决定）。

影响：

1. 前端/SDK 难以稳定建模错误分支。
2. 监控与告警分桶颗粒度不足。

---

## 2.4 P1：输入边界治理不完整（`u64 as i64`）

现象：

1. `session_id` 在模型层仍使用 `as i64`。
2. 超界值可能进入不透明行为（例如负值查询语义）。

影响：

1. 错误语义不稳定，不利于排障。
2. 安全边界不够显式。

---

## 2.5 P1：可观测性与审计不充分

现象：

1. handler 当前是“透传式”，缺结构化日志闭环（请求输入摘要、结果、耗时、失败分类）。
2. 缺少“谁在何时触发了 rejudge”的专属审计落库记录。

影响：

1. 线上排障需要跨日志拼接，成本高。
2. 合规与追责链条不完整。

---

## 2.6 P1：缺少接口级限流与显式幂等策略

现象：

1. API067 未像 `judge/jobs` 那样做 user/ip 限流。
2. 也未提供 `Idempotency-Key` 或等价幂等协议。

影响：

1. 高频重复触发可能造成无效调度压力。
2. 运维误操作放大后，难以快速解释“重复受理”现象。

---

## 2.7 P2：状态准入与业务语义仍有歧义

现象：

1. `can_request_judge` 允许 `judging | closed`。
2. 但 API 名称与运维理解更偏“已完成后的重判动作”。

影响：

1. 读者容易误解可用场景。
2. 可能在非预期状态触发补队列逻辑。

---

## 3. 解决方案（企业视角）

## 3.1 方案 A：最小止血（1~2 天）

动作：

1. 补齐 OpenAPI 错误谱系（`400/401/403/500`）。
2. 增 route/model 专项回归测试。
3. `session_id` 改 checked 转换并返回稳定 `400`。
4. handler 增结构化日志。
5. 学习文档与开发文档同步收敛。

优点：

1. 风险低、落地快。
2. 立刻提升契约稳定性与可维护性。

缺点：

1. 不解决“rejudge 语义不真”的根问题。

适用：

1. 近期只需要先保证接口可控、可观测、可测试。

---

## 3.2 方案 B：语义硬切到“真重判”（推荐）

核心思想：

1. 预发布阶段直接硬切，不保留长期兼容层。
2. 引入“重判运行版本（run/version）”建模，允许同一 session 多次重判并保留历史。

建议设计：

1. 新增 `rejudge_run_id`（或 `judge_run_no`）贯穿 phase job / final job / final report。
2. 把唯一约束从 `UNIQUE(session_id)` 调整为 `UNIQUE(session_id, rejudge_run_id)`。
3. API067 每次触发创建新 run，并生成新的 phase/final 任务链。
4. `GET judge-report` 默认返回最新 run，可按 run 查询历史。
5. 新增 `ops_judge_rejudge_actions` 审计表，记录 `requested_by/request_id/run_id/reason/result`。

优点：

1. 接口语义与行为一致，产品可解释性强。
2. 具备完整历史追溯能力，便于风控与复盘。
3. 从架构上解决“看似重判但可能无动作”的根因。

缺点：

1. 涉及迁移、查询口径、前后端契约协同，改造面更大。

适用：

1. 明确需要“可重复重判 + 可追溯版本”能力。

---

## 3.3 方案 C：接口语义降级为“dispatch poke”（备选）

动作：

1. 保持现有实现，不做大迁移。
2. 将 API067 文案/命名/文档明确改为“补队列与唤醒触发”。

优点：

1. 改造成本最低。

缺点：

1. 放弃“真实重判”能力，不符合多数产品预期。
2. 后续仍需引入新接口实现真正重判。

---

## 4. 推荐路线（结合产品需求）

推荐：`A 先止血 + B 硬切落地`。

理由：

1. 先用 A 快速补齐契约、测试、边界、观测，降低短期风险。
2. 再按 B 做结构性改造，彻底解决语义偏差。
3. 该项目尚未正式发布，硬切成本与风险都低于“长期兼容双轨”。

## 5. 拟执行分阶段（供你审核）

## S1（短周期，建议立即执行）

1. OpenAPI 补齐 `400/401/403/500`。
2. route/model 回归补齐。
3. `session_id` checked 转换。
4. handler 结构化日志补齐。
5. 文档与测试证据归档。

DoD：

1. `request_judge_rejudge` 关键错误分支都有自动化回归。
2. 错误码谱系在 OpenAPI 与运行时一致。

## S2（中周期，语义硬切）

1. 设计并落地 run/version 模型迁移。
2. phase/final/report 查询改为“最新 run + 可回溯历史 run”。
3. API067 改为“每次创建新 run”的真重判。
4. 增加重判审计表与审计查询面。

DoD：

1. 同一 session 可执行多次重判并保留历史。
2. 默认读最新，历史可回溯。

## S3（上线前收口）

1. 增加 rejudge 专项限流与幂等策略。
2. 增加 Ops 告警指标（频率、成功率、degraded 比例、平均耗时）。
3. 完成一次故障注入演练与复盘文档。

DoD：

1. 高频误触发不会放大调度压力。
2. 观测与追责链路完整。
