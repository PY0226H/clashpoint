## 1. 全流程问题分析（详细）

## 1.1 流程现状（简图）

`verify_token_header_only -> require_phone_bound -> update_debate_topic_ops_handler -> update_debate_topic_by_owner -> UPDATE debate_topics ... RETURNING`

这个链路当前具备：

- 鉴权、手机号门禁、RBAC
- 字段规范化和白名单校验
- 单 SQL 原子更新 + 404 判定

但在“写接口治理”维度与 053（创建）相比明显偏轻。

## 1.2 风险清单（按优先级）

## 1.3 逐项深度分析

### `UTOPIC-R1`：更新可绕过“重复辩题治理”

- 现象：`POST /ops/topics` 已有规范化查重；`PUT /ops/topics/{id}` 未复用该治理。
- 根因：更新逻辑只做字段校验 + 直接 UPDATE，未做“排除自身”的重复检查。
- 风险：通过更新可以制造同分类下重复标题，破坏 053 已建立的治理边界。
- 影响：列表与运营后台可出现语义重复条目，影响内容质量与后续统计。

### `UTOPIC-R2`：无乐观并发控制（覆写风险）

- 现象：当前没有 `revision` / `updatedAt` 前置条件；后写覆盖先写。
- 根因：PUT 请求体不携带版本条件，SQL 也无 `AND updated_at = ...`。
- 风险：两个运营几乎同时编辑同一辩题时，先提交结果会被静默覆盖。
- 影响：内容回滚、事故排查困难，且客户端通常感知不到被覆盖。

### `UTOPIC-R3`：API 契约与运行态不对齐

- 现象：OpenAPI 仅写 `200/400/404/409`；实际还会出现 `401/403/500`，提取器失败可见 `422`。
- 根因：注解未覆盖中间件与 extractor 级错误路径。
- 风险：SDK 自动生成类型不全、前端/测试误判、接口联调反复。
- 影响：研发效率与跨团队协作成本上升。

### `UTOPIC-R4`：更新缺审计

- 现象：已有 `ops_debate_topic_audits` 仅记录 `create/create_replay`，无 `update`。
- 根因：更新链路没有审计表写入。
- 风险：发生误改后无法快速定位操作者、修改时刻、请求上下文。
- 影响：运维治理能力不足，不利于后续合规要求升级。

### `UTOPIC-R5`：更新写路径治理偏轻

- 现象：无 user/ip 限流、无请求幂等。
- 根因：创建链路已有 request_guard 组件，更新链路未接入。
- 风险：前端重试风暴、脚本误触发会造成高频写入和抖动。
- 影响：数据库写压力与观测噪声增加，故障期放大连锁风险。

### `UTOPIC-R6`：ID 强转存在边界语义问题

- 现象：Path `u64` 在 SQL 绑定前直接 `as i64`。
- 根因：未复用 `safe_u64_to_i64`（项目中已存在安全转换 helper）。
- 风险：超 `i64::MAX` 输入会出现符号翻转，返回语义不直观。
- 影响：虽然低频，但属于输入边界治理缺口。

### `UTOPIC-R7`：错误码语义层次混杂

- 现象：`DebateConflict` 最终返回如 `debate conflict: ops_permission_denied:...`。
- 根因：当前错误输出使用 `Display` 文案，未分离 `code` 与 `message`。
- 风险：客户端需做字符串切片/模糊匹配；日志聚类不稳定。
- 影响：前后端契约维护成本高，监控告警聚合粒度差。

### `UTOPIC-R8`：字节长度口径与中文认知偏差

- 现象：`title/description/stance/contextSeed` 采用 `String::len()`（字节）。
- 根因：实现统一选择字节口径。
- 风险：用户看到“字符没超”，但接口判定超长。
- 影响：产品体验层面的解释成本增加。

### `UTOPIC-R9`：category 白名单硬编码

- 现象：白名单常量在代码中。
- 根因：当前选择快速落地方案，未引入配置中心/字典表。
- 风险：运营新增分类必须走发版流程。
- 影响：迭代效率受限，运营自治能力不足。

### `UTOPIC-R10`：更新路由回归覆盖不足

- 现象：当前有 service 级更新测试，但缺 route 级专项矩阵（401/403/404/409/422 等）。
- 根因：重心先落在 create（053），update 未同步补齐。
- 风险：中间件与 extractor 行为变更不易被提前发现。
- 影响：联调阶段才暴露问题，修复成本更高。

## 2. 解决方案（企业主流 + 推荐落地）

## 2.1 方案总览

### 方案 A（推荐，主流渐进式）

目标：在不引入大规模架构改造的前提下，2 个迭代内把更新接口提升到与创建接口接近的治理等级。

特点：

- 改造风险可控
- 与现有代码结构最兼容
- 交付节奏快，适合当前本地开发阶段

### 方案 B（增强版，平台化治理）

目标：统一 Topic 写路径治理中台（Create/Update/Patch 同一套管线），同时推动错误模型、审计和字典系统平台化。

特点：

- 上限高
- 改动面更大
- 适合 A 收口后再做

---

## 2.2 方案 A 详细设计（推荐执行）

### A-P0（必须先做）

1. **补齐更新侧重复治理（解决 `R1`）**
- 在 `update_debate_topic_by_owner` 增加规范化冲突检查：
  - 规则仍是 `LOWER(BTRIM(title))+category`。
  - 查重时排除当前 `id`（`WHERE id <> $current_id`）。
- 冲突返回稳定码：`debate_topic_duplicate_title_in_category`。
- 收益：堵住“创建受控、更新绕过”。

2. **加乐观并发控制（解决 `R2`）**
- 主流做法一（推荐）：请求体增加 `expectedUpdatedAt`。
- SQL 追加条件：`AND updated_at = $expected_updated_at`。
- 未命中行返回 `409 debate_topic_revision_conflict`。
- 收益：显式防覆写，客户端可提示“数据已被他人更新，请刷新后重试”。

3. **修正 OpenAPI 契约（解决 `R3`）**
- 为 update handler 补齐 `401/403/500`。
- 在接口文档中明确 extractor 失败可能 `422`。
- 收益：SDK、联调与自动化测试契约一致。

### A-P1（强烈建议）

4. **新增更新审计（解决 `R4`）**
- 扩展 `ops_debate_topic_audits.action`：加入 `update`（必要时加 `update_replay`）。
- 审计建议字段：`before_snapshot_hash`、`after_snapshot_hash` 或最小 diff 字段。
- 收益：可追溯、可归责、可复盘。

5. **给更新接口补限流与幂等（解决 `R5`）**
- 复用 create 的 request_guard 组件：
  - user 限流、ip 限流
  - `Idempotency-Key`（可选）
- 收益：降低重试风暴和误操作冲击。

6. **替换不安全强转（解决 `R6`）**
- 用 `safe_u64_to_i64(topic_id, "debate_topic_id_invalid")`。
- 超界输入统一 `400`。
- 收益：边界语义可预期。

7. **回归测试补齐（解决 `R10`）**
- route 层新增最小矩阵：
  - 200、400（各类校验码）、401、403、404、409、422
- 并发用例：revision 冲突。
- 治理用例：重复标题冲突。

### A-P2（产品与平台体验优化）

8. **错误模型结构化（解决 `R7`）**
- 响应结构从 `{error: string}` 升级为 `{code, message, details?}`（保持向后兼容窗口）。
- 收益：前端机读稳定，监控聚类更准。

9. **长度口径产品决策（解决 `R8`）**
- 二选一：
  - 继续字节口径（低风险）并在前端明确提示“按字节计数”；
  - 切字符口径（体验更好）并补迁移和回归。

10. **category 字典配置化（解决 `R9`）**
- 引入配置表或集中配置，服务启动时缓存。
- 收益：运营扩类无需发版。

## 2.3 方案 B（平台化增强）

作为 A 收口后的下一阶段：

1. 抽象 Topic 写管线（权限 -> 校验 -> 并发 -> 持久化 -> 审计 -> 观测）。
2. Create/Update/Patch 共享同一治理模块，避免能力漂移。
3. 错误模型、审计模型、字典模型统一平台化。

适用时机：当 Topic 进入高频运营阶段，且多端/多团队协作成本开始显著上升时。

---

## 2.4 我给你的最终推荐

建议执行 **方案 A（P0 -> P1）**，原因：

1. 与现有项目阶段匹配：小步快跑、风险可控。
2. 直接消除当前最关键的正确性风险（重复、覆写、契约不一致）。
3. 能快速把 054 治理水平拉齐到 053 的工程基线附近。

建议执行顺序：

1. `A-P0-1` 重复治理
2. `A-P0-2` 乐观锁
3. `A-P0-3` OpenAPI 修正
4. `A-P1-4/5/6/7` 审计+限流幂等+边界转换+测试矩阵
