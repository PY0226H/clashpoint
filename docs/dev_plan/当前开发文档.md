更新时间：2026-04-09
范围：围绕 Ops RBAC 撤销写接口完整链路（鉴权、门禁、限流、权限、并发、审计、观测、测试）进行问题分析与方案设计。

## 1. 当前实现链路快照（基线）

当前请求主链路：

1. `verify_token_header_only`
2. `require_phone_bound`
3. `revoke_ops_role_assignment_handler`
4. `If-Match` 头解析（语法校验）
5. user/ip 双限流（`ops_rbac_roles_write`）
6. `revoke_ops_role_assignment_by_owner_with_meta`
   - `role_manage` 上下文判定（Owner / `platform_role_admin`）
   - 委派管理员 guardrail（不能管理 Owner、不能管理 `platform_role_admin`）
   - 事务级 advisory lock + revision 条件写校验
   - `DELETE ... RETURNING` 撤销角色行（幂等）
   - success 审计同事务写入 outbox
7. 请求后 `dispatch_once`，后台 worker 周期推进 outbox 重试
8. 响应返回 `x-rbac-revision`；Owner 自写时返回 `x-rbac-warning=owner_self_role_assignment_no_effect`

当前关键特征：

1. Owner 真源来自 `platform_admin_owners`（非硬编码 `user_id=1`）。
2. DELETE 接口当前不接入 `idempotency-key`；状态幂等由 `removed` 字段表达。
3. 服务端强制 `If-Match`（API 路径），缺失返回 `ops_rbac_if_match_required`。
4. 并发冲突返回 `ops_rbac_revision_conflict`，并写入审计/outbox。
5. 写接口具备 user/ip 双限流，Redis disabled 启用本地 fallback。
6. IP key 仅在 trusted proxy 命中时采信转发头，否则回退 user-scoped unknown key。
7. 审计链路已具备 outbox 可回放能力，成功写审计已同事务入队。

### 2.2 深入分析

#### `RBACDEL-R1`：Owner 真源缺失导致控制面写不可用（P0）

- 触发条件：`platform_admin_owners` 无有效 owner 行。
- 根因：控制面权限判定依赖 owner 真源表。
- 影响：DELETE/PUT/list/me 等 RBAC 管理面接口出现 500。
- 当前防线：已落地 owner 真源守护和稳定错误码 `platform_owner_not_configured`。
- 残余风险：环境迁移/脏库恢复窗口仍需运维侧 runbook 保证。

#### `RBACDEL-R2`：`user_id` 边界输入风险（P0）

- 触发条件：`user_id > i64::MAX`。
- 根因：DB 侧 ID 采用 `i64`，路径参数为 `u64`。
- 影响：若不做安全转换会发生语义漂移或潜在误删风险。
- 当前防线：`checked_u64_to_i64` + `ops_role_target_user_id_out_of_range`。
- 残余风险：无。

#### `RBACDEL-R3/R4`：角色管理能力与越权边界（P1）

- 触发条件：引入委派角色管理后，若无限制可能形成“委派链条升级”。
- 根因：`platform_role_admin` 同时具备管理能力与可写角色目标，若不加约束会失控。
- 影响：委派管理员可误/恶意接管 owner 或创造更多 role_admin。
- 当前防线：
  1. 委派管理员不能管理 Owner。
  2. 委派管理员不能管理 `platform_role_admin`。
- 残余风险：策略已收敛；主要依赖持续测试防回归。

#### `RBACDEL-R5/R6`：并发冲突与条件写协议（P1）

- 触发条件：多操作者几乎同时执行 PUT/DELETE。
- 根因：若无条件写，后写覆盖前写且冲突不可见。
- 影响：运维控制面出现“误删/误覆盖”且客户端误以为成功。
- 当前防线：
  1. API 路径强制 `If-Match`。
  2. 事务内 revision 比对，不匹配返回 `409 ops_rbac_revision_conflict`。
  3. 事务级 advisory lock 串行化“校验+落库”窗口。
- 残余风险：客户端若不处理 409 重试，体验层仍可能出现“操作未生效”的认知偏差。

#### `RBACDEL-R7`：DELETE 无幂等键导致审计噪音（P1）

- 触发条件：客户端超时重试、网关重放、用户重复点击。
- 根因：DELETE 当前不使用 `idempotency-key` 去重。
- 影响：
  1. 数据状态幂等（`removed=false`）仍成立；
  2. 但会产生多条 success 审计，提升噪音与排障成本。
- 当前防线：通过 `removed` 字段区分“实删/空删”。
- 残余风险：审计聚合需额外按 `request_id/removed` 去噪。

#### `RBACDEL-R8`：`removed=false` 语义信息损失（P1）

- 触发条件：目标用户不存在或存在但无角色。
- 根因：DELETE 当前不做目标用户存在性预检。
- 影响：运维无法仅凭响应区分“错误 user_id”与“已撤销完成”。
- 当前防线：UI 侧可通过列表快照与 `rbacRevision` 做辅助校验。
- 残余风险：排障时可能增加人工确认步骤。

#### `RBACDEL-R9/R10`：限流可靠性与来源可信边界（P1）

- 触发条件：Redis disabled、代理未透传真实来源、客户端伪造转发头。
- 根因：控制面接口写链路天然暴露于重试与恶意放大场景。
- 影响：限流失效或误限流，影响运维操作可用性。
- 当前防线：
  1. disabled fallback 本地限流；
  2. trusted proxy 门控；
  3. untrusted 时使用 `unknown_user_scope:{user_id}`。
- 残余风险：生产白名单未运维化时，IP 维精度仍会下降（安全优先）。

#### `RBACDEL-R11`：审计可靠性与合规证据（P1）

- 触发条件：请求后写审计失败、短时数据库抖动。
- 根因：控制面请求不能被审计写失败阻断。
- 影响：若无补偿，可能出现“操作成功但无审计证据”。
- 当前防线：
  1. success 审计与业务写同事务入 outbox；
  2. 请求后快投递 + worker 周期重试。
- 残余风险：最终一致窗口仍存在，但证据不会丢失。

#### `RBACDEL-R12`：错误优先级策略未冻结（P2）

- 现象：
  1. `If-Match` 非法在 handler 早期直接返回 400；
  2. `If-Match` 缺失由 model 在权限校验后返回 400；
  3. 非授权调用可能先得到 409（权限）而非 400（缺失 If-Match）。
- 影响：对调用方而言，错误优先级存在“按分支变化”的感知差异。
- 当前防线：行为已稳定，且 route/model 测试覆盖关键路径。
- 残余风险：若未来重构未显式约束优先级，可能出现契约漂移。

#### `RBACDEL-R13`：DELETE invalid `If-Match` 路由断言缺口（P2）

- 现象：PUT 有 invalid `If-Match` 回归，DELETE 当前主要覆盖 stale/missing。
- 影响：删除路径的 header 语法校验回归防线相对薄弱。
- 当前防线：`parse_ops_rbac_if_match_header` 已统一复用，逻辑正确。
- 残余风险：未来修改 parser 时，DELETE 侧可能产生静默回归。

#### `RBACDEL-R14`：观测和压测基线缺口（P2）

- 现象：指标埋点已在代码内，但缺值班看板、阈值与压测证据封板。
- 影响：上线前无法客观判断冲突率、误限流率、回放队列健康度。
- 当前防线：已有日志与计数器基础。
- 残余风险：运维可观测性不足会扩大故障定位时间。

## 3. 解决方案（企业主流 + 推荐落地）

### 3.1 企业主流方案（控制平面治理标准化）

适用场景：多环境、多角色值班、强合规审计。

#### 方案 A：控制面写接口统一治理基座（主流）

1. 权限模型：Owner 真源 + 受限委派（禁止委派链条扩散）。
2. 并发协议：全写接口统一 `If-Match` 条件写与冲突 machine code。
3. 限流治理：user/ip 双维 + trusted proxy 信任边界 + disabled fallback。
4. 审计策略：写操作成功审计同事务 outbox，失败路径统一分类入审计。
5. 错误契约：统一 machine code，不依赖 message 文案分支。
6. 运维体系：看板、告警阈值、压测基线、runbook 一体化。

优点：治理能力完整、可规模化复用、符合企业运维模型。  
代价：推进成本较高，涉及跨模块统一改造。

#### 方案 B：删除语义双模式（严格模式 + 幂等模式）

1. 默认保持当前幂等模式：`200 + removed=false`。
2. 增加可选严格模式（例如 query `strict=true`）：
   - 目标用户不存在时返回 `404`；
   - 仅用于运维排障或脚本审计场景。

优点：兼容“重试友好”与“问题定位精确”两类诉求。  
代价：契约复杂度上升，需要前后端与文档同步。

#### 方案 C：DELETE 审计去噪治理（可选）

1. 为 DELETE 引入可选 `idempotency-key`（短 TTL）。
2. 或按 `request_id + operator + target + window` 在审计侧聚合去重。

优点：减少重试噪音，提高审计信噪比。  
代价：实现复杂度增加，且可能损失“每次调用事实”。

### 3.2 我推荐的方案（适配 EchoIsle 当前阶段）

基于“项目仍处于预发布、可硬切换”的前提，建议采用 **小步硬切 + 先运维闭环**：

1. **生产 trusted proxy 白名单落地**（对应 `RBACDEL-R10/R14`）
   - 补齐 `trusted_proxy_ids/cidrs` 与误配 runbook。
2. **跨控制面写接口复用 forwarded-header 治理**（对应 `RBACDEL-R10`）
   - 将 auth/payment/ticket 写路径收敛到同一信任边界策略。
3. **RBAC 写链路观测与压测封板**（对应 `RBACDEL-R14`）
   - 输出冲突率、限流命中率、outbox 积压与投递时延基线。

#### 阶段 S2（可选增强）

1. **补齐 DELETE invalid `If-Match` route 测试**（对应 `RBACDEL-R13`）。
2. **冻结错误优先级策略并文档化**（对应 `RBACDEL-R12`）。
3. **评估 DELETE 去噪策略**（对应 `RBACDEL-R7`，按噪音实测决定是否引入）。

#### 阶段 S3（产品策略增强，可后置）

1. 评估是否需要 strict delete 模式（对应 `RBACDEL-R8`）。
2. 若不做 strict 模式，则在控制台交互上强化“removed=false 的排障提示”。

推荐原因：

1. 当前代码级高风险已收敛，收益最大的是运维与观测闭环。
2. 不引入兼容层和大规模协议重构，符合当前开发节奏。
3. 每一步都可独立验收，失败可局部回滚。

### 5.2 S2 评审结论（本轮已给出 Go/No-Go）

#### `RBACDEL-S2-1` strict delete 模式评估（对应 `R8`）

结论：**No-Go（当前阶段不引入 strict 模式）**。

原因：

1. 当前产品处于预发布，默认应保持协议简单、可硬切，避免同时维护“幂等模式 + 严格模式”的双语义。
2. 现有 `removed=false` 已能表达“最终状态满足”，与控制面重试模型一致。
3. strict 模式会增加前后端分支、脚本调用复杂度、回归矩阵与文档维护成本。

触发重评条件（任一满足再重开）：

1. 线上运维场景中，`removed=false` 引发的误判工单持续出现且达到周级阈值；
2. 安全/合规要求必须区分“用户不存在”与“无角色行”；
3. 多租户或外部 API 消费方明确要求可选严格语义。

#### `RBACDEL-S2-2` DELETE 审计去噪评估（对应 `R7`）

结论：**短期不引入 DELETE `idempotency-key`；优先做观测侧去噪**。

推荐路径：

1. 保留原始审计事实（不在写路径丢信息）。
2. 在报表/看板层按 `operator_user_id + target_user_id + role + request window` 进行聚合视图去噪。
3. 基于 2~4 周样本评估重复 success 审计占比，再决定是否需要把 `idempotency-key` 升级为协议能力。

决策理由：

1. 写路径引入 `idempotency-key` 会增加协议与缓存一致性复杂度；
2. 当前合规场景更需要“保留事实”而不是“提前丢弃重复事件”；
3. 观测层去噪成本低、可迭代快，适合当前阶段。
