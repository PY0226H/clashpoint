# API 052 开发文档：`GET /api/internal/ai/judge/dispatch/metrics`

更新时间：2026-04-10
范围：接口文档对齐 + 完整流程问题分析 + 方案建议（本轮先产出分析与方案，不改业务代码）

---

## 1. 完整流程拆解（从任务入队到指标读取）

### 1.1 入口与鉴权

1. 路由挂载在 `internal_ai` 子路由：`/api/internal/ai/judge/dispatch/metrics`。
2. 所有 `/api/internal/ai/*` 请求先过 `verify_ai_internal_key`。
3. 鉴权通过后进入 `get_judge_dispatch_metrics_handler`。

### 1.2 指标读取链路（API 052 本身）

1. `get_judge_dispatch_metrics_handler` 调用 `AppState::get_judge_dispatch_metrics`。
2. `AppState::get_judge_dispatch_metrics` 返回 `dispatch_metrics.snapshot()`。
3. `snapshot()` 从 `AiJudgeDispatchMetrics` 的 `AtomicU64` 读取累计值并返回 `GetJudgeDispatchMetricsOutput`。

结论：API 052 是纯内存读，不直接触发 DB/Redis/Kafka/外部 HTTP。

### 1.3 指标写入链路（返回值的数据来源）

1. `spawn_ai_judge_dispatch_worker` 按 `dispatch_interval_secs` 周期运行，并支持触发通道唤醒。
2. 每个 tick 执行：
   - `dispatch_pending_judge_phase_jobs_once`
   - `enqueue_due_judge_final_jobs_once`
   - `dispatch_pending_judge_final_jobs_once`
3. phase/final dispatch 成功返回 report 后，通过 `observe_tick_success` 统一累计到总指标。
4. phase/final dispatch 若返回 `Err`，通过 `observe_dispatch_worker_error` 累加 `tick_error_total`。

### 1.4 派发失败分类来源

派发请求发往 `ai_judge_service`，失败会映射为 `DispatchFailureCode`：

1. 契约失败：`ResponseAcceptedFalse`、`ResponseJobIdMismatch`
2. HTTP 失败：`Http4xx`、`Http429`、`Http5xx`、`HttpUnexpectedStatus`
3. 网络失败：`NetworkSendFailed`
4. 内部失败：`PayloadBuildFailed`、`BuildClientFailed`

最终这些分类被累计到 `failedContractTotal`、`failedHttp4xxTotal`、`failedHttp429Total`、`failedHttp5xxTotal`、`failedNetworkTotal`、`failedInternalTotal`。

## 2. 详细问题分析（深入、完整）

以下问题按“可观测真实性、排障效率、生产治理风险”三个维度排序。

### P1. `timedOutFailedTotal` 口径空转（字段存在但当前永远是 0）

证据：

1. `JudgeDispatchTickReport` 包含 `timed_out_failed` 字段。
2. 但在 phase/final 汇总到总指标时固定传 `timed_out_failed: 0`。
3. 发送请求处没有单独的 timeout failure code，timeout 目前落到 `NetworkSendFailed`。

影响：

1. 超时风暴无法从单独指标感知。
2. 值班时会把“超时”与“网络故障”混成一类，根因定位变慢。

风险等级：高（直接影响故障定位准确性）。

### P2. `tickErrorTotal` 统计覆盖不完整

证据：

1. `tickErrorTotal` 只在 phase/final dispatch 返回 `Err` 时累加。
2. `enqueue_due_judge_final_jobs_once` 出错仅 `warn!`，不计入 `tickErrorTotal`。

影响：

1. Worker 的“真实异常面”被低估。
2. enqueue 阶段问题（例如 SQL 回归、索引退化导致失败）可能长期隐蔽。

风险等级：高（监控漏报）。

### P3. `failedHttp5xxTotal` 混入 `HttpUnexpectedStatus`

证据：

1. 映射逻辑里 `DispatchFailureCode::HttpUnexpectedStatus` 也计入 `failed_http_5xx`。
2. `HttpUnexpectedStatus` 语义并不等价 5xx（可包含 1xx/3xx 等异常状态）。

影响：

1. 5xx 告警会被污染，误导“下游服务崩溃”判断。
2. 影响容量决策与回滚决策准确性。

风险等级：中高。

### P4. `tickSuccessTotal` 容易被误用为吞吐成功量

证据：

1. 成功 tick 即累加，即使 `claimed=0`（空转 tick）也算成功。
2. 该字段是“循环成功次数”，不是“业务派发成功条数”。

影响：

1. 仪表盘若直接把 `tickSuccessTotal` 当成功吞吐，会得出错误健康结论。
2. 可导致延迟发现 backlog 积压。

风险等级：中。

### P5. 指标缺少关键维度（phase/final、触发来源）

证据：

1. 目前输出是总量聚合，不区分 phase vs final。
2. 不区分 `polling` 触发还是 `event trigger` 触发。

影响：

1. 只能看到“总坏了”，看不到“哪里坏了”。
2. 故障定位需要回看日志，响应时间变长。

风险等级：中高。

### P6. 指标仅内存累计，实例重启后归零

证据：

1. 全部指标存放在 `AtomicU64`，未持久化。
2. API 返回值没有 `processStartAt` 或 `uptime` 辅助字段。

影响：

1. 监控曲线出现断崖，若未做实例维度处理会误判恢复。
2. 跨实例聚合与历史对比依赖外部系统，接口自身不可解释“归零原因”。

风险等级：中。

### P7. 快照一致性是“近似一致”而非“严格一致”

证据：

1. `snapshot()` 对每个原子字段独立 `Relaxed` 读取。
2. 多字段可能来自略有时差的不同时刻。

影响：

1. 在高并发写入时，`claimed - dispatched - failed` 等关系可能出现短暂不闭合。
2. 若调用方误以为是事务快照，会触发误报。

风险等级：中（可接受但必须显式说明）。

### P8. 内部接口安全模型仍偏“静态密钥”

证据：

1. 当前只校验 `x-ai-internal-key` 是否等于配置项。
2. 无签名时间窗、无 nonce、防重放能力有限。

影响：

1. 密钥泄露后攻击面较大。
2. 对内网横向移动防护不足。

风险等级：中高（上线前必须补强）。

### P9. 自动化测试覆盖不足（指标语义层）

证据：

1. 现有 handler 测试主要覆盖 `401` 与“空快照返回”。
2. 未覆盖关键映射语义：429/4xx/契约失败/timeout 分类是否正确计数。

影响：

1. 后续重构容易引入口径回归但不易被及时发现。
2. 文档与实现再次漂移概率上升。

风险等级：中。

## 3. 解决方案（企业主流 + 推荐落地）

本节按“可快速落地、对产品价值高”的顺序给出方案。

### 3.1 推荐主方案（S1，优先执行）

目标：先把口径正确性补齐，再做维度化。

方案：

1. 补齐 timeout 专项口径：
   - 在发送错误映射里识别 `reqwest::Error::is_timeout()`。
   - 新增 `DispatchFailureCode::Timeout` 或直接在 report 层单独累加 `timed_out_failed`。
   - 保持 `failedNetworkTotal` 作为大类，同时补 `timedOutFailedTotal` 子类。
2. 修复 tick error 覆盖：
   - `enqueue_due_judge_final_jobs_once` 失败也纳入错误计数。
   - 最好拆分为 `phaseTickErrorTotal`、`finalEnqueueErrorTotal`、`finalTickErrorTotal`（比单一总数更可排障）。
3. 修复状态分类准确性：
   - 将 `HttpUnexpectedStatus` 从 `failedHttp5xxTotal` 拆分为单独字段（例如 `failedHttpUnexpectedTotal`）。

为什么符合产品需求：

1. 直接提升“值班定位速度”和“告警可信度”。
2. 风险低、改造面小，不影响核心业务流。

### 3.2 企业常用增强方案（S2，建议紧跟）

目标：让指标从“能看”升级到“可运营”。

方案：

1. 维度化指标输出：
   - 拆出 phase/final 分桶。
   - 区分 trigger source（`polling` / `event`）。
2. 补充 backlog 指标：
   - `queuedPhaseJobs`、`queuedFinalJobs`、`oldestQueuedAgeSecs`。
   - 这些指标比单纯失败计数更能反映“是否阻塞”。
3. 暴露派生指标：
   - `dispatchSuccessRate`、`retryShare`、`contractFailureShare`。
   - 由服务端统一口径输出，避免前端/监控平台各算各的。

为什么符合产品需求：

1. 可直接支撑“健康大盘 + 自动告警 + 值班 SOP”。
2. 降低人工读日志依赖，提高运维自助能力。

### 3.3 平台级治理方案（S3，上线前完成）

目标：完成企业生产级安全与跨实例观测闭环。

方案：

1. 内部鉴权升级：
   - 从静态 key 升级到 `key-id + HMAC(signature,timestamp,nonce)`。
   - 增加时间窗校验、防重放缓存、密钥轮换机制。
2. 统一指标出口：
   - 接入 Prometheus/OpenTelemetry 指标导出。
   - API 052 保留为调试/运维补充接口，不作为唯一观测入口。
3. 跨实例可解释性：
   - 增加 `instanceId`、`processStartAtMs` 字段。
   - 告警规则使用“rate + 标签聚合”而非单实例绝对值。

为什么符合产品需求：

1. 满足生产安全基线。
2. 支撑水平扩容后的稳定运维。
