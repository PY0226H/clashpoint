## 2.2 逐项深度分析

### `USession-R1`：契约声明不完整

- 现象：OpenAPI 只写了 `200/400/404/409`。
- 实际：中间件和 extractor 路径会产生 `401/403/422`，系统错误会有 `500`。
- 影响：前端/SDK 按文档做分支时会漏处理，联调成本高。

### `USession-R2`：写接口无限流

- 现象：update handler 未接入 `request_guard`。
- 风险：后台误操作脚本或自动重试可在短时间内刷写单条/多条场次。
- 后果：DB 压力、告警噪声和排障复杂度上升。

### `USession-R3`：并发更新“最后写入覆盖”

- 现象：无 `expectedUpdatedAt`/revision 检查；两个运营同时编辑时后写覆盖先写。
- 风险：配置漂移，值班中难以判断“最终值为何被覆盖”。
- 备注：`FOR UPDATE` 只保证单事务内部一致，不解决跨请求意图冲突。

### `USession-R4`：缺更新审计

- 现象：目前 `ops_debate_session_audits` 仅用于 create/create_replay。
- 风险：无法回答“谁改了状态/时间/人数，改前改后是什么”。

### `USession-R5`：ID 边界转换风险

- 现象：`session_id as i64`。
- 风险：超 `i64::MAX` 输入会发生符号翻转语义，不应落到 SQL 层。
- 影响：低频但高隐蔽，属于边界治理缺口。

### `USession-R6`：状态机缺方向约束

- 现象：`scheduled/open/running/judging/closed/canceled` 均可任意互转。
- 风险：例如 `closed -> open`、`canceled -> running` 会带来业务语义反转。
- 后果：裁判链路、用户预期、运营通知语义可能冲突。

### `USession-R7`：`actual_start_at` 语义缺失

- 现象：update 只改 `status/scheduled_start_at/end_at/max_participants_per_side`，不动 `actual_start_at`。
- 风险：手工设置 `running` 后，`actual_start_at` 仍为空；数据语义不完整。
- 后果：依赖该字段的统计、审计、报表会失真。

### `USession-R8`：应用时间与数据库时间源不一致

- 现象：校验用 `Utc::now()`，SQL `joinable` 计算/列表查询用 `NOW()`。
- 风险：应用机与 DB 时钟偏差时，边界请求会出现“校验通过但返回不可加入/反之”。

### `USession-R9`：过期场次可被更新为 `scheduled`

- 现象：仅 `open/running` 强制 `endAt > now`，`scheduled`/`judging`/`closed`/`canceled` 未统一未来时态。
- 风险：可能产生“计划态但实际已结束”的脏语义（与 create 端收紧策略不一致）。

### `USession-R10`：锁等待时间不可控

- 现象：`SELECT ... FOR UPDATE` 无 lock_timeout。
- 风险：当存在长事务占锁时，ops 请求可能长时间阻塞，影响控制台可用性。

### `USession-R11`：route 测试矩阵薄弱

- 现象：当前 update 主要有 model 层用例，route 层未见完整矩阵。
- 风险：中间件或 extractor 变更后，问题只能在联调阶段暴露。

### `USession-R12`：错误机读稳定性不足

- 现象：400/409 多为文案错误字符串。
- 风险：客户端只能做 brittle 字符串匹配，不利于跨端一致性和本地化。

### `USession-R13`：`joinable` 口径漂移

- 现象：update 返回的 `joinable` 公式不含容量条件；list 已含容量条件。
- 后果：更新响应里的 `joinable` 与列表页读取可能不一致。

### `USession-R14`：计划变更无实时广播

- 现象：DB trigger 仅在 `status` 变化时 `pg_notify`。
- 风险：仅改 `scheduled_start_at/end_at/maxParticipantsPerSide` 时，客户端无推送感知。
- 后果：运营改期后，用户端可能在较长时间内看到旧值（依赖主动刷新）。

## 3. 解决方案（大厂实践 + 推荐路径）

## 3.1 方案 A（推荐，分阶段硬化）

目标：在不引入复杂兼容层前提下，把 API056 从“可用更新接口”升级到“治理型写接口”。

### A-P0（优先落地）

1. **补齐 OpenAPI 响应谱系（`USession-R1`）**
- handler 注解补齐：`401/403/422/500`。
- 收益：契约与运行态一致，前端/SDK 联调稳定。

2. **`session_id` 安全转换（`USession-R5`）**
- 使用 `safe_u64_to_i64(session_id, "debate_session_id_invalid")`。
- 收益：边界输入稳定收敛为 400，避免符号翻转。

3. **`joinable` 公式统一（`USession-R13`）**
- update 返回与 list 使用同一口径（包含容量约束）。
- 收益：写后读语义一致，减少前端状态闪烁。

4. **route 最小矩阵补齐（`USession-R11`）**
- 至少覆盖 `200/400/401/403/404/409/422/500`。
- 收益：中间件、提取器、业务校验回归可持续守护。

### A-P1（强烈建议）

5. **接入更新侧限流（`USession-R2`）**
- user/ip 双限频；异常时返回稳定 `429` code。
- 收益：运营误操作和脚本风暴有硬闸门。

6. **接入乐观并发控制（`USession-R3`）**
- 请求体增加可选 `expectedUpdatedAt`（或 `revision`）。
- `WHERE id=$1 AND updated_at=$expected`，不匹配返回稳定 `409 revision_conflict`。
- 收益：多运营并发编辑不再 silent overwrite。

7. **补更新审计（`USession-R4`）**
- 复用 `ops_debate_session_audits`，新增 `action=update`。
- 审计内容建议记录：操作者、会话ID、字段差异（before/after JSON）、request_id。
- 收益：值班追溯和合规审计闭环。

8. **统一时间源与时态规则（`USession-R8`/`R9`）**
- 在 `FOR UPDATE` 查询里带 `NOW() AS db_now`，校验统一基于 DB 时钟。
- 冻结策略：若产品不允许计划态过期，则对 `scheduled` 同样要求 `endAt > db_now`。
- 收益：跨机器时钟漂移风险收敛，create/update 策略一致。

9. **状态机治理（`USession-R6`/`R7`）**
- 方案建议：
  - 默认状态转换矩阵（禁止明显逆向跳转）。
  - 保留“运营紧急 override”但要求显式参数（如 `force=true`）并强审计。
  - 当状态进入 `open/running` 且 `actual_start_at` 为空时自动回填 `db_now`。
- 收益：既保留运维逃生能力，又减少语义污染。

10. **锁等待治理（`USession-R10`）**
- 设置合理 `lock_timeout`（例如 500ms~1000ms）并映射稳定冲突码。
- 收益：避免长阻塞，提升运营台可用性。

### A-P2（体验与平台化）

11. **结构化错误模型（`USession-R12`）**
- 推进 `{code,message,details?}` 统一，客户端完全机读 `code`。

12. **计划变更事件补齐（`USession-R14`）**
- 对 `scheduled_start_at/end_at/max_participants_per_side` 的变化引入事件（或复用 outbox）。
- 收益：客户端与 ops 面板可近实时对齐排期。

## 3.2 方案 B（平台化统一治理管线）

适用于后续 Ops 写接口全面收口：

1. 抽象统一写管线：`鉴权 -> 限流 -> 并发控制 -> 事务写入 -> 审计 -> 观测`。
2. 统一错误码、审计 schema、回归测试模板。
3. 让 topic/session/create/update 共享同一治理基座。

优点：长期一致性和维护成本最佳；缺点：一次性改造面较大。

## 3.3 最终推荐（供你审核）

建议采用 **方案 A（P0->P1）**，按“先正确性与契约、再治理增强”的顺序推进。

推荐落地顺序：

1. `USession-R1/R5/R13/R11`（契约 + 边界 + 返回语义一致 + 回归守护）
2. `USession-R2/R3/R4`（限流 + 并发治理 + 审计）
3. `USession-R8/R9/R6/R7/R10`（时间源统一 + 状态机治理 + 锁等待治理）
4. `USession-R12/R14`（错误结构化 + 计划变更实时化）

## 4. 验收建议（DoD 草案）

1. OpenAPI 与运行态错误谱系一致（含 401/403/422/500）。
2. update 具备 user/ip 限流与稳定冲突语义。
3. 并发更新具备 revision 冲突检测，不再 silent overwrite。
4. 更新审计可追溯（含字段 diff）。
5. update 返回 `joinable` 与 list 读取口径一致。
6. route 回归矩阵通过，覆盖中间件与 extractor 失败路径。
