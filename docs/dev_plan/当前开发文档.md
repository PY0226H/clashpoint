# API058 开发文档：`GET /api/debate/ops/rbac/roles`

更新时间：2026-04-08
范围：围绕 `GET /api/debate/ops/rbac/roles` 及其治理协同接口（`GET /api/debate/ops/rbac/me`、`PUT/DELETE /api/debate/ops/rbac/roles/{userId}`）的完整链路（路由、中间件、RBAC 服务、前端消费、运维治理）进行问题分析与方案设计。

---

## 一、当前实现快照（问题分析基线）

1. 路由：`/api/debate/ops/rbac/roles` 挂在 `phone_required_api` 下，先过 `verify_token_header_only` 与 `require_phone_bound`。
2. Handler：`list_ops_role_assignments_handler` 已接入 user/ip 双限流、读审计日志、专项计数与延迟采样，再调用 `state.list_ops_role_assignments_by_owner(&user)`。
3. 权限校验：`list_ops_role_assignments_by_owner` 内部调用 `ensure_platform_admin_for_ops_rbac`，仅平台 Owner 可访问。
4. Owner 真源：`get_platform_admin_user_id()` 当前读取 `platform_admin_owners`（不再硬编码 `user_id=1`）。
5. 查询逻辑：`platform_user_roles JOIN users`，按 `updated_at DESC, user_id DESC` 返回角色分配；`rbac/me` 与 `rbac/roles` 已补齐 `rbacRevision`，且 `rbac/roles` 默认 `piiLevel=minimal`（可显式请求 `full`）。
6. OpenAPI：`GET /api/debate/ops/rbac/roles`、`GET /api/debate/ops/rbac/me`、`PUT/DELETE /api/debate/ops/rbac/roles/{userId}` 已声明 `429` 并对齐主要错误谱系（`200/400/401/403/404/409/429/500`）。
7. 前端消费：`frontend/packages/ops-domain` 与 `OpsConsolePage` 直接调用并展示角色分配结果。
8. 测试：已覆盖 `rbac/me`、`rbac/roles`、`rbac/roles/{userId}` 的 route 回归矩阵（含 `200/400/401/403/404/409/429/500` 关键路径）。

## 2.2 逐项深度分析

### `RBACROLES-R1` OpenAPI 与运行态不一致（P0）

- 现象：`list_ops_role_assignments_handler` 注解只写 `200/409`。
- 事实：运行态还会出现 `401`（token）、`403`（phone gate）、`500`（owner 真源缺失或 SQL 异常）。
- 风险：
  1. SDK/前端错误分支覆盖不完整。
  2. 联调时“文档显示不会出现”的状态码被线上返回，导致返工。

### `RBACROLES-R2` 缺 route 级回归（P0）

- 现象：当前主要有 model 测试，缺此接口 route 测试。
- 风险：
  1. 中间件顺序或行为变化（鉴权/手机门禁）可能静默破坏接口。
  2. OpenAPI 改动与真实行为易再次漂移。

### `RBACROLES-R3` owner 真源缺失恢复路径不显式（P0）

- 现象：`platform_admin_owners` 缺失会触发 `platform_owner_not_configured`（500）。
- 风险：
  1. 一旦配置异常，整个角色管理读写面不可用。
  2. 缺少标准化 runbook 时，值班定位恢复慢。

### `RBACROLES-R4` 权限拒绝状态码语义争议（P1）

- 现象：非 Owner 返回 `409 DebateConflict`。
- 风险：
  1. 跨端通常将权限拒绝理解为 `403`，而 `409` 常用于资源冲突。
  2. 语义不统一会增加客户端特判与认知负担。
- 收口结论：RBAC 管理面本阶段冻结为 `409`，并在 `GET/PUT/DELETE /ops/rbac/roles*` 的 OpenAPI 与 route 回归中保持一致。

### `RBACROLES-R5` 缺限流（P1）

- 现状：`rbac/roles`、`rbac/me`、`rbac/roles/{userId}` 已落地 user/ip 双限流，统一返回 `429 rate_limit_exceeded:*`。
- 剩余风险：
  1. 阈值仍是首版经验值，需要结合值班看板继续调优。
  2. 尚未形成压测基线报告，不利于容量评审。

### `RBACROLES-R6` 可观测不足（P1）

- 现状：已落地 `ops_rbac_roles_list_*`、`ops_rbac_me_*`、`ops_rbac_roles_write_*` 计数与 latency 采样，并补齐 `audit_event/user_id/request_id/decision` 关键 tracing 字段；`ops metrics dictionary` 已纳入 `ops.rbac.*` key。
- 剩余风险：
  1. dashboard 与告警阈值尚未配置，值班仍依赖日志检索。
  2. 延迟分位数看板未落地，趋势分析能力不足。

### `RBACROLES-R7` 读审计缺失（P1）

- 现状：`rbac/roles` 读取、`rbac/me` 读取、`rbac/roles/{userId}` 写入均已输出结构化审计日志（含 `request_id` 与关键业务字段），并已落地 `ops_rbac_audits` 审计表（记录 success/failed/rate_limited_user/rate_limited_ip）；同时已补齐受控查询脚本与留存清理脚本。
- 剩余风险：
  1. 留存策略虽有可执行基线（180 天 + 批次清理），但合规侧最终周期与归档层策略尚未冻结。
  2. 当前是“脚本级受控查询面”，是否升级为 API 查询面仍待评审。

### `RBACROLES-R8` 错误模型机读稳定性不足（P1）

- 现状：后端 `ErrorOutput` 已扩展为 `error + code + message + details?`，客户端 `toOpsDomainError` 已升级为“前缀剥离 + code 归一 + code 优先分支”。
- 风险：
  1. 仍需在更多非 RBAC 域逐步收敛 code 命名规范。
  2. `details` 字段语义（可选结构）还需后续统一约定。

### `RBACROLES-R9` 全量返回长期风险（P2）

- 现象：当前无分页/limit。
- 当前合理性：角色分配规模目前极小。
- 已推进：`rbac/me` 与 `rbac/roles` 已引入 `rbacRevision`，前端可基于版本字段做缓存与刷新一致性治理。
- 长期风险：当组织规模扩展或引入多租户后，响应体与查询成本会变大。

### `RBACROLES-R10` ID 强转边界防护不足（P2）

- 现状：`map_assignment_row` 已改为显式 `u64::try_from` 安全转换，异常时返回稳定 server code。
- 风险：
  1. 正常路径风险已显著降低。
  2. 后续仍需结合数据质量巡检，确保异常码不会被静默吞掉。

### `RBACROLES-R11` PII 最小化策略未固化（P2）

- 现状：`rbac/roles` 已收口为默认 `piiLevel=minimal`（邮箱/姓名脱敏），并支持显式 `piiLevel=full`（用于 owner 排障）。
- 剩余风险：
  1. full 模式触发条件与审计要求尚未形成制度化策略（例如操作留痕、审批约束、导出限制）。
  2. 后续若权限边界变化，仍需持续校验脱敏策略是否随权限模型同步更新。

### `RBACROLES-R12` 手机门禁策略待产品冻结（P2）

- 现象：接口强依赖 `require_phone_bound`。
- 风险：
  1. Owner 应急账号若未绑手机，无法查看角色分配，可能影响应急操作。
  2. 但从安全角度看，门禁可提升控制面安全。
- 结论：需要产品与安全策略共同定稿。

## 三、解决方案（企业主流 + 推荐）

## 3.1 方案候选

### 方案 A（推荐，短中期主流）：契约与可观测先收口

核心目标：先把“文档-运行态一致性”和“回归防线”补齐，快速降低生产不确定性。

- 关键动作：
  1. OpenAPI 补齐 `401/403/500`（保留 `200/409`）。
  2. 增加 route 回归：`200/401/403/409/500`。
  3. 增加最小指标：`success/forbidden/conflict/error` 计数 + `latency`。
  4. 增加 owner 真源缺失 runbook（自动修复或手动恢复步骤）。

- 优点：
  1. 投入低、见效快。
  2. 直接解决 P0 风险。
  3. 不引入大规模架构变更。

### 方案 B（推荐，企业标准化）：RBAC 管理面治理统一

核心目标：把 `rbac/me`、`rbac/roles`、`rbac/roles/{userId}` 三个接口作为一个治理域统一收敛。

- 关键动作：
  1. 统一结构化错误模型：`{ code, message, details? }`。
  2. 统一权限拒绝语义（是否从 `409` 迁移到 `403`，需一次性冻结规范）。
  3. 引入读取审计（至少记录 owner、请求时间、结果数、request_id）。
  4. 增加 `revision` 字段为前端缓存/刷新提供稳定依据。

- 优点：
  1. 跨端行为更稳定。
  2. 值班排障效率显著提升。
  3. 为后续角色治理扩展打基础。

### 方案 C（长期）：策略引擎与数据最小化升级

- 关键动作：
  1. 抽象 policy 层统一权限判定。
  2. 按场景分级输出 PII（可选脱敏模式）。
  3. 分页化与多租户扩展预案。

- 结论：当前阶段可规划但不建议立即重投入。
