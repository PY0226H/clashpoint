# API057 开发文档：`GET /api/debate/ops/rbac/me`

更新时间：2026-04-08
范围：围绕 `GET /api/debate/ops/rbac/me` 的完整链路（路由、中间件、RBAC 服务、跨模块消费、前端依赖）做问题分析与方案设计。

---

## 一、当前实现快照（用于问题分析基线）

1. 路由：`/api/debate/ops/rbac/me` 挂在 `phone_required_api` 下，先经过 `verify_token_header_only`、`require_phone_bound`。
2. Handler：`get_ops_rbac_me_handler` 仅调用 `state.get_ops_rbac_me(&user)` 返回 200。
3. Service：
   - Owner 判断：`get_platform_admin_user_id()` 当前硬编码返回 `1`
   - Owner 分支：`is_owner=true`，四项权限全 `true`，`role=null`
   - 非 Owner 分支：查 `platform_user_roles` 后按角色矩阵映射权限，`role_manage=false`
4. 角色表：`platform_user_roles.role` 受 CHECK 约束（`ops_admin/ops_reviewer/ops_viewer`）。
5. 跨模块依赖：`get_ops_rbac_me` 被 debate topic 可见性、IAP 产品管理视图、裁判报告读权限、Ops 前端控制台共同依赖。
6. 测试现状：有 model 测试与前端 mock e2e；缺后端 route 级真实链路用例。

## 2.2 逐项深度分析

### `RBACME-R1` Owner 真源硬编码（P0）

- 现象：`get_platform_admin_user_id()` 固定返回 `1`。
- 根因：当前实现把平台 Owner 作为代码常量而非数据配置。
- 风险放大：
  1. 环境内若超级账号不是 `id=1`，Owner 判定将失真。
  2. `role_manage` 能力与真实管理者错位，可能导致“有管理员但无法管角色”。
  3. 权限判定逻辑在多环境迁移时脆弱。
- 影响面：Ops 角色管理、Ops Console 展示、跨模块权限判定。

### `RBACME-R2` Owner 与角色表双真源（P0）

- 现象：`grant_platform_admin` 可以把用户置为 `ops_admin`，但 `isOwner` 仍只认 `user_id=1`。
- 根因：Owner 身份和角色体系并行存在，且无统一主权威。
- 风险：
  1. 权限语义不一致：用户是 `ops_admin` 但不是 owner，`role_manage=false`。
  2. 客户端认知混乱：同为“管理员”，可见与可操作能力不同步。

### `RBACME-R3` `role_manage` 仅绑定 id=1（P0）

- 现象：非 Owner 即便 `ops_admin` 也不能管理角色。
- 根因：`role_manage` 不在通用权限矩阵里，且无可配置机制。
- 风险：
  1. 运维接班/轮值困难，无法安全移交。
  2. 一旦 Owner 账号异常，角色管理面可能“单点失效”。

### `RBACME-R4` OpenAPI 与运行态契约不一致（P1）

- 现象：注解只声明 200，但运行态存在 `401/403/500`。
- 根因：文档注解遗漏中间件失败路径与内部异常路径。
- 风险：SDK/前端错误处理分支不足，联调返工。

### `RBACME-R5` 缺 route 级后端回归（P1）

- 现象：当前主要是 model 层单测，缺真实路由链路验证。
- 根因：历史上更关注 service correctness，较少覆盖中间件组合。
- 风险：鉴权门禁、序列化、注解与 handler 连接处变更难以及时发现。

### `RBACME-R6` 错误机读模型弱（P1）

- 现象：大量基于 `error` 字符串（如 `ops_permission_denied:*`）驱动前端分支。
- 根因：缺统一 `{code,message,details}` 结构化错误治理。
- 风险：文案变更会造成客户端逻辑脆断。

### `RBACME-R7` 可观测与审计不足（P1）

- 现象：`rbac/me` 无专属 metrics/tracing 维度，无策略命中审计。
- 根因：接口作为“读接口”被低估了控制平面价值。
- 风险：权限异常时定位慢，无法快速区分“数据问题 vs 策略问题”。

### `RBACME-R8` 快照一致性策略未明确（P2）

- 现象：角色变更后前端缓存策略未与后端版本号联动。
- 根因：缺 `rbac_revision` 或 etag 机制。
- 风险：短时间显示旧权限，引发“按钮存在但后端拒绝”体验波动。

### `RBACME-R9` 权限语义分散（P2）

- 现象：同一 RBAC 结果被多个模块消费，各自做业务判定。
- 根因：缺统一权限策略契约层（policy module）。
- 风险：后续新增权限时多处修改，易产生行为漂移。

### `RBACME-R10` 手机门禁与运维场景冲突（P2）

- 现象：`rbac/me` 受 `require_phone_bound` 约束。
- 根因：路由组统一门禁策略。
- 风险：紧急运维账号若未绑手机，连“查看自身权限”都不可达。
- 说明：此项是否为问题取决于产品安全策略，需产品确认。

## 三、解决方案（企业主流 + 推荐方案）

## 3.1 方案候选

### 方案 A（推荐，企业主流）：统一 Owner 真源 + 策略收敛

核心思路：把“Owner 身份”从硬编码迁移到数据库真源，所有 RBAC 入口统一读取该真源。

- 设计要点：
  1. 新增平台治理表（示例：`platform_admin_owners` 或 `platform_settings(owner_user_id)`）。
  2. `get_platform_admin_user_id()` 改为查表（无记录时显式错误，不再静默假设 1）。
  3. `ensure_platform_admin_for_ops_rbac`、`get_ops_rbac_me`、相关权限入口统一使用同一读取函数。
  4. 保持 pre-release 硬切换，不做长期双轨兼容。
- 优点：
  1. 彻底消除 `id=1` 假设。
  2. Owner/角色语义可治理、可迁移、可审计。
  3. 为后续多 owner 或租户化打基础。
- 成本：中等，需要 migration + 服务改造 + 测试补齐。

### 方案 B（过渡轻量）：Owner ID 配置化（Env）

- 把 Owner ID 从硬编码提到配置项（如 `ECHO_PLATFORM_OWNER_USER_ID`），服务启动时加载。
- 优点：改动小、可快速落地。
- 缺点：仍是配置真源，非数据真源；运维变更可追踪性弱。

### 方案 C（长期演进）：Owner 与 Role 完全策略化（Policy Engine）

- 引入统一策略层：`is_owner`、`role_permissions`、`resource_scope` 一体判定。
- 优点：扩展性最强。
- 缺点：当前阶段过重，不符合“未发布阶段避免过度设计”的节奏。

   - 补 OpenAPI：`401/403/500`。

## 3.3 具体实施建议（可直接进入开发计划）

### 阶段 1：真源统一

1. migration：新增 Owner 真源表并初始化。
2. `rbac.rs`：`get_platform_admin_user_id()` 改查库。
3. `lib.rs` 启动引导：dev super 初始化后确保 Owner 真源一致。

### 阶段 2：契约与测试封板

1. `debate_ops.rs`：补齐 `get_ops_rbac_me_handler` OpenAPI 响应码声明。
2. 新增 handler route 测试（真实中间件链），覆盖 200/401/403。
3. 增加异常路径测试（Owner 真源缺失时的错误行为）。

### 阶段 3：稳定性治理

1. 增加 tracing 字段：`user_id/is_owner/role/source`（注意脱敏）。
2. 增加 metrics 计数器与延迟 histogram。
3. 预留 `rbac_revision` 字段（或在响应中新增 `policyVersion`）。

## 3.4 验收标准（Definition of Done）

1. 不再出现任何 `owner_id=1` 硬编码判定路径。
2. `GET /api/debate/ops/rbac/me` OpenAPI 文档包含 `200/401/403/500`。
3. 后端 route 级测试覆盖主路径并稳定通过。
4. 在“Owner 非 1”场景下，`isOwner` 与 `roleManage` 行为仍正确。
5. Ops Console 权限显示与后端授权行为一致，无明显漂移。

## 四、未解决问题（本轮保留）

1. `RBACME-R10`：手机门禁是否应对运维紧急账号放宽，需要产品最终裁决。

## 五、结论

`/ops/rbac/me` 虽然是读接口，但属于 Ops 控制平面的“根接口”。当前最大风险不是性能，而是权限真源一致性。建议优先完成 Owner 真源统一（方案 A）并封住文档/测试/可观测三条防线，这样后续所有 Ops 能力才有稳定地基。
