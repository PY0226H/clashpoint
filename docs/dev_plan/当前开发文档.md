# 当前开发文档（API 059：`PUT /api/debate/ops/rbac/roles/:user_id`）

更新时间：2026-04-09
范围：围绕 Ops RBAC 写接口完整链路（鉴权、门禁、限流、权限、DB、审计、观测、测试）进行问题分析与方案设计。

---

## 1. 当前实现链路快照（基线）

当前请求主链路：

1. `verify_token_header_only`
2. `require_phone_bound`
3. `upsert_ops_role_assignment_handler`
4. user/ip 双限流（`ops_rbac_roles_write`）
5. `idempotency-key`（可选）+ Redis lock（fail-open）
6. `upsert_ops_role_assignment_by_owner_with_meta`（Owner/委派管理员）
7. `platform_user_roles` 原子 CTE UPSERT + 幂等重放窗口
8. `ops_rbac_audit_outbox_jobs` 入队 + `dispatch_once` 投递 `ops_rbac_audits`（失败重试）+ `x-rbac-revision` 响应头

当前关键特征：

1. Owner 真源已经切到 `platform_admin_owners`（非硬编码 `1`）。
2. 角色输入已归一化并白名单约束。
3. 写接口具备 user/ip 双限流，且 Redis Disabled 有本地 fallback。
4. PUT 已支持可选 `idempotency-key` 重放窗口，减少重试噪音。
5. 审计失败已具备 `error_code/failure_reason` 分类能力。
6. `role_manage` 已支持受限委派（`platform_role_admin`），并限制其不能管理 Owner/同级委派角色。
7. RBAC 审计已接入 outbox 可回放链路（enqueue + retry）。
8. 并发语义仍是 last-write-wins（无显式版本冲突协议）。
9. PUT RBAC 对 415/422 提取失败已纳入专项审计与指标（不再仅依赖 access log）。

### 2.2 深入分析

#### `RBACPUT-R1`：Owner 真源缺失导致控制平面写接口不可用（P0）

- 触发条件：`platform_admin_owners` 无 `singleton_key=true` 行。
- 根因：运行时没有强约束“owner 真源一定存在”，仅在调用时抛错。
- 影响：所有 RBAC 写操作 500，运维无法通过 API 恢复角色配置。
- 当前防线：migration 初始化插入 `id=1`（但无法覆盖后续误删或脏库场景）。
- 残余风险：环境恢复/迁移阶段易被触发，属于“控制面硬故障”。

#### `RBACPUT-R2`：`u64` 到 `i64` 的无保护转换（P0）

- 触发条件：path `user_id > i64::MAX`。
- 根因：`target_user_id as i64` 为截断语义，不是安全转换。
- 影响：查询/写入目标用户出现异常语义（可能映射为负数），错误码与真实原因不一致。
- 当前防线：无。
- 残余风险：边界值攻击和异常输入下可观测性差，且不符合严格输入契约。

#### `RBACPUT-R3`：预检 + 写入分离导致竞态窗口（P1）

- 触发条件：`SELECT users` 通过后，目标用户在 UPSERT 前被删除。
- 根因：存在 read-check-write 时间窗。
- 影响：可能从预期 `404` 退化为数据库层外键失败（5xx 或非预期错误）。
- 当前防线：有预检但非原子。
- 残余风险：在高并发/批处理环境下概率增大。

#### `RBACPUT-R4`：角色管理权单点（P1，S2-1 已收敛）

- 触发条件：Owner 不在线、账号临时不可用、应急交接。
- 根因：`role_manage` 被硬绑定 Owner，未支持治理委派。
- 影响：控制面运维弹性不足，组织层面存在单点瓶颈。
- 当前防线：已引入 `platform_role_admin` 委派能力，Owner 之外可执行 RBAC 角色管理。
- 受限边界：委派管理员不能管理 Owner，也不能分配/撤销 `platform_role_admin`，防止委派链条失控。
- 残余风险：仍需后续补齐并发冲突协议（`R5`）与审计可靠投递（`R9`）。

#### `RBACPUT-R5`：并发写冲突不可见（P1）

- 触发条件：多操作者近同时更新同一 `user_id` 角色。
- 根因：UPSERT 缺少版本条件（如 `expected_updated_at` / `revision`）。
- 影响：后写覆盖前写，调用方无法识别冲突，误以为配置稳定。
- 当前防线：审计日志可回溯，但是事后能力。
- 残余风险：控制平面操作误覆盖概率随团队协作增加而上升。

#### `RBACPUT-R6`：写接口缺少幂等协商（P1，S1 已收敛）

- 触发条件：客户端重试/网关重放/超时后补发。
- 根因：PUT 语义幂等依赖“最终状态相同”，但当前实现每次都会刷新 `updated_at` 并落审计。
- 影响：审计噪音增加，时序分析复杂，`updated_at` 失真（看起来像真实角色变更）。
- 当前防线：S1 已引入可选 `idempotency-key` + 30s TTL 重放窗口，并在 replay 场景抑制重复 success 审计。
- 残余风险：TTL 窗口外重复请求仍按新写处理，属预期语义。

#### `RBACPUT-R7`：限流在 Redis 关闭或 fail_open 场景可能失效（P1，S1 已收敛）

- 触发条件：Redis disabled 或启动策略 fail_open 且 Redis 不可达。
- 根因：`RedisStore::Disabled` 对限流直接返回 `allowed=true`。
- 影响：写接口在退化场景丧失限流保护，增加恶意/误操作放大风险。
- 当前防线：S1 已在 RBAC 写路径启用 Disabled 本地 fallback，同时保留 Redis 运行时错误 fallback。
- 残余风险：本地 fallback 为进程内计数，跨实例总量仍需网关/集中层协同控制。

#### `RBACPUT-R8`：IP 限流在无转发头时共享 `unknown` 桶（P1）

- 触发条件：代理未透传 `x-forwarded-for` / `x-real-ip`。
- 根因：IP key 缺失回退常量 `unknown`。
- 影响：不同真实客户端互相干扰，出现误限流；同时失去真实来源区分。
- 当前防线：仍有 user 维度限流。
- 残余风险：BFF 或网关接入初期常见，问题隐蔽。

#### `RBACPUT-R9`：审计 best-effort 可能丢失关键合规证据（P1，S2-2 已部分收敛）

- 触发条件：审计写库失败（DB 短暂抖动、连接池问题等）。
- 根因：`insert_ops_rbac_audit_log_best_effort` 仅 warning，不反压主请求。
- 影响：控制平面关键写操作可能“执行成功但无审计记录”。
- 当前防线：S2-2 已增加 `ops_rbac_audit_outbox_jobs`，请求先入 outbox，再尝试即时投递到 `ops_rbac_audits`；失败任务可重试回放。
- 残余风险：当前为“请求驱动 dispatch_once”，尚未独立 worker 化，长时间低流量时重试推进速度受限。

#### `RBACPUT-R10`：审计维度过粗（P2，S1 已收敛）

- 触发条件：`failed` 分支。
- 根因：审计表 `decision` 枚举过少。
- 影响：后期难按失败类型聚合（权限拒绝、参数错误、系统失败被混在一起）。
- 当前防线：S1 已新增 `error_code/failure_reason` 字段，失败分支可按类型聚合查询。
- 残余风险：审计仍是 best-effort，可靠投递问题归属 `R9/S2-2`。

#### `RBACPUT-R11`：Extractor 失败路径观测缺口（P2，S2-3 已收敛）

- 触发条件：请求体非法、Content-Type 异常。
- 根因：`Json` 提取失败发生在 handler 执行前。
- 影响：这类请求不会进入 RBAC 写接口专项指标与审计落库。
- 当前防线：S2-3 已将 PUT handler 调整为显式接收 `JsonRejection`，并在 rejection 分支补齐失败分类、审计 outbox 入队与指标统计。
- 落地细节：新增稳定 machine code（`ops_rbac_roles_write_content_type_invalid` / `ops_rbac_roles_write_body_invalid_json` 等），并补齐 415/422 route 回归与审计断言。
- 残余风险：当前收敛范围仅覆盖 API 059 PUT 路径，其他控制面写接口仍需按同模式补齐。

#### `RBACPUT-R12`：Owner 自写角色语义易误解（P2）

- 触发条件：Owner 给自己写 `ops_viewer/ops_admin/...`。
- 根因：Owner 权限判断走 `platform_admin_owners` 短路，不依赖 `platform_user_roles`。
- 影响：UI/运维若误把角色行当作 Owner 权限来源，会产生错误认知。
- 当前防线：代码逻辑正确，但文档与控制台提示不足。
- 残余风险：人为误操作沟通成本。

#### `RBACPUT-R13`：回归测试尚未覆盖关键边界（P2）

- 触发条件：owner 真源缺失、IP 限流分支、超大 user_id 输入等。
- 根因：测试重点在主干 happy-path 与典型错误。
- 影响：回归时可能出现“线上才暴露”的边界问题。
- 当前防线：现有 route + model 测试。
- 残余风险：控制面稳定性受影响。

## 3. 解决方案（企业主流 + 推荐落地）

### 3.1 企业主流方案（治理完备型）

适用场景：多环境、多运维角色、合规审计要求高。

#### 方案 A：控制平面治理标准化

1. **权限模型升级**：在 Owner 之外引入可委派 `role_manage` 能力（如 `platform_role_admin`）。
2. **并发写协议**：为角色记录引入 `revision`（或 `expectedUpdatedAt`），冲突返回 `409` 机器码。
3. **幂等写协议**：支持 `idempotency-key`，短窗口去重重放。
4. **审计强一致策略**：角色写入与审计通过同事务 outbox 保证“业务成功必有审计”。
5. **限流治理固化**：控制平面写接口在生产启用 Redis fail_closed；IP 来源限定可信网关头。
6. **错误契约标准化**：所有失败路径统一 machine code，并在 OpenAPI 明确。

优点：治理能力完整、适配企业安全审计。
代价：改造范围较大，涉及 schema、协议、前后端联动。

### 3.2 我推荐的方案（适配 EchoIsle 当前阶段）

基于“项目尚未正式发布、可硬切换”的前提，建议走 **分阶段硬切**，优先消灭高风险点。

#### 阶段 S0（本迭代，必须完成）

1. **修复输入边界**：`u64 -> i64` 改 `try_from`，越界直接 `400 ops_role_target_user_id_out_of_range`。
2. **消除竞态窗口**：将“目标用户存在校验 + UPSERT”收敛为单 SQL（CTE/原子写法）。
3. **补齐关键测试**：
   - owner 真源缺失（PUT 500）
   - IP 限流触发
   - `user_id` 越界
4. **运维守护**：启动自检或健康检查增加 `platform_admin_owners` 存在性断言，避免隐式 500。

#### 阶段 S1（下一迭代，强烈建议）

1. **写侧幂等键**：为 PUT 增加可选 `idempotency-key`（TTL 10~30s）。
2. **审计字段增强**：在 `ops_rbac_audits` 增加 `error_code`（或 `failure_reason`），提升可检索性。
3. **限流可控退化**：当 Redis Disabled 时，至少启用进程内 fallback（而不是完全放开）。
4. **响应版本线索**：返回 `rbacRevision`（便于前端刷新判定）。

#### 阶段 S2（中期演进）

1. **委派治理能力**：在 Owner 模式外引入受限“角色管理员”能力。
2. **审计可靠投递**：从 best-effort 升级为“可回放审计 outbox”。
3. **控制平面可观测看板**：分离成功/权限拒绝/参数错误/系统错误四类指标。

推荐原因：

1. 先解 P0/P1 实际稳定性风险，最小代价提升上线安全边际。
2. 不引入长期兼容层，符合当前“预发布可硬切”的项目策略。
3. 每阶段都可独立验收，失败可局部回滚。

## 6. 执行进展补充（2026-04-08，S1）

5. 回归增强：新增幂等 replay、幂等 key 长度边界、Disabled fallback、失败分类断言等 route 级测试。
