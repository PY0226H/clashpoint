# 在线辩论 + AI裁判平台压测方案 v1

- 文档版本: `v1`
- 生成日期: `2026-02-27`
- 适用项目: `/Users/panyihang/Documents/EchoIsle`

## 1. 压测目标

本次压测目标不是“证明系统完美”，而是先建立可量化基线，用结果指导下一步优化。

核心目标：
1. 验证当前 MVP 在关键链路下的容量上限与退化行为。
2. 找出 P95/P99 延迟、错误率、资源瓶颈（CPU/内存/DB/连接池）。
3. 评估“预期 1w 同时在线”所需的架构与参数调整方向。

## 2. 关键结论先行（给第一次做压测）

1. 可以先在本机压测，但本机结果只能用于“发现问题”，不能用于“上线容量承诺”。
2. 要判断是否满足上线目标，必须在阿里云预发环境做接近生产拓扑的压测。
3. “先压测拿数据，再决定优化方向”是正确流程，应作为固定工程循环。

## 3. 压测范围与链路优先级

本方案优先压关键营收与核心体验链路，不做全量无差别压测。

P0 链路（先测）：
1. 辩论房间实时消息链路（WebSocket 收发、广播、状态回补）。
2. 辩论发言 + 置顶消费链路（消息写入、钱包扣费、置顶展示）。
3. AI 裁判触发到结果回写链路（request judge job -> dispatch -> report submit）。

P1 链路（第二轮）：
1. Lobby/话题列表查询链路。
2. 运营后台话题与场次管理链路。
3. 支付验单接口链路（mock 与 apple 分环境验证）。

## 4. 环境分层策略

## 4.1 阶段 A（本机基线压测）

目标：快速暴露明显性能/并发问题。

环境特征：
1. 单机运行（前后端/数据库可能同机）。
2. 网络路径极短，延迟偏理想化。
3. 结论用于问题定位，不用于容量承诺。

通过标准：
1. 关键链路无明显功能性错误（高并发下不崩溃、不大面积 5xx）。
2. 能拿到首版性能瓶颈清单。

## 4.2 阶段 B（阿里云预发压测）

目标：评估是否接近上线目标。

环境要求：
1. 拓扑尽量接近生产（应用实例、PostgreSQL、Redis/Kafka 如有）。
2. 与生产同版本配置基线（连接池、超时、限流、日志级别）。
3. 压测流量由独立压测机发起，避免与被测服务同机争抢资源。

通过标准：
1. 达到本方案第 7 节阈值。
2. 可输出“当前可承载规模 + 扩容曲线”。

## 5. 负载模型设计（v1）

## 5.1 用户行为模型（娱乐辩论场景）

按 1 个活跃辩题房间抽象：
1. 80% 用户只读（收消息、看置顶、看判决）。
2. 18% 用户普通发言（低频）。
3. 2% 用户高频互动（发言 + 置顶消费）。

建议从小到大分 4 档：
1. 档位 L1: 500 在线
2. 档位 L2: 2,000 在线
3. 档位 L3: 5,000 在线
4. 档位 L4: 10,000 在线（目标档）

## 5.2 测试类型

1. 基线测试（Baseline）: 小流量，确认系统与指标采集正常。
2. 阶梯升压（Step Load）: 每 10 分钟升一档并发，观察拐点。
3. 稳态测试（Soak）: 在目标并发运行 60-120 分钟，观察内存泄漏与抖动。
4. 峰值冲击（Spike）: 从 30% 容量快速冲到 100%，验证弹性与退化。

## 6. 工具与观测（建议栈）

压测工具（可二选一）：
1. `k6`：适合 HTTP + WebSocket 场景，脚本化简单，推荐优先。
2. `Locust`：Python 生态好，适合复杂用户行为编排。

观测指标（必须采集）：
1. 应用层：QPS、成功率、P50/P95/P99、WS 在线连接数、广播耗时。
2. 资源层：CPU、内存、文件句柄、线程数、网络带宽。
3. 数据层：PostgreSQL QPS、慢 SQL、连接池占用、锁等待。
4. 队列层（如启用 Kafka）：生产/消费速率、lag、重试率。
5. AI 链路：任务排队时延、判决完成时延、失败率、重试率。

## 7. 验收阈值（v1）

以下为 v1 建议阈值，可在第一轮数据后调整：

1. 实时消息发送接口：
   - 成功率 `>= 99.5%`
   - P95 `< 300ms`
   - P99 `< 800ms`
2. 房间 WebSocket 广播端到端：
   - P95 `< 1s`
   - 连接错误率 `< 1%`
3. 置顶消费链路（扣费+写入）：
   - 成功率 `>= 99.9%`
   - P95 `< 500ms`
4. AI 判决链路（赛后报告）：
   - `P95 <= 5 分钟`
   - 目标值 `2 分钟`

## 8. 执行步骤（可直接照做）

1. 准备压测脚本：
   - HTTP 场景：登录、发言、置顶、查询判决。
   - WebSocket 场景：建连、订阅、发消息、接收广播。
2. 打通观测面板：
   - 至少能看到应用延迟、错误率、数据库连接与慢 SQL。
3. 先跑阶段 A（本机）：
   - 仅跑 L1/L2 档，定位明显瓶颈并修复。
4. 再跑阶段 B（阿里云预发）：
   - 完整跑 L1-L4 + Soak + Spike。
5. 记录每轮结果：
   - 哪个档位开始恶化、恶化指标、对应资源瓶颈。
6. 输出优化清单并按收益排序：
   - 优先做“对 P95 和错误率影响最大的改动”。

## 9. 结果记录模板（每轮必填）

1. 测试时间与环境（本机/预发，机器规格，版本号）。
2. 压测模型（并发、用户行为比例、持续时长）。
3. 核心结果（成功率、P95/P99、错误率、最大稳定并发）。
4. 资源画像（CPU、内存、DB、连接池、队列 lag）。
5. 瓶颈判断（最先到达瓶颈的组件与证据）。
6. 优化建议（按优先级 P0/P1/P2）。

## 10. 首轮优化方向判定规则（v1）

1. 若 P95 超标且 CPU 高：优先检查热点接口、序列化、锁竞争、同步阻塞。
2. 若错误率上升且 DB 连接池打满：优先调连接池、索引、慢 SQL、批量写策略。
3. 若 WS 广播延迟高但 HTTP 正常：优先查广播风暴、房间分片、消息分发队列。
4. 若 AI 报告超时：优先查任务队列、并发额度、OpenAI 调用超时与重试策略。

## 11. 风险与边界

1. 本机压测结论不能直接映射生产容量。
2. 如果压测流量模型与真实用户行为偏差大，结论会失真。
3. 仅看平均值没有意义，必须看 P95/P99 与错误率。
4. 压测前必须固定版本与配置，否则结果不可比较。

## 12. 下一步建议

1. 先落地阶段 A（本机基线）并产出首轮瓶颈报告。
2. 基于首轮结果做一轮小优化。
3. 进入阿里云预发做阶段 B，给出“当前最大稳定并发”与扩容建议。
