## 0. 文档目的

本 PRD 用于统一 AI 与团队对产品最终形态的理解，明确：

1. 产品目标、边界与长期定位。
2. 目标用户、目标规模与关键体验。
3. 主要功能模块、核心业务规则与成功指标。
4. 产品质量要求与最终发布形态。
5. 本文档描述“产品目标能力与最终形态”，不作为开发路线、排期、当前代码盘点或阶段缺口管理文档。

## 1. 产品定义

### 1.1 产品定位

在线辩论 + AI 裁判平台，核心价值是提供“赛后官方结论”，而非赛中聊天型 AI。

### 1.2 核心价值主张

1. 对辩论结果给出结构化、可解释、可追溯的官方判决。
2. 在争议场景（平局）中提供可执行的社区决议机制（投票 + 二番战）。
3. 提供娱乐性和观赏性，同时保持规则一致与公平优先。

### 1.3 产品边界

1. AI 裁判仅做阵营（正/反）级别判定与评分，不做个人评分。
2. 支持 `pro/con/draw` 三种判决结果。
3. 用户不可见置信度等内部风控字段。
4. 暂不开放用户申诉入口。
5. 产品内容供给聚焦游戏（云顶之弈）领域，不以多领域并行为默认目标。
6. 账号体系支持多登录方式并存；手机号作为唯一主联系标识，邮箱为可选联系信息。

### 1.4 最终发布目标

产品最终目标为 iPhone 应用、Mac 桌面应用与 Web 版本三端正式上线，并形成可持续运营的在线辩论产品。

## 2. 目标用户与目标用户量

### 2.1 目标用户角色

1. 参赛用户：加入正反方、实时发言、参与平局投票。
2. 观战用户：围观辩论过程，查看 AI 判决与理由。
3. 运营用户：管理辩题与场次、维护内容供给、观察核心运营指标。

### 2.2 目标用户量

采用“在线并发等效”作为统一规模口径：

1. L1：500 在线（基础可用档）。
2. L2：2,000 在线（公测可用档）。
3. L3：5,000 在线（增长可用档）。
4. L4：10,000 在线（目标档）。

说明：以上为产品规模目标，用于描述最终产品需要承载的量级；10,000 在线需在真实发布环境中完成验证。

## 3. 产品目标与成功指标

### 3.1 北极星目标

让“辩论结束后 5 分钟内获得可读、可信、可执行的官方裁决”成为稳定用户体验。

发布目标补充：

1. iPhone 应用、Mac 桌面应用与 Web 版本需提供一致的核心产品能力。
2. iPhone 与 Mac 版本需满足 Apple 平台上线所需的产品、支付、合规与审核要求。
3. Web 版本需可对真实用户稳定开放访问与使用。

### 3.2 核心指标

1. AI 判决时效：`P95 <= 5 分钟`，目标值 `2 分钟`。
2. 判决可用率：可成功返回判决报告的任务占比 `>= 98%`。
3. 平局流程完成率：出现 draw 后进入“投票/二番战”可执行终局的比例 `>= 99%`。
4. 实时辩论体验：
   - 发言链路成功率 `>= 99.5%`
   - 端到端实时广播 `P95 < 1s`
5. 付费链路（置顶）：
   - 扣费 + 置顶成功率 `>= 99.9%`
   - `P95 < 500ms`

## 4. 核心业务流程

1. 用户可通过邮箱/手机号/微信等认证方式登录；完成登录后进入主界面，可查看私聊/群聊、交锋广场入口、搜索框、个人信息与充值入口。
2. 运营创建并排期辩题与场次（含待开启与进行中）。
3. 用户进入交锋广场，浏览进行中/待开启辩论；可加入待开启场次（选正方或反方），也可进入进行中场次观战。
4. 用户在 Debate Room 发言，可选择付费置顶消息。
5. 场次结束后用户发起 AI 判决任务。
6. 系统生成并展示裁判结果（胜负、评分、理由、背景与摘要）。
7. 若结果为 draw，系统自动进入平局投票流程。
8. 投票按门槛与多数决议；若进入二番战则自动生成下一场次。
9. 用户可继续加入二番战，形成可持续的对战链路。

## 5. 功能需求清单（产品口径）

### 5.0 重要

关于AI裁判服务的完整PRD，请去完整查看AI裁判完整PRD.md。


### 5.1 账号与基础访问

1. 平台应支持以下认证方式并长期共存：
   - 邮箱 + 密码登录（邮箱注册需绑定手机号并完成短信验证）。
   - 手机号 + 密码登录。
   - 手机号 + 短信验证码登录。
   - 微信登录（首次授权后需绑定手机号；密码可选，后续可补设密码）。
2. 用户可完成注册、登录、登录态恢复与退出登录。
3. 手机号是账号主联系标识：同一手机号仅允许绑定一个主账号。
4. 邮箱为可选字段；若填写邮箱，应满足格式正确且全局唯一。
5. 微信首次登录未完成手机号绑定前，不可进入辩论与判决核心页面。
6. 安全敏感动作（如后置设置密码）需短信验证码二次校验。
7. 用户仅可访问其已授权的个人与群组资源（平台单租户模型，无 workspace 概念）。
8. 未登录用户不可访问辩论与判决核心页面。
9. 登录后首页需提供四类核心入口：
   - 我的会话：展示当前用户全部私聊与群聊群组。
   - 交锋广场：进入辩论场次大厅。
   - 搜索框：按辩题关键词搜索可加入场次。
   - 个人信息与充值入口。

验收标准：

1. 四类认证方式均可完成登录并进入首页（邮箱密码、手机密码、手机验证码、微信）。
2. 微信首登用户在未绑定手机号时会被引导到绑定流程，完成后方可进入核心页面。
3. 同一手机号重复绑定时不得创建重复主账号，需返回明确提示并引导进入已绑定账号登录路径。
4. 设置/更新密码场景下，短信验证码缺失、错误或过期必须拒绝。
5. 登录后可在同一会话内连续访问 Lobby、Room、Judge Report。
6. 非授权用户访问受限资源时返回明确错误提示。
7. 登录后首页可直接触达“聊天、广场、搜索、个人中心/充值”四项核心入口。

### 5.2 辩题与场次（Lobby）

1. 交锋广场需区分展示：
   - 正在进行中的辩论（支持点击观战）。
   - 待开启辩论（支持选择阵营加入）。
2. 用户可按辩题/状态浏览场次。
3. 用户可加入正方或反方阵营。
4. 场次状态按生命周期推进：`scheduled/open/running/judging/closed`（含取消态管理）。

验收标准：

1. 仅在可加入状态允许加入阵营。
2. 场次人数、状态变化在列表和房间中保持一致。
3. 正在进行中的场次可直接进入观战态页面。
4. 待开启场次可完成“选择正方/反方并加入”。

### 5.3 辩题搜索

1. 广场提供搜索框，支持按辩题关键词检索场次。
2. 搜索结果应覆盖“待开启”和“进行中”两类场次，并清晰展示状态。
3. 搜索结果可直接执行“加入（待开启）/观战（进行中）”动作。

验收标准：

1. 输入关键词后可返回匹配辩题场次列表。
2. 搜索为空或无结果时提供明确反馈。

### 5.4 辩论房间（Room）

1. 支持实时发言流与历史消息分页。
2. 支持房间级实时状态更新与断线重连恢复。
3. 房间内展示置顶消息区与钱包余额。

验收标准：

1. 重连后消息与状态可恢复到最新一致状态。
2. 历史分页不丢消息、不重复消息。

### 5.5 付费置顶与充值

业务背景：

1. 辩论采用滚动式实时发言，普通消息可能被高频新消息快速淹没。
2. 用户可通过充值获得积分，再消费积分购买“发言置顶”，提升曝光和可见性。

3. 参赛用户可为自己发言购买置顶时长。
4. 置顶按时长扣费，余额不足需阻断并提示。
5. 同一幂等键重复提交不得重复扣费。
6. 置顶规则：时长范围 30s-600s，按 30s 计费单位结算。
7. 个人信息区必须提供充值入口，用户可查看余额并进入充值流程。

验收标准：

1. 置顶成功后消息进入置顶区并显示到期时间。
2. 重复请求不产生重复账单。
3. 用户可从个人信息入口完成“查看余额 -> 充值 -> 返回辩论消费”闭环。

### 5.6 AI 裁判任务触发

1. 用户可在场次进入 judging/closed 后触发 AI 判决。
2. 支持“允许重判”开关（受系统规则约束）。
3. 系统返回任务状态并支持刷新。

验收标准：

1. 非法状态下请求判决必须拒绝并提示原因。
2. 任务状态从请求到完成全程可查询。

### 5.7 AI 裁判报告展示

报告必须包含：

1. 胜负结果（pro/con/draw）。
2. 阵营总分。
3. 四维评分：逻辑、证据、反驳、表达。
4. 正反方核心观点摘要。
5. 裁决理由（用户可读版本）。
6. 阶段摘要（支持分页窗口）。
7. 背景/RAG来源信息（用于增强可解释性）。

验收标准：

1. 报告字段完整、语义一致、支持分页浏览。
2. 用户端不暴露置信度等内部字段。

### 5.8 平局投票与二番战

平局规则：

1. AI 结果为 draw 时自动创建投票。
2. 投票窗口为 5 分钟。
3. 参与门槛为本场参与用户的 70%。
4. 达门槛后按多数票决议：
   - 同意平局：结束，不开二番战。
   - 不同意平局：开启二番战。
5. 超时未达门槛：默认开启二番战。

二番战规则：

1. 决议为开启二番战时系统自动创建下一场次。
2. 二番战应可在产品内直接被发现和加入。

验收标准：

1. 投票状态可查询、可提交、可终局。
2. 开启二番战分支必须产生可用场次 ID 并可进入。

### 5.9 运营后台（Ops）

1. 运营可创建/编辑辩题。
2. 运营可创建/编辑场次，支持状态管理。
3. 运营权限与普通用户权限严格隔离。
4. 运营可为特定辩题配置定时开启计划（仅在指定时间窗口开放）。
5. 定时开启场次默认具备正反方阵营设置与可加入名额规则。

验收标准：

1. 非运营角色执行运营动作必须被拒绝。
2. 运营改动应实时反映到 Lobby 与 Room。
3. 定时场次应在设定时间前显示为“待开启”，到点后按规则进入可参与状态。

### 5.10 裁判观测与运营复盘

1. 提供判决刷新统计与汇总视图（成功率、失败、重试、合并事件等）。
2. 支持运营按时间窗口查看裁判表现。
3. 支持后续扩展到成本与收益复盘。

验收标准：

1. 汇总数据可刷新、可按会话维度查看。
2. 数据缺失或异常时有明确提示。

## 6. 公平性与内容质量需求

1. 判决输入应盲化用户身份相关标签（避免身份偏置）。
2. 每次判决必须记录规则版本（rubric_version）。
3. 判决应保留证据可追溯能力（至少支持消息级引用链）。
4. 启用双次评估差异控制；胜方不一致触发重判/平局保护机制。
5. 平台可保留内部人工复核能力，但不对用户开放申诉入口。

---

## 7. 非功能需求（产品视角）

1. 时效：判决为赛后异步，不阻塞用户主操作。
2. 可理解性：判决文案需兼具理性与可读性，系统默认“理性优先”。
3. 一致性：同一场次所有用户看到的最终结果必须一致。
4. 可恢复性：实时链路异常后，用户可通过重连/刷新恢复到正确状态。
5. 认证安全性：登录与验证码链路需具备限流、验证码有效期与敏感操作二次校验能力。
6. 隐私最小化：认证与分析链路应尽量避免明文账号标识扩散，满足最小必要使用原则。
