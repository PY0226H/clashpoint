# AI裁判完整PRD（聚焦 `ai_judge_service`）

更新时间：2026-04-11  
文档状态：持续维护（用于统一 AI 裁判服务目标与最终形态）  
适用范围：`ai_judge_service` 服务级产品目标与能力定义

---

## 0. 文档目的

本文档用于统一 AI 与团队对 `ai_judge_service` 最终形态的理解，明确：

1. 该服务在整个平台中的职责边界与长期定位。
2. 该服务必须具备的判决链路、输入输出结构与结果要求。
3. 该服务必须满足的公平性、可解释性、稳定性与可运维能力。
4. 该服务支撑 iPhone 应用、Mac 桌面应用与 Web 版本一致裁判体验所需的基础能力。

说明：

1. 本文档聚焦 `ai_judge_service`，不展开 Lobby、Room、账号、支付、投票 UI 等前后端产品细节。
2. 本文档描述的是服务目标能力与最终形态，不承载开发排期、里程碑拆解、当前代码盘点、阶段缺口或发布前 checklist 管理。

## 1. 服务定位

### 1.1 服务定义

`ai_judge_service` 是在线辩论平台中的 AI 裁判执行服务，负责接收主业务系统派发的裁判任务，执行多阶段评审，结合 RAG 背景信息生成结构化判决结果，并以内部回调方式回传结果或失败状态。

### 1.2 核心定位

该服务的定位是：

1. 赛中阶段性判决 + 赛后官方裁决系统。
2. 阵营级别的结构化评审引擎。
3. 可解释、可追溯、可审计的内部 AI 能力服务。

该服务不是：

1. 赛中聊天机器人。
2. 面向用户的开放式问答系统。
3. 用户个人评分系统。
4. 用户申诉与人工仲裁系统。

### 1.3 核心价值

`ai_judge_service` 的核心价值在于：

1. 为一场辩论稳定产出唯一、一致、可解释的官方结论。
2. 用结构化评分与证据链提升用户对结果的理解与接受度。
3. 为上游系统提供 draw、重判保护、复盘审计、告警追踪等能力支撑。

---

## 2. 服务边界

### 2.1 本服务负责

1. 接收内部派发的裁判任务。
2. 执行多 Agent 判决链路。
3. 对辩论消息进行阶段总结、终局评分与结果生成。
4. 结合话题背景与知识检索增强判决依据。
5. 输出结构化裁判报告。
6. 处理 OpenAI / RAG / 运行时降级与失败回调。
7. 提供 trace、replay、audit alert、RAG diagnostics 等运维接口。
8. 记录内部审计字段、错误码、证据引用与运行时退化信息。

### 2.2 本服务不负责

1. 用户登录、认证、授权。
2. 房间实时消息广播。
3. 辩题创建、场次排期与 Lobby 展示。
4. 平局投票的用户交互与票务状态管理。
5. 二番战场次的最终创建与编排。
6. 钱包、充值、置顶扣费。

### 2.3 与主系统关系

1. `chat_server` / 主业务系统负责维护辩论生命周期、组织输入数据、触发裁判任务、保存最终结果并对外提供产品 API。
2. `ai_judge_service` 只面向内部调用面，输出内部结构化结果，不直接对终端用户提供开放接口。

## 3. 产品目标

### 3.1 北极星目标

让“辩论结束后 5 分钟内获得可读、可信、可执行的官方裁决”成为稳定能力，并将该能力沉淀为平台级基础服务。

### 3.2 服务级目标

1. 对每个合法裁判任务给出 `pro/con/draw` 三选一结果。
2. 辩论进行中每 `100` 条消息自动产出阶段性总结、阶段性评分与阶段性证据包。
3. 给出阵营总分与四维评分。
4. 生成面向前端展示的辩论总结、双方分析与判决理由。
5. 在不确定场景下触发平局保护与一致性控制。
6. 在运行时异常场景下保证“可失败、可回调、可追踪、可复盘”。
7. 为 iPhone 应用、Mac 桌面应用与 Web 版本提供一致、可复用的裁判基础能力。

### 3.3 成功指标

1. 判决时效：`P95 <= 5 分钟`，目标值 `2 分钟`。
2. 判决可用率：成功返回判决报告的任务占比 `>= 98%`。
3. 判决结果一致性：同一任务所有消费者看到的最终结果必须一致。
4. 回调成功率：报告回调与失败回调链路总体成功率应达到生产可用要求。
5. 复盘可用性：关键任务具备 trace、重放与审计字段查询能力。

## 4. 服务输入输出定义

### 4.1 输入来源

服务输入由主业务系统组织后内部发送，内容包括：

1. 任务信息：`job_id`、`scope_id`、会话 ID、请求人、请求时间等。
2. 场次信息：状态、开始结束时间等。
3. 辩题信息：标题、描述、分类、正反方立场、话题背景种子。
4. 发言消息列表：按时间顺序排列的消息窗口。
5. 背景知识信息：与当前辩题对应的人工编写、人工整理的 Markdown 背景知识文档集合，或其文档 ID / 版本信息。
6. 判决规则版本：`rubric_version`。
7. 任务链路信息：`trace_id`、`idempotency_key`、`judge_policy_version`、`topic_domain`、`retrieval_profile`。

### 4.2 输入要求

1. 输入必须是内部可信调用，不对外开放。
2. 输入消息必须已做身份盲化，不得携带用户身份、粉丝、充值、余额、VIP 等偏置字段。
3. 消息列表为空时，服务不得继续评审，必须回调失败。
4. 非法状态下的评审请求应由上游优先拦截；服务侧仍需具备失败兜底能力。

### 4.3 输出结果

服务应输出结构化裁判结果，至少包含：

1. 胜负结果：`pro / con / draw`
2. 阵营总分：`pro_score / con_score`
3. 四维评分：逻辑、证据、反驳、表达
4. 面向用户的整场辩论总结
5. 面向用户的双方表现分析
6. 面向用户的判决原因说明
7. 阶段摘要
8. 证据引用
9. RAG 来源摘要
10. 审计与链路追踪字段
11. 阶段性产物（每 100 消息窗口）：
    - 正反方原文保真总结
    - 正反方 RAG 检索结果
    - 阶段性判决 Agent1/2/3 评分与依据
    - 阶段性加权总分

### 4.4 输出分层

对外分为两层语义：

1. 用户展示层
   - 可读的胜负、整场辩论总结、双方表现分析、判决原因、阶段信息、背景来源信息
2. 内部审计层
   - `rubric_version`
   - `judge_policy_version`
   - `promptHash`
   - `retrievalSnapshot`
   - `errorCodes`
   - `degradationLevel`
   - `winnerFirst / winnerSecond`
   - `verdictEvidenceRefs`
   - `auditAlerts`

约束：

1. 用户不可见 `confidence` 等内部风控字段。
2. 内部输出必须支撑复盘与质量迭代。

### 4.5 阶段性产物持久化要求

对每个阶段窗口（`phase_no`）必须持久化以下字段：

1. `phase_no`、`message_start_id`、`message_end_id`、`message_count`
2. `pro_message_ids`、`con_message_ids`
3. `pro_summary_grounded`、`con_summary_grounded`（需包含消息 ID 引用）
4. `pro_retrieval_bundle`、`con_retrieval_bundle`
5. `agent1_score`、`agent2_score`、`agent3_weighted_score`
6. `agent1_rationale`、`agent2_rationale`
7. `prompt_hashes`（各 Agent 的 prompt/配置快照）
8. `token_usage`、`latency_ms`、`error_codes`

约束：

1. 阶段总结必须可回放，可追溯到原始消息 ID。
2. 阶段总结不得引入窗口外信息或未出现观点。
3. 阶段产物必须可被终局 Agent 直接消费。

## 5. 核心业务流程

### 5.1 标准流程

1. 主系统在辩论进行中每累计 `100` 条新消息时，自动触发阶段性裁判任务（非用户手动触发）。
2. `ai_judge_service` 校验内部调用、幂等键与输入盲化要求。
3. 阶段性总结 Agent 产出正反方保真总结，并基于与该辩题对应的人工维护 Markdown 背景知识文档分别做 RAG 检索，保存双边检索包。
4. 阶段性判决 Agent1/2/3 完成评分、依据与加权融合，写入阶段产物库。
5. 主系统在场次进入可判决状态后自动触发终局任务（可复用全部阶段产物）。
6. 终局 Agent 汇总所有阶段检索包和阶段评判结果，生成最终胜负、整场辩论总结、双方分析与判决原因。
7. 展示 Agent 将终局结果重写为适合前端展示的用户可读文案（不改变核心事实与结论）。
8. 服务生成完整结构化裁判报告并通过内部回调回传主系统。
9. 服务记录 trace、审计快照、运行时错误码、告警信息与阶段化证据链。

### 5.2 异常流程

1. 若输入为空或非法，服务标记失败并回调失败原因。
2. 若模型调用失败但满足降级条件，服务可按策略降级或回退。
3. 若关键链路失败且不可恢复，服务必须返回 failed callback。
4. 若发现合规或一致性问题，服务可阻断提交并进入告警链路。

### 5.3 平局相关流程

1. 当模型直接判断为 `draw` 时，输出平局结果。
2. 当双次评估胜方不一致时，触发平局保护，输出 `draw`。
3. 服务本身不执行投票与二番战创建，但必须提供足够结果让上游继续后续流程。

### 5.4 自动触发与并行流程

1. 阶段任务触发器必须支持 `message_count % 100 == 0` 的自动调度。
2. 阶段性判决 Agent2 的“正攻反守”与“反攻正守”两条链路必须并行执行。
3. 每个阶段任务必须具备幂等能力，重复触发不得产生重复阶段结果。
4. 当某阶段失败时，不影响已成功阶段；系统应支持阶段补偿重跑。

## 6. 多 Agent 判决需求

### 6.1 总体要求

服务应采用“赛中阶段链路 + 赛后终局链路”的多 Agent 结构，以平衡时延、成本、稳定性与可解释性。

### 6.2 辩论阶段性总结 Agent（每 100 消息）

阶段性总结 Agent 负责：

1. 对当前 100 条消息按 `side` 严格分流，形成正方/反方独立语料。
2. 产出 `pro_summary_grounded` 与 `con_summary_grounded`，要求覆盖本方所有观点、反驳与关键论据。
3. 总结结果必须附带消息 ID 引用，支持覆盖率校验与证据回溯。

约束：

1. 当前窗口大小按已确认方案固定为 `100` 条消息。
2. 总结绝对不能编造额外内容，禁止引入窗口外信息。
3. 阶段输出可作为内部 RAG 与后续 Agent 输入，不等同于展示文案。

### 6.3 阶段性检索子流程（Summary Agent Step2）

基于 `pro_summary_grounded` 与 `con_summary_grounded` 分别执行检索：

1. 每侧独立生成检索查询，执行向量检索或混合检索。
2. 检索结果按侧保存为 `pro_retrieval_bundle` / `con_retrieval_bundle`。
3. 检索结果必须可复用给阶段性判决 Agent 与终局 Agent。

策略要求：

1. 支持切片、向量召回、可选重排和检索后去重。
2. 支持检索前查询改写（关键词扩展、立场词约束）。
3. 支持检索后质量过滤（限定人工维护知识文档范围、重复片段折叠、低相关性截断）。

### 6.4 辩论阶段性判决 Agent 1

Agent 1 输入：

1. 当前阶段 100 条原始消息。
2. `pro_retrieval_bundle` 与 `con_retrieval_bundle` 的并集。

Agent 1 输出：

1. 阶段性胜负倾向与 `pro/con` 分数。
2. 多维评分与裁决依据。
3. 阶段性总结结论（供终局 Agent 消费）。

### 6.5 辩论阶段性判决 Agent 2（双向并行）

Agent 2 采用“理想反驳命中度”评测，且正反互换并行执行：

1. 正向链路：
   - 输入：`pro_retrieval_bundle` + 正方原始消息。
   - 生成：模型输出“理想反方反驳答案”。
   - 评估：用该理想答案对照反方真实总结，得到反方命中度评分。
2. 反向链路：
   - 输入：`con_retrieval_bundle` + 反方原始消息。
   - 生成：模型输出“理想正方反驳答案”。
   - 评估：用该理想答案对照正方真实总结，得到正方命中度评分。

约束：

1. 两条链路必须并行，且可独立重试。
2. “理想答案”只用于评分标尺，不得直接写入用户展示内容。

### 6.6 辩论阶段性判决 Agent 3（加权融合）

Agent 3 负责融合 Agent1 与 Agent2 的分数：

1. `phase_pro_score = w1 * agent1_pro + w2 * agent2_pro`
2. `phase_con_score = w1 * agent1_con + w2 * agent2_con`
3. 约束 `w2 > w1`（Agent2 权重更高），默认建议 `w1=0.35, w2=0.65`。

### 6.7 终局判决 Agent

终局 Agent 输入：

1. 全部阶段的 `pro/con retrieval bundles` 集合。
2. 全部阶段的 Agent1/2/3 产物。
3. 终局必要元信息（rubric、judge_policy、topic_domain）。

终局 Agent 输出：

1. 最终胜负 `pro/con/draw`。
2. 阵营总分与四维评分。
3. 面向用户展示的整场辩论总结。
4. 面向用户展示的双方表现分析。
5. 面向用户展示的判决原因说明。
6. 证据引用与阶段回溯摘要。

### 6.8 展示 Agent

展示 Agent 负责：

1. 将内部判决结果重写为用户可读版本。
2. 将“整场辩论总结 / 双方表现分析 / 判决原因”组织为前端可直接展示的稳定字段。
3. 保持“理性优先，同时具备观赏性”的文风目标。
4. 不改变最终胜负和核心事实。

约束：

1. 展示文风必须支持系统级开关。
2. 系统默认风格为“理性优先”。
3. 前端展示字段必须结构稳定，不能只返回单一大段混合文案。
4. 展示层不得编造不存在的证据或结论。

## 7. 判决规则与评分模型

### 7.1 胜负集合

服务必须支持以下三种判决：

1. `pro`
2. `con`
3. `draw`

### 7.2 评分维度

阵营评分维度固定为四项：

1. 逻辑
2. 证据
3. 反驳
4. 表达

### 7.3 评分与结果约束

1. 评分对象仅到阵营层，不评分到个人。
2. 不提供个人排行榜。
3. 不对外展示内部置信度。

### 7.4 规则版本化

每次判决必须记录：

1. `rubric_version`
2. `judge_policy_version`

目的：

1. 支持复盘与回归比较。
2. 支持后续模型与规则版本演进。

---

## 8. 一致性控制与平局保护

### 8.1 双次评估

服务必须支持对终局裁判进行双次独立评估，以降低单次输出波动导致的不公平结论。

### 8.2 已确认触发规则

已确定的一致性保护规则为：

1. 当双次评估胜方不一致时，触发重判保护。
2. 触发后，最终输出应转为 `draw` 或进入平局保护语义。

### 8.3 目标效果

1. 避免模型在高争议场景下输出过于激进、不可复核的单边结论。
2. 为上游投票与二番战机制提供稳定的触发依据。

### 8.4 可扩展策略

后续可扩展：

1. 低分差保护
2. 更复杂的一致性阈值策略
3. 不同领域的差异化平局保护参数

基础要求是“胜方不一致触发保护”。

## 9. RAG 与知识需求

### 9.1 目标

服务应通过 RAG 使用与辩题对应的人工维护 Markdown 背景知识文档，提供更强的背景理解和可解释性，降低纯对话上下文导致的事实漂移。

### 9.2 内容领域聚焦

服务知识与判决规则默认聚焦单领域：

1. 游戏
2. 默认话题域：云顶之弈

### 9.3 知识来源定义

默认知识来源为与辩题绑定的人工编写、人工整理的 Markdown 文档，要求：

1. 文档内容由人工维护，不依赖线上网页抓取作为默认事实来源。
2. 每份文档应可标识所属辩题域、主题、版本与更新时间。
3. 文档内容应适合按标题、段落或语义片段切分后进入检索索引。

### 9.4 检索能力要求

服务应支持：

1. 话题背景种子上下文注入
2. Markdown 文档解析与分段检索
3. 文件型知识检索
4. 向量型知识检索
5. 后续扩展为混合检索与重排

### 9.5 知识管线要求

知识准备链路应支持：

1. 人工维护 Markdown 文档导入
2. 文档清洗与结构解析（标题、段落、元数据）
3. 切片
4. 写入向量数据库
5. 文档版本与更新时间管理

### 9.6 更新策略

默认更新策略：

1. 以人工编辑和提交新的 Markdown 文档为准。
2. 文档新增、修改或替换后，应支持增量重建索引。
3. 索引侧应保留文档版本信息，支持回溯当次判决使用的知识版本。

### 9.7 来源展示要求

用户侧报告应展示背景知识来源摘要，且能让用户理解“本次判决参考了哪些背景知识文档或文档片段”，但不应暴露内部无关实现细节。

内部侧应保留：

1. `retrievalSnapshot`
2. 召回来源
3. 检索配置与诊断信息

### 9.8 阶段性双边检索策略

对阶段性总结的正反方检索，需满足以下要求：

1. 分边检索：正方总结仅生成正方查询，反方总结仅生成反方查询。
2. 分层切片：按“观点/论据/反驳”语义切片，保留消息 ID 反向索引。
3. 混合召回：优先向量召回，允许叠加词法召回并做 RRF 融合。
4. 重排：可选轻量重排，目标是提升前 `K` 条证据精度。
5. 检索后处理：去重、仅保留当前辩题允许使用的 Markdown 知识文档片段、冲突来源标注。
6. 输出持久化：每侧至少保存 `query`、`top_k`、`chunk_ids`、`source_refs`、`scores`。

## 10. 公平性与合规需求

### 10.1 输入盲化

服务必须拒绝未盲化输入，包括但不限于：

1. 用户 ID
2. 充值相关信息
3. 粉丝标签
4. 余额信息
5. VIP 标识

### 10.2 偏置控制

服务在评审阶段必须以“阵营内容质量”为核心，而非用户身份信息。

### 10.3 证据链要求

每次判决至少在内部层面保留消息级证据引用能力，支持：

1. 引用消息 ID
2. 引用原因
3. 胜方支持点与对方关键点的说明

### 10.4 合规阻断

当服务检测到关键展示字段缺失、结果不一致或其他合规异常时，应具备：

1. 告警
2. 阻断提交
3. 失败回调
4. 运维复盘入口

---

## 11. 运行时策略与降级要求

### 11.1 Provider 策略

服务应支持：

1. OpenAI 主路径
2. Mock / 测试路径
3. 可配置的降级与回退策略

### 11.2 降级原则

当外部依赖异常时，服务应尽量遵循以下原则：

1. 优先保证任务可追踪。
2. 其次保证结果可回调。
3. 无法产出可靠结果时，宁可明确失败，也不要伪造高置信结论。

### 11.3 可接受降级场景

1. 阶段评审失败时允许规则摘要兜底。
2. 聚合失败时允许安全降级。
3. 展示重写失败时允许回落到原始摘要。
4. RAG 不可用时允许 fail-open 继续判决，但必须记录退化与错误码。

### 11.4 不可接受行为

1. 无痕失败。
2. 回调成功但结果字段不完整且无告警。
3. 对用户展示内部敏感字段。
4. 在生产环境启用不受控的 mock 路径。

---

## 12. 可观测性与运维能力

### 12.1 Trace 能力

服务必须支持按任务查询：

1. 请求快照
2. 当前状态
3. 回调状态
4. 报告摘要
5. 回放历史

### 12.2 Replay 能力

服务必须支持无副作用重放，以便：

1. 排查模型输出异常
2. 做回归复盘
3. 对比版本差异

### 12.3 审计告警

服务必须支持：

1. 告警产生
2. 告警确认
3. 告警恢复
4. 告警 outbox 投递状态记录

### 12.4 RAG Diagnostics

服务必须支持按任务查看：

1. 检索后端
2. 检索来源
3. 检索 profile
4. 检索退化原因
5. topic memory 复用情况

### 12.5 运行时错误码

服务至少应能规范表达以下类型错误：

1. `judge_timeout`
2. `rag_unavailable`
3. `model_overload`
4. `consistency_conflict`

---

## 13. Topic Memory 需求

### 13.1 目标

服务可维护“同辩题域、同规则版本”的高质量历史判决记忆，用于增强后续相似任务的背景理解与一致性。

### 13.2 准入要求

仅当历史判决满足一定质量门槛时，才允许写入 topic memory，例如：

1. 具备足够证据引用
2. 具备足够理由长度
3. 综合质量分达到阈值

### 13.3 使用要求

1. topic memory 仅作为辅助上下文，不可直接替代当前任务判断。
2. topic memory 的复用必须被记录到审计与诊断字段中。
3. topic memory 不得引入用户身份等隐私信息。

---

## 14. 内部接口需求

### 14.1 调用方式

主系统与 `ai_judge_service` 的同步调用方式默认采用：

1. `HTTP JSON`

接口能力应允许后续扩展为：

1. 在线同步面可升级为 `gRPC`
2. 异步任务与状态流可接入 `Kafka`
3. 长期目标是 `gRPC + Kafka` 共存

### 14.2 安全要求

1. 所有内部接口必须校验内部密钥。
2. 非法调用必须拒绝。
3. 服务不得暴露面向公众用户的开放裁判入口。

### 14.3 幂等要求

1. 服务应支持 `idempotency_key`。
2. 同一幂等键重复请求不得重复产生不一致结果。
3. 已成功任务可返回幂等重放结果。
4. 已失败任务允许清理后重试。

---

## 15. 非功能需求

### 15.1 时效

1. 裁判为赛后异步执行，不阻塞用户核心实时操作。
2. 需满足 `P95 <= 5 分钟` 的产品目标。

### 15.2 稳定性

1. 服务应支持运行时重试。
2. 外部依赖异常应有明确退化策略。
3. 失败必须可回调、可审计。

### 15.3 一致性

1. 同一任务的最终结果必须唯一。
2. 上游、运维接口与回放报告中的关键字段必须一致。

### 15.4 可理解性

1. 输出必须兼顾理性与可读性。
2. 展示文本需支持用户快速理解“为什么赢、为什么平”。

### 15.5 隐私最小化

1. 输入与日志中应最小化暴露用户身份信息。
2. 只记录完成审计所需的最小必要字段。

---

## 16. 服务结果要求

### 16.1 功能结果要求

1. 合法输入任务可生成完整报告。
2. 报告中包含胜负、总分、四维评分、整场辩论总结、双方表现分析、判决原因、阶段结果和背景来源摘要。
3. `draw` 场景能输出明确平局语义。
4. 双次评估胜方不一致时会触发平局保护。
5. 输入未盲化时会被拒绝。
6. 同一幂等键重复提交不会生成重复结果。
7. 失败任务会走 failed callback。
8. 每 100 条消息会自动触发阶段任务，并成功落盘阶段产物。
9. 阶段性总结可回溯到消息 ID，且覆盖率达到配置阈值。
10. Agent2 双向链路可并行执行，且任一侧失败可独立补偿。
11. Agent3 输出的阶段总分可被终局 Agent 正确复用与复算。

### 16.2 RAG 结果要求

1. 与当前辩题绑定的人工维护 Markdown 知识文档能被正确检索使用。
2. 非当前辩题允许使用的知识文档不会进入最终结果来源集。
3. RAG 不可用时服务仍可按策略退化，并记录错误码与退化级别。

### 16.3 运维结果要求

1. 可通过 trace 接口查看任务全链路状态。
2. 可通过 replay 接口进行无副作用重放。
3. 告警可完成 raised -> acked -> resolved 状态流转。
4. outbox 投递状态可查询与回写。

### 16.4 非功能结果要求

1. 判决链路满足目标 SLA。
2. 成功率满足 `>= 98%`。
3. 结果对同一任务保持一致。
4. 用户输出不暴露内部置信度等敏感字段。
