# AI裁判完整PRD（聚焦 `ai_judge_service`）

更新时间：2026-04-13  
文档状态：持续维护（用于统一 AI 裁判服务目标与最终形态）  
适用范围：`ai_judge_service` 服务级产品目标与能力定义

---

## 0. 文档目的

本文档用于统一 AI 与团队对 `ai_judge_service` 最终形态的理解，明确：

1. 该服务在整个平台中的职责边界与长期定位。
2. 该服务必须提供的裁判能力、结果形态与可信能力。
3. 该服务必须满足的公平性、可解释性、稳定性、可追溯性与可运维要求。
4. 该服务支撑 iPhone 应用、Mac 桌面应用与 Web 版本一致裁判体验所需的基础能力。

说明：

1. 本文档聚焦 `ai_judge_service`，不展开 Lobby、Room、账号、支付、投票 UI 等平台细节。
2. 本文档描述的是服务目标能力与最终形态，不承载开发排期、阶段缺口、当前代码盘点或技术实现细节。
3. 本文档的重点是“服务应该成为什么”，而不是“当前代码已经做到了什么”。

## 1. 服务定位

### 1.1 服务定义

`ai_judge_service` 是在线辩论平台中的内部 AI 裁判服务，负责接收主业务系统派发的案件对象，对多人辩论进行结构化审理，产出官方裁决、丰富裁决内容、内部审计产物与可信承诺信息，并回传主业务系统。

### 1.2 核心定位

该服务的定位是：

1. 平台的赛后官方裁决中枢，而不是赛中聊天型 AI。
2. 阵营级别的结构化评审与裁决服务，而不是个人评分器。
3. 兼顾判决、解释、公平门禁与内部复核的制度化 AI 服务。
4. 可持续演进为“可验证裁决基础设施”的平台底层能力。

该服务不是：

1. 面向终端用户的开放问答系统。
2. 房间实时互动或主持机器人。
3. 钱包、区块链或 NFT 的主入口。
4. 任意场景都开放自由申诉的公共仲裁平台。

### 1.3 核心价值

`ai_judge_service` 的核心价值在于：

1. 为一场辩论稳定产出唯一、一致、可解释的官方结论。
2. 把“只给一个胜负”升级成“给出一份可读、丰富、可追溯的裁决书”。
3. 在高争议、不确定或偏差风险场景下，提供 draw、内部复核与受约束 challenge 能力。
4. 为上游系统提供公平控制、结果承诺、复盘审计、质量治理与长期演进基础。

## 2. 服务边界

### 2.1 本服务负责

1. 接收内部派发的裁判案件。
2. 对案件输入做规范化、盲化、规则检查与准入判断。
3. 将辩论内容组织为可裁判的案件对象与推理素材。
4. 提取双方核心争点、主张关系、回应关系与关键转折点。
5. 结合批准来源的背景知识与证据材料支撑裁判过程。
6. 组织多代理裁判、稳定性检查、公平门禁与终局仲裁。
7. 输出用户可读的裁决报告、上游可消费的结构化结果、内部可复盘的审计产物。
8. 提供 trace、replay、review、告警、质量分析与可信承诺能力。

### 2.2 本服务不负责

1. 用户登录、认证、权限体系与钱包能力。
2. 房间实时消息广播与断线重连。
3. 辩题创建、场次排期、Lobby 展示与用户加入流程。
4. 投票与二番战的前端交互和最终业务编排。
5. 平台对外的公开区块链交互入口。

### 2.3 与主系统关系

1. 主业务系统负责维护辩论生命周期、组织案件输入、保存最终结果并对外提供产品 API。
2. `ai_judge_service` 只面向内部调用面，输出内部结构化结果，不直接对普通终端用户开放。
3. 主业务系统负责将 AI 裁判结果展示给用户，并承接 draw 投票、二番战、受约束 challenge 等产品交互。

## 3. 产品目标

### 3.1 北极星目标

让“辩论结束后 5 分钟内获得可读、可信、可追溯、可执行的官方裁决，或明确进入受保护复核状态”成为稳定能力，并将该能力沉淀为平台级制度化服务。

### 3.2 服务级目标

1. 对每个合法案件给出阵营级别的官方结论：`pro / con / draw`。
2. 对高争议或高不确定案件支持进入 `review required` 保护流程，而不是强行给出结论。
3. 生成面向用户展示的丰富裁决内容，而不是只输出胜负和分数。
4. 在公平性风险场景下执行稳定性检查、内部复核与结果保护。
5. 在运行时异常场景下保证“可失败、可追踪、可回放、可解释”。
6. 支撑 iPhone 应用、Mac 桌面应用与 Web 版本的一致裁判体验。
7. 为未来“可验证裁决、组织规则扩展与协议化治理”预留可信基础。

### 3.3 成功指标

1. 判决时效：正式公开结果的 `P95 <= 5 分钟`，目标值 `2 分钟`。
2. 判决可用率：成功进入终局结果或明确进入复核状态的案件占比 `>= 99%`。
3. 结果一致性：同一案件所有消费者看到的最终公开结果必须一致。
4. 报告完整性：公开裁决报告必须稳定包含规定字段与核心解释内容。
5. 可追溯性：关键案件必须可查询到案件快照、裁决依据、版本信息与复盘链路。
6. 公平治理可用性：高风险案件必须可触发告警、复核或保护性结果，而不是静默放行。

## 4. 核心对象定义

### 4.1 Case Dossier

服务应将每次裁判任务组织为统一的 `Case Dossier`，至少包含：

1. 场次与会话标识。
2. 辩题、立场、上下文与规则版本。
3. 结构化 transcript 快照。
4. 输入盲化与准入检查结果。
5. 裁判策略与可见性约束。

### 4.2 Reason Graph

服务应从辩论内容中形成可裁判的推理图谱，至少表达：

1. 双方核心主张。
2. 支撑关系与反驳关系。
3. 已回应与未回应争点。
4. 关键转折点与关键发言。

### 4.3 Evidence Ledger

服务应形成可追溯的证据账本，至少包含：

1. 消息级引用链。
2. 背景知识与外部来源引用。
3. 证据冲突与可靠性标记。
4. 关键裁决证据的出处与用途。

### 4.4 Verdict Ledger

服务应锁定最终裁决事实，至少包含：

1. 最终结果。
2. 阵营总分与维度分。
3. 关键裁决原因与决定性因素。
4. 被采纳与未被采纳的核心判断依据。
5. draw 或复核要求的触发理由。

### 4.5 Fairness Report

服务应为每个案件形成公平性审计产物，至少包含：

1. 稳定性检查结果。
2. 裁判团分歧度。
3. 偏差、身份泄漏或证据不足告警。
4. 是否允许自动终局。
5. 是否需要内部复核。

### 4.6 Opinion Pack

服务应形成统一的结果输出包，至少包括：

1. 面向用户的裁决报告。
2. 面向业务系统的结构化裁判结果。
3. 面向运营与复盘的审计摘要。
4. 面向可信层的结果承诺信息。

## 5. 核心业务流程

### 5.1 标准裁决流

1. 主业务系统在合法状态下创建裁判案件并派发至 `ai_judge_service`。
2. 服务完成输入规范化、盲化、规则检查与案件建档。
3. 服务形成结构化辩论记录、争点图谱与证据账本。
4. 多个裁判代理从逻辑、证据、反驳等不同视角进行独立判断。
5. 公平门禁层检查结果稳定性、分歧度、证据充分性与偏差风险。
6. 仲裁层锁定最终裁决事实，或决定进入 draw / 复核保护流程。
7. 裁决书层生成用户可读的丰富结果内容。
8. 服务输出结构化结果、审计信息与可信承诺产物，并回传主业务系统。

### 5.2 保护性裁决流

在以下场景，服务不应强行给出普通胜负结论：

1. 关键证据不足。
2. 裁判团分歧过高。
3. 稳定性检查不通过。
4. 输入存在身份泄漏或规则污染风险。
5. 合规或策略门禁命中阻断条件。

服务应支持的保护性结果包括：

1. `draw`
2. `review required`
3. `blocked failed`

### 5.3 复核与 Challenge 流

服务应支持受约束的复核能力，满足以下要求：

1. 平台可按赛事、价值等级或规则版本决定是否开放 challenge。
2. 复核必须绑定原案件快照、原裁决版本与补充材料。
3. 复核结论必须区分“维持原判”“改判”“仍需保留平局/待复核”。
4. 复核过程应写入统一案件历史，便于后续审计与运营治理。

### 5.4 可信承诺流

服务应为关键案件生成可信承诺产物，用于证明：

1. 本次裁决绑定了哪份案件材料。
2. 本次裁决绑定了哪版规则与哪版 Judge Kernel。
3. 本次结果没有被赛后静默修改。
4. 复核与结果变更有清晰的状态历史。

## 6. Agent 服务能力需求

### 6.1 Clerk Agent

服务应具备收案与净化能力，完成：

1. 内部任务准入判断。
2. 输入字段与内容盲化。
3. 资格与规则约束检查。
4. 统一案件对象生成。

### 6.2 Recorder Agent

服务应具备庭审记录能力，完成：

1. 重建辩论时间线。
2. 组织结构化 transcript。
3. 提取阶段视图、回合关系与引用关系。
4. 保证后续裁判建立在完整、可回溯的记录之上。

### 6.3 Reason Graph Agent

服务应具备争点与主张图谱构建能力，完成：

1. 提取核心 claim。
2. 识别支撑与反驳关系。
3. 标记争点命中与遗漏。
4. 形成可用于裁判与解释的推理素材。

### 6.4 Evidence Agent

服务应具备证据与背景核验能力，完成：

1. 判断哪些争点需要背景知识支撑。
2. 从批准来源中获取证据材料。
3. 标记来源、冲突与可靠性。
4. 将证据写入证据账本，而不是直接代替裁判结论。

### 6.5 Judge Panel

服务应具备多视角裁判团能力，至少覆盖：

1. 逻辑判断。
2. 证据判断。
3. 反驳质量判断。
4. 表达与可理解性判断。

要求：

1. 裁判团不是单一自由生成器，而是受约束的结构化判断系统。
2. 裁判团应能输出清晰的采纳理由与拒绝理由。

### 6.6 Fairness Sentinel

服务应具备公平哨兵能力，完成：

1. 稳定性与一致性检查。
2. 偏差与身份污染检测。
3. 裁判团分歧度评估。
4. 证据充分性判断。
5. 自动终局资格判定。

### 6.7 Chief Arbiter

服务应具备首席仲裁能力，完成：

1. 汇总裁判团结论。
2. 综合公平门禁结果。
3. 锁定最终裁决事实。
4. 在必要时要求 draw 或 review。

### 6.8 Opinion Writer

服务应具备裁决书生成能力，完成：

1. 基于最终裁决事实生成用户可读报告。
2. 保证解释不脱离证据账本与裁决账本。
3. 将丰富内容稳定映射为适合前端展示的结构。

## 7. Judge Kernel 与裁决规则

### 7.1 结果集合

正式公开结果集合为：

1. `pro`
2. `con`
3. `draw`

案件处理状态应允许存在：

1. `review required`
2. `blocked failed`

### 7.2 评分维度

公开评分应至少包含：

1. 逻辑
2. 证据
3. 反驳
4. 清晰度

约束：

1. 评分必须服务于阵营级结论，不做公开个人评分。
2. 风格、篇幅或修辞不得凌驾于实质论证质量之上。

### 7.3 裁决约束

Judge Kernel 必须满足以下约束：

1. 公开结果只能来自受版本控制的裁判规则体系。
2. 最终公开解释不得反向改写已锁定的裁决事实。
3. 身份信息、消费能力、历史声誉等不得直接进入单场裁判得分。
4. 证据不足或分歧过高时，应允许 draw 或 review，而不是强行判胜负。

### 7.4 规则版本化

服务必须对以下对象进行版本化管理：

1. `rubric`
2. `judge policy`
3. `Judge Kernel`
4. 复核规则
5. 可选的组织宪章或赛事制度版本

## 8. 裁决结果与展示要求

### 8.1 面向用户的裁决报告

用户可见结果至少应包含：

1. 最终结果或复核状态。
2. 阵营总分。
3. 四维评分。
4. 正反方核心观点与争点摘要。
5. 裁决理由。
6. 关键转折点。
7. 阶段摘要或争点展开（支持分页、折叠或分层浏览）。
8. 证据引用与背景来源说明。
9. 必要的裁判说明与制度说明。

### 8.2 面向业务系统的结构化结果

上游系统消费的结构化结果至少应包含：

1. 公开结论与状态。
2. 展示所需的富内容结构，包括阶段摘要或争点展开能力。
3. draw / review / challenge 所需的业务标记。
4. 版本信息与结果承诺标记。

### 8.3 面向内部的审计层

内部结果必须支撑：

1. 复盘。
2. 质量治理。
3. 公平性分析。
4. 规则回放。
5. 可信承诺与审计导出。

约束：

1. 用户不可见内部置信度、原始 prompt、原始风控字段等敏感内部信息。

## 9. 知识与证据需求

### 9.1 目标

服务应通过知识与证据能力提升以下能力：

1. 背景理解。
2. 事实支撑。
3. 可解释性。
4. 证据引用可追溯性。

### 9.2 知识来源原则

服务应优先使用：

1. 平台维护的主题知识库。
2. 官方、可信、批准来源。
3. 赛事或组织允许的专属资料。

默认不应依赖：

1. 无约束的公共网页抓取。
2. 无出处的开放文本拼接。

### 9.3 使用原则

知识与证据的作用是：

1. 为争点提供背景材料与证据支撑。
2. 帮助识别事实冲突与证据不足。
3. 提升解释质量与来源展示能力。

知识与证据不应：

1. 直接替代案件本身的辩论表现。
2. 因“命中更多来源”而自动判某方更强。

### 9.4 展示原则

面向用户的裁决报告应支持合理展示：

1. 来源摘要。
2. 关键引用。
3. 冲突来源提示。

同时满足：

1. 不暴露不必要的内部检索细节。
2. 不暴露受限原文或敏感证据内容。

---

## 10. 公平性与可信要求

### 10.1 输入盲化与中立性

服务必须默认遵守：

1. 输入盲化。
2. 身份最小化。
3. 资格与身份验证只影响准入，不影响单场评分。

### 10.2 稳定性与偏差控制

服务必须支持：

1. 多裁判视角交叉判断。
2. 结果分歧控制。
3. 风格偏差与标签偏差检查。
4. 在不稳定案件上触发保护性结果。

### 10.3 证据链要求

服务必须支持：

1. 消息级引用链。
2. 背景来源引用链。
3. 关键裁决依据可追溯。
4. 证据冲突可标记。

### 10.4 复核与 Challenge 要求

服务必须支持：

1. 平台内部复核能力。
2. 受约束的 challenge 能力。
3. 复核过程绑定原裁决版本、原案件快照与新增材料。

### 10.5 可验证信任层

服务最终形态应具备：

1. 案件承诺能力。
2. 裁决承诺能力。
3. Judge Kernel 版本承诺能力。
4. 复核状态可追溯能力。
5. 面向未来公开验证与协议化扩展的接口预留。

---

## 11. 可观测性与治理能力

### 11.1 Trace 与 Replay

服务必须支持：

1. 关键案件链路追踪。
2. 输入与结果回放。
3. 问题案件复盘。
4. 规则与结果变更审计。

### 11.2 Review 与 Alert

服务必须支持：

1. 高风险案件进入 review queue。
2. 偏差、证据不足、失败状态的告警能力。
3. challenge 与复核状态可查询。

### 11.3 Benchmark 与 Drift

服务必须支持长期质量治理，包括：

1. 公平性 benchmark。
2. 规则版本对比。
3. 漂移检测。
4. 新旧策略影子评估。

### 11.4 Policy Rollout

服务必须支持：

1. 裁判规则演进。
2. 版本灰度与回滚。
3. 多赛事、多组织的规则并存与切换。

## 12. 内部接口与协作要求

### 12.1 调用方式

主系统与 `ai_judge_service` 的默认协作方式应满足：

1. 仅内部调用。
2. 支持异步裁判任务。
3. 支持结构化结果回传。
4. 支持失败回传。

### 12.2 安全要求

内部调用面必须满足：

1. 身份可信。
2. 权限隔离。
3. 数据最小化。
4. 不对外暴露公众裁判入口。

### 12.3 幂等要求

服务必须保证：

1. 同一案件不会因重复触发产生多个公开最终结果。
2. 失败、复核、重试链路具备明确状态边界。

---

## 13. 非功能需求

### 13.1 时效

裁判为赛后异步执行，不阻塞用户核心实时操作。

### 13.2 稳定性

服务必须支持生产可用的失败处理、重试、降级与恢复能力。

### 13.3 一致性

同一案件的最终公开结果在所有消费者侧必须一致。

### 13.4 可理解性

裁决文案需兼具理性、克制与可读性，不能只给结论，不给理由。

### 13.5 隐私最小化

服务应尽量避免明文身份、资格或敏感证据扩散，满足最小必要使用原则。

### 13.6 可验证性

关键案件应具备结果承诺、版本承诺与状态历史可追溯能力。

---

## 14. 服务结果要求

### 14.1 功能结果要求

1. 合法案件可进入正式裁判或明确进入保护性流程。
2. 用户能看到结构完整、语义一致的裁决报告。
3. 上游系统能消费稳定的结构化结果并继续业务流程。

### 14.2 公平结果要求

1. 身份相关信息不得直接影响判决。
2. 关键裁决必须具备消息级证据链。
3. 不稳定案件不得静默放行为普通胜负结果。

### 14.3 可信结果要求

1. 关键结果可绑定规则版本与 Kernel 版本。
2. 复核与 challenge 过程可追溯。
3. 平台不可赛后静默改口。

### 14.4 运维结果要求

1. 关键案件可 trace。
2. 高风险案件可 replay。
3. 质量问题可告警、可复盘、可治理。

---

## 15. 待后续扩展问题

1. 不同赛事、学校、企业或组织是否需要独立宪章与规则版本。
2. 受约束 challenge 的开放范围应如何按赛事等级或用户等级控制。
3. `Reason Passport` 与资格证明应在何时进入平台正式主线。
4. 可验证承诺是否需要面向外部公开验证与链上锚定。
