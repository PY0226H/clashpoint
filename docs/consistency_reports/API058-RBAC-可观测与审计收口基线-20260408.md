# API058 RBAC 可观测与审计收口基线（B4/B5）

更新时间：2026-04-08  
适用范围：`GET /api/debate/ops/rbac/roles`、`GET /api/debate/ops/rbac/me`、`PUT/DELETE /api/debate/ops/rbac/roles/{userId}`

## 1. 目标

1. 给 RBAC 管理面建立可落地的 dashboard 与告警阈值基线。  
2. 在审计落库已生效前提下，冻结留存策略与受控查询面，降低治理不确定性。  

## 2. 指标字典落地

已在 `GET /api/debate/ops/observability/metrics-dictionary` 纳入 RBAC 专项 key（`ops.rbac.*`）：

1. `ops.rbac.roles_list.*`
2. `ops.rbac.me.*`
3. `ops.rbac.roles_write.*`

关键字段覆盖：

1. 请求总量：`request_total`
2. 成功/失败：`success_total`、`failed_total`
3. 限流：`rate_limited_total`
4. 业务分桶：`permission_denied_total`、`owner_total`、`non_owner_total`、`upsert_total`、`revoke_total`
5. 时延：`latency_p95_ms`

## 3. Dashboard 基线（建议）

| 看板面板 | 指标 key | 时间窗 | 建议阈值 |
|---|---|---|---|
| Roles List 错误率 | `ops.rbac.roles_list.failed_total / request_total` | 5m | Warning: > 1% ; Critical: > 3% |
| Roles List 限流命中 | `ops.rbac.roles_list.rate_limited_total` | 5m | Warning: > 20 ; Critical: > 60 |
| RBAC Me 错误率 | `ops.rbac.me.failed_total / request_total` | 5m | Warning: > 0.5% ; Critical: > 2% |
| RBAC Me 限流命中 | `ops.rbac.me.rate_limited_total` | 5m | Warning: > 30 ; Critical: > 90 |
| Roles Write 失败率 | `ops.rbac.roles_write.failed_total / request_total` | 5m | Warning: > 1% ; Critical: > 2% |
| Roles Write 限流命中 | `ops.rbac.roles_write.rate_limited_total` | 5m | Warning: > 10 ; Critical: > 30 |
| Roles List 时延 | `ops.rbac.roles_list.latency_p95_ms` | 5m | Warning: > 300ms ; Critical: > 800ms |
| RBAC Me 时延 | `ops.rbac.me.latency_p95_ms` | 5m | Warning: > 200ms ; Critical: > 500ms |
| Roles Write 时延 | `ops.rbac.roles_write.latency_p95_ms` | 5m | Warning: > 300ms ; Critical: > 700ms |

说明：

1. 阈值为工程初始值，需在一周样本后回标。  
2. 报警建议按“连续 2 个窗口触发”再升级，避免抖动误报。  

## 4. 值班排障顺序（Runbook）

1. 先看 `failed_total` 与 `rate_limited_total`：判断是系统失败还是流量治理问题。  
2. 若失败升高：按 `audit_event` 检索 `*_failed`，优先确认 owner 真源、权限冲突、DB 错误。  
3. 若限流升高：同时看 user/ip 分桶，判断是否单账号误用或来源 IP 冲击。  
4. 写链路异常时，检查 `upsert_total/revoke_total` 变化，排除前端重试风暴。  
5. 完成处置后，记录“触发时间、告警项、根因、修复动作、恢复时间”。  

## 5. 审计策略（B5）

### 5.1 当前落库状态（已执行）

1. 审计表：`ops_rbac_audits`（迁移：`chat/migrations/20260408042000_ops_rbac_audits.sql`）。  
2. 覆盖接口：`rbac/roles`、`rbac/me`、`rbac/roles/{userId}`。  
3. 事件类型：`roles_list_read`、`rbac_me_read`、`role_upsert`、`role_revoke`。  
4. 决策分桶：`success`、`failed`、`rate_limited_user`、`rate_limited_ip`。  
5. 关键字段：`operator_user_id`、`target_user_id`、`request_id`、`result_count`、`role`、`removed`、`created_at`。  

### 5.2 留存策略（可执行基线）

建议基线：

1. 热数据保留：180 天。  
2. 清理方式：按批次删除，避免长事务放大锁等待。  
3. 先 dry-run 再 apply。  

已提供脚本：

1. dry-run：
   `bash chat/scripts/ops_rbac_audits_retention.sh --days 180`
2. apply：
   `bash chat/scripts/ops_rbac_audits_retention.sh --days 180 --apply --chunk-size 5000 --max-batches 20`

### 5.3 受控查询面（可执行基线）

已提供运维查询脚本（支持时间窗、操作者、目标用户、事件、决策过滤 + JSONL 导出）：

1. 明细查询：
   `bash chat/scripts/ops_rbac_audits_query.sh --from 2026-04-08T00:00:00Z --to 2026-04-09T00:00:00Z --event-type roles_list_read --limit 200`
