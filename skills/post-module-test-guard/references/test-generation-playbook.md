# Test Generation Playbook

## 原则
1. 一次模块改动至少覆盖：
- 1 条主流程
- 1 条失败流程
- 1 条边界条件

2. 修复类改动必须带回归测试。
3. 测试名称体现行为，不体现实现细节。

## Rust 补测建议
- 优先在同文件补 `#[cfg(test)]` 单元测试，覆盖纯逻辑分支。
- 涉及路由、数据库、middleware 的行为，优先集成测试（`tests/`）。
- 断言包含：状态码、关键字段、错误路径。

## 前端补测建议（当前仓库）
- 缺少统一单测框架时，优先补 E2E 场景。
- 选择对用户有可见影响的路径：登录、发消息、文件上传、路由守卫。

## 失败后的修复顺序
1. 先看是否测试期望写错（规范/接口是否变化）。
2. 再看实现是否回归（逻辑分支是否漏改）。
3. 最后看环境问题（依赖、数据库、端口、配置）。

## 完成定义
- 测试门禁通过。
- 新增测试能证明本次改动行为。
- 面试时可回答“如何证明改动是对的”。
